# 工具优先栈

这一层的原则很简单：能用现成工具就用现成工具。目标不是证明你能把所有底层轮子都手搓出来，而是把一个可靠的 TypeScript agent 脚手架搭稳。

## 默认选择

| 关注点      | 优先工具                                        | 什么时候用                                | 不要做什么                          |
| ----------- | ----------------------------------------------- | ----------------------------------------- | ----------------------------------- |
| 包管理      | `pnpm`                                          | 新项目默认用                              | 不要为了省事混用多个包管理器        |
| TS 运行     | `tsx`                                           | 跑脚本、调试、CLI                         | 不要手写编译后再跑的复杂流程        |
| CLI 参数    | `commander` / `cac` / `yargs`                   | 解析命令、参数、help                      | 不要手写参数解析器                  |
| 配置加载    | Node `--env-file` / `dotenvx` / `dotenv`        | 加载 `.env`                               | 不要在代码里硬编码配置              |
| 配置校验    | `zod`                                           | 校验 `.env`、CLI 参数、接口返回、LLM 输出 | 不要把校验散落在 `if` 里            |
| Schema 体系 | `zod` / JSON Schema                             | 作为工具边界和输出契约                    | 不要把 schema 和业务逻辑混在一起    |
| 日志        | `pino` / `winston`                              | CLI、服务、任务流程日志                   | 不要用一堆 `console.log` 当系统日志 |
| 测试        | `Vitest` / `Jest`                               | 单测、集成测试                            | 不要自己写测试 runner               |
| HTTP mock   | `msw` / `nock`                                  | 模拟外部 API、RPC、模型端点               | 不要自己写 mock 框架                |
| 质量工具    | `ESLint` / `Prettier` / `lint-staged` / `husky` | 保持风格和提交质量                        | 不要手写格式化脚本                  |

## 选型规则

1. 先选官方或社区最常用的工具。
2. 一个职责只保留一个主工具，别把同类工具叠太多。
3. 如果已有 SDK 或框架提供现成能力，优先接入，不自己重做。
4. 只有在学习原理、或者现成工具明显不适合时，才做最小实现。

## 这层的底线

- 不手写参数解析器。
- 不手写测试框架。
- 不手写日志框架。
- 不手写 HTTP mock 框架。
- 不手写环境变量解析和配置校验。
- 不把私钥、API key、RPC key 打进仓库或日志。

## 推荐默认组合

```text
pnpm + TypeScript strict + tsx + commander/cac + zod
+ pino + Vitest + msw/nock + ESLint + Prettier
```
