## 模块 1：TypeScript strict 基础

模块 1 的核心目标是：**不要用 `any` 糊外部输入，而是用 TypeScript strict 把“不可信输入”一步步收窄成“可信业务对象”。**

这不是最终项目推荐的校验方式。模块 1 先手写 type guard，是为了理解 TypeScript 的类型收窄机制；到模块 5，会用 zod / JSON Schema 替代大部分手写校验。

### 这节学了什么

- `unknown` 比 `any` 更安全：外部输入先当作 `unknown`，使用前必须检查。
- `as const` 可以把数组里的字符串保留成字面量类型。
- `(typeof ACTIONS)[number]` 可以从常量数组里提取联合类型，例如 `"quote" | "execute" | "status"`。
- `ToolResult<T>` 用联合类型表达成功/失败，比随便抛异常或返回 nullable 更清楚。
- `result.ok` 是判别字段，TypeScript 会根据它自动区分成功分支和失败分支。
- `satisfies` 用来检查对象符合某个类型，同时尽量保留对象自身的精确类型。
- `Pick` / `Partial` / `Record` 可以从核心业务类型派生出不同场景需要的类型。
- type guard 形如 `value is SomeType`，用于把 `unknown` 安全收窄成具体类型。

### 复习时重点看

1. `as const` 和 `(typeof ACTIONS)[number]`

```ts
const ACTIONS = ["quote", "execute", "status"] as const;
type TaskAction = (typeof ACTIONS)[number];
```

要记住：这里没有创建新数组，`TaskAction` 是类型，结果是：

```ts
type TaskAction = "quote" | "execute" | "status";
```

2. `ToolResult<T>` 成功/失败联合类型

```ts
type ToolResult<T> = { ok: true; data: T } | { ok: false; error: ToolError };
```

复习点：

- `ok: true` 时才能访问 `data`。
- `ok: false` 时才能访问 `error`。
- `result.ok` 能让 TypeScript 自动收窄类型。

3. `unknown` 到可信对象的收窄流程

```ts
function parseTaskIntent(value: unknown): ToolResult<TaskIntent> {
  if (!isRecord(value)) {
    return { ok: false, error: { code: "INVALID_INPUT", message: "input must be an object" } };
  }

  // 只有确认 value 是对象后，才开始检查字段。
}
```

复习点：外部输入不要直接信。CLI 参数、HTTP 返回、LLM 输出、MCP tool input、链上 RPC 返回，进入程序时都可以先当作 `unknown`。

4. `Pick` / `Partial` / `Record`

```ts
type QuoteInput = Pick<TaskIntent, "service" | "maxAmount">;
type TaskPatch = Partial<Pick<TaskIntent, "maxAmount" | "expiresAt">>;
type TaskById = Record<TaskIntent["taskId"], TaskIntent>;
```

复习点：

- `Pick`：从大类型里挑字段。
- `Partial`：把字段变成可选。
- `Record`：描述 key/value 映射表。

### 这段示例代码在模拟什么

```ts
const rawInput = JSON.parse(JSON.stringify(defaultIntent)) as unknown;
const parsed = parseTaskIntent(rawInput);
```

它在模拟一个外部输入。真实项目里，外部输入可能来自：

- CLI 参数
- `.env`
- HTTP API
- LLM 输出
- MCP tool input
- 链上 RPC 返回

所以这里故意把 `defaultIntent` 转成 JSON 再转回来，并标记为 `unknown`，提醒自己：进入系统边界的数据不能直接信。

### 易错点

- 不要把 `unknown` 换成 `any`。`any` 会让 TypeScript 放弃保护。
- 不要以为 `type TaskAction = ...` 创建了数组。`type` 只存在于类型层面，运行时没有这个东西。
- 不要在 `parsed.ok` 判断前访问 `parsed.data`。
- 不要忘记 `ACTIONS` 后面的 `as const`。没有它，`TaskAction` 很可能退化成普通 `string`。
- 不要把模块 1 的手写校验当成最终项目模式。最终要用 zod 统一 schema。

## 模块 2：ESM、Node.js 与 tsx

模块 2 的核心目标是：**理解现代 Node.js 项目怎么用 ESM 写模块，并用 `tsx` 直接运行 TypeScript 文件。**

这节不是让你研究模块系统历史，而是为了后面顺利接现代 SDK：MCP SDK、LangChain.js、LangGraph.js、viem、x402 SDK、A2A SDK 等都更偏现代 ESM / TypeScript 生态。

### 这节学了什么

- `"type": "module"` 会告诉 Node：这个包里的 `.js` 文件默认按 ESM 规则运行。
- ESM 用 `import` / `export`，而不是 CommonJS 的 `require` / `module.exports`。
- `tsx` 可以直接运行 `.ts` 文件，不需要先手动编译成 `.js`。
- `module: "NodeNext"` 和 `moduleResolution: "NodeNext"` 会让 TypeScript 按 Node 的 ESM 规则理解模块。
- ESM 里没有默认的 `__filename` / `__dirname`，常用 `import.meta.url` 获取当前模块的 file URL。
- ESM 相对导入经常写 `.js` 后缀，例如 `import { add } from "./math.js"`，即使开发时源文件是 `math.ts`。
- `JSON.stringify(value, null, 2)` 里的 `null` 表示不自定义字段替换，`2` 表示用 2 个空格缩进输出。

### 复习时重点看

1. `package.json` 的 ESM 开关

```json
{
  "type": "module"
}
```

复习点：这个字段影响 Node 怎么解释 `.js` 文件。没有它时，Node 默认更偏 CommonJS；有它时，项目默认进入 ESM 模式。

2. ESM 导入和导出

```ts
// math.ts
export function add(a: number, b: number): number {
  return a + b;
}

export const version = "esm-demo" as const;
```

```ts
// module2-esm.ts
import { add, version } from "./math.js";
```

复习点：`export` 暴露能力，`import` 引入能力。这里是命名导出 / 命名导入。

3. 为什么导入写 `./math.js`

```ts
import { add, version } from "./math.js";
```

复习点：在 `NodeNext` / ESM 模式下，TypeScript 会按 Node 运行时规则理解路径。开发时文件是 `math.ts`，但运行时对应的是编译后的 `math.js`，所以相对导入常写 `.js` 后缀。

4. `import.meta.url`

```ts
console.log(import.meta.url);
```

复习点：它表示当前 ESM 模块自己的 `file://` 地址。CommonJS 里的 `__filename` / `__dirname` 在 ESM 里不是默认可用的。

5. `JSON.stringify(..., null, 2)`

```ts
console.log(JSON.stringify({ mode: "esm", result: 5 }, null, 2));
```

复习点：

- 第一个参数：要转成 JSON 字符串的对象。
- 第二个参数 `null`：不做字段过滤或替换。
- 第三个参数 `2`：用 2 个空格缩进，让输出更好读。

### 这段示例代码在模拟什么

```ts
import { add, version } from "./math.js";

function main(): void {
  const result = add(2, 3);

  console.log(
    JSON.stringify(
      {
        mode: "esm",
        version,
        result,
        fileUrl: import.meta.url,
      },
      null,
      2,
    ),
  );
}

main();
```

它在模拟一个最小模块化程序：

- `math.ts` 负责导出函数和常量。
- `module2-esm.ts` 负责导入并使用它们。
- `tsx` 负责直接运行 TypeScript 文件。
- 输出 JSON 证明代码确实跑起来了，也展示了当前模块的 `import.meta.url`。

### 易错点

- 不要把 ESM 和 CommonJS 混着写。ESM 用 `import/export`，CommonJS 用 `require/module.exports`。
- 不要忘记 `package.json` 的 `"type": "module"`。
- 在 `NodeNext` 下，相对导入写 `./math.js` 是正常的，不是写错。
- `import.meta.url` 不是普通文件路径，而是 `file://` URL。
- `JSON.stringify(obj, null, 2)` 只是美化输出，不会改变对象本身。
- `tsx` 是开发运行工具，不等于生产打包方案。现在用它是为了快速学习和跑脚本。

### 自测题

1. `"type": "module"` 在 `package.json` 里起什么作用？
2. ESM 和 CommonJS 的导入导出写法分别是什么？
3. 为什么 TypeScript 文件里会写 `import ... from "./math.js"`？
4. `import.meta.url` 输出的是什么？
5. `tsx` 解决了什么问题？
6. `JSON.stringify(obj, null, 2)` 里的 `null` 和 `2` 分别是什么意思？

### 模块 2 通过标准

- 能解释 ESM 是什么。
- 能解释 `"type": "module"` 的作用。
- 能写出一个文件 `export`，另一个文件 `import` 的最小例子。
- 能解释为什么相对导入常写 `.js` 后缀。
- 能解释 `import.meta.url` 是当前模块的 URL。


## 模块 3：CLI 参数解析

模块 3 的核心目标是：**让项目有一个可维护的命令行入口，用成熟库解析参数，而不是自己手写 `process.argv`。**

这节不是为了做复杂业务，而是先把 CLI 的外壳搭起来。后面的 `.env` 配置、zod 校验、日志、HTTP mock、链上查询、MCP tool 调用，都会接到这个入口后面。

### 这节学了什么

- `commander` 可以创建命令行程序，负责解析 command、option、required option。
- `program.name()` 设置 CLI 显示名称。
- `program.description()` 设置 CLI 描述。
- `program.version()` 设置版本号，可通过 `--version` 查看。
- `showHelpAfterError()` 让命令出错时顺便显示帮助信息。
- `showSuggestionAfterError()` 让命令拼错时给出相近命令建议。
- `command("quote")` / `command("execute")` / `command("status")` 定义子命令。
- `requiredOption()` 定义必填参数。
- `option()` 定义可选参数。
- `.action()` 里放命令真正要执行的逻辑。
- `program.parse(process.argv)` 会读取终端传进来的参数并触发对应命令。
- `process.exitCode = 1` 表示程序执行失败，但不会立刻强行退出。

### 复习时重点看

1. `package.json` 里的脚本入口

```json
{
  "scripts": {
    "agent": "tsx src/cli.ts"
  }
}
```

复习点：`agent` 是脚本名，`tsx src/cli.ts` 是真正执行的命令。运行：

```bash
pnpm agent quote --service risk-check --amount 5
```

大致等价于：

```bash
pnpm tsx src/cli.ts quote --service risk-check --amount 5
```

2. 创建 CLI 程序对象

```ts
const program = new Command();

program
  .name("agent")
  .description("Agent toolbox CLI")
  .version("0.1.0")
  .showHelpAfterError()
  .showSuggestionAfterError();
```

复习点：这段不是业务逻辑，而是在配置 CLI 的基础信息和使用体验。它会影响 `--help`、`--version`、错误提示和拼写建议。

3. 参数解析和参数校验

```ts
function parseNonEmpty(value: string): string {
  const trimmed = value.trim();

  if (trimmed.length === 0) {
    throw new InvalidArgumentError("must not be empty");
  }

  return trimmed;
}

function parsePositiveAmount(value: string): number {
  const amount = Number(value);

  if (!Number.isFinite(amount) || amount <= 0) {
    throw new InvalidArgumentError("amount must be a positive number");
  }

  return amount;
}
```

复习点：命令行参数刚进来时本质上是字符串。这里先把空字符串、负数、非数字挡掉，让后面的业务函数拿到更干净的数据。

4. 子命令 `quote`

```ts
program
  .command("quote")
  .description("Get a quote for an agent service")
  .requiredOption("--service <name>", "service name, for example risk-check", parseNonEmpty)
  .requiredOption("--amount <usdc>", "max payment amount in USDC", parsePositiveAmount)
  .option("--json", "print JSON output", false)
  .action((options: QuoteOptions) => {
    quote({
      service: options.service,
      amount: options.amount,
      json: options.json ?? false,
    });
  });
```

复习点：

- `.command("quote")` 定义一个叫 `quote` 的子命令。
- `.requiredOption(...)` 表示这个参数必须传。
- `parseNonEmpty` / `parsePositiveAmount` 是参数解析器。
- `.option("--json", ..., false)` 表示 `--json` 默认是 `false`。
- `.action(...)` 是参数解析成功后真正执行的函数。

5. 业务函数只接收整理后的对象

```ts
type QuoteInput = {
  service: string;
  amount: number;
  json: boolean;
};

function quote(input: QuoteInput): void {
  const result = {
    ok: true,
    action: "quote",
    service: input.service,
    amount: input.amount,
    currency: "USDC",
    estimatedFee: 0.05,
    taskId: "task_001",
  };

  printOutput(result, input.json);
}
```

复习点：业务函数不应该自己到处读 `process.argv`，而是接收已经整理好的 `QuoteInput`。这样以后测试、复用、替换真实业务逻辑都更容易。

6. 错误状态和 exit code

```ts
if (!input.taskId.startsWith("task_")) {
  printOutput(
    {
      ok: false,
      error: {
        code: "TASK_NOT_FOUND",
        message: `task not found: ${input.taskId}`,
      },
    },
    input.json,
  );

  process.exitCode = 1;
  return;
}
```

复习点：CLI 不只要打印错误，还要用 exit code 告诉外部系统这次命令失败了。`0` 通常表示成功，非 `0` 表示失败。

### 这段示例代码在模拟什么

```bash
pnpm agent quote --service risk-check --amount 5 --json
pnpm agent execute --task-id task_001 --dry-run --json
pnpm agent status --task-id task_001 --json
```

它在模拟一个最小的 agent 工具箱命令行：

- `quote`：查询某个服务报价。
- `execute`：执行某个任务。
- `status`：查询某个任务状态。

现在这些命令只返回假数据。真实项目里可以逐步替换为：

- `quote` 调 HTTP API、MCP tool 或服务市场报价接口。
- `execute` 发起 agent 任务、调用合约或触发 x402 支付。
- `status` 查询数据库、链上事件或 agent 执行结果。

所以模块 3 的重点是：先把命令入口、参数解析、输出格式和错误状态练稳。

### 易错点

- 不要自己手写复杂的 `process.argv` 解析；CLI 参数交给 `commander`。
- 不要把 `program.name("agent")` 理解成创建了一个系统命令。它只是 CLI 显示名称。
- 不要忘记最后的 `program.parse(process.argv)`。没有它，命令不会真正被解析和执行。
- 不要把所有逻辑都塞进 `.action()`。更好的做法是 `.action()` 负责接参数，然后调用独立业务函数。
- 不要把命令行传入的金额直接当数字用。CLI 参数进来通常是字符串，要先解析和校验。
- 不要只打印错误却不设置 exit code。脚本、CI、自动化流程需要靠 exit code 判断成功或失败。
- 不要把现在的假数据当成真实业务。这个模块练的是 CLI 壳子，不是链上逻辑。

## 模块 4：配置、环境变量与密钥安全

模块 4 的核心目标是：**程序启动时先读取配置、校验配置，并且不要把密钥泄漏到代码、日志或仓库里。**

这节开始把 CLI 从“能接命令”推进到“能安全启动”。后面接 RPC、API、钱包、模型服务时，都会依赖 `.env` 和配置校验这一层。

### 这节学了什么

- `.env` 存放本机真实配置，比如 `RPC_URL`、`CHAIN_ID`、`SERVICE_API_KEY`。
- `.env.example` 存放示例配置，用来告诉别人项目需要哪些环境变量。
- `.gitignore` 里要加入 `.env`，避免把真实密钥提交到仓库。
- Node 的 `--env-file=.env` 可以在启动时读取 `.env`。
- `process.env` 是 Node 里读取环境变量的对象。
- `zod` 可以校验环境变量格式，避免配置错误进入业务逻辑。
- `z.coerce.number()` 可以把字符串环境变量转成数字再校验。
- `.default("info")` 可以给可省略配置设置默认值。
- `safeParse()` 不会直接抛异常，而是返回成功/失败结果。
- 日志里不能直接打印 secret，API key 和 RPC URL 都要脱敏。
- 配置失败时设置 `process.exitCode = 1`，让脚本和 CI 知道启动失败。

### 复习时重点看

1. `.env` 和 `.env.example` 的区别

```bash
# .env.example
RPC_URL=https://base-sepolia.example
CHAIN_ID=84532
LOG_LEVEL=info
SERVICE_API_KEY=replace-me
```

```bash
# .env
RPC_URL=https://your-rpc.example
CHAIN_ID=84532
LOG_LEVEL=debug
SERVICE_API_KEY=sk-live-example
```

复习点：`.env.example` 可以提交到仓库，因为它不放真实 secret；`.env` 是本机真实配置，不能提交。

2. `.gitignore` 保护本地密钥

```gitignore
.env
```

复习点：这行的作用是让 Git 忽略 `.env`。Web3 项目里，私钥、RPC key、API key 都不能进仓库。

3. 启动命令

```bash
node --env-file=.env --import tsx src/playground/module4-config.ts
```

复习点：

- `node`：启动 Node.js。
- `--env-file=.env`：先读取 `.env`，放进 `process.env`。
- `--import tsx`：让 Node 可以直接执行 TypeScript 文件。
- `src/playground/module4-config.ts`：真正运行的配置校验脚本。

4. 配置 schema

```ts
const EnvSchema = z.object({
  RPC_URL: z.url(),
  CHAIN_ID: z.coerce.number().int().positive(),
  LOG_LEVEL: z.enum(["debug", "info", "warn", "error"]).default("info"),
  SERVICE_API_KEY: z.string().min(1).optional(),
});
```

复习点：

- `RPC_URL` 必须是 URL。
- `CHAIN_ID` 从环境变量读进来是字符串，用 `z.coerce.number()` 转成数字。
- `LOG_LEVEL` 只能是四个值之一，没填就默认 `info`。
- `SERVICE_API_KEY` 是可选项，但如果填了，不能为空字符串。

如果你的 zod 版本里 `z.string().url()` 被画横线，就用 `z.url()` 或 `z.httpUrl()`。

5. `safeParse(process.env)`

```ts
const parsed = EnvSchema.safeParse(process.env);

if (!parsed.success) {
  console.error(
    JSON.stringify(
      {
        ok: false,
        error: {
          code: "INVALID_ENV",
          message: "configuration is invalid",
        },
      },
      null,
      2,
    ),
  );

  process.exitCode = 1;
  return;
}

const config = parsed.data;
```

复习点：`process.env` 是外部输入，不能直接信。只有 `safeParse` 成功后，`parsed.data` 才是通过校验的配置对象。

6. 密钥脱敏

```ts
function maskSecret(value: string): string {
  if (value.length <= 8) {
    return "***";
  }

  return `${value.slice(0, 4)}...${value.slice(-4)}`;
}
```

复习点：日志和 JSON 输出里不要完整打印 API key、私钥、token。能不打印就不打印；必须展示时，只展示头尾几位。

7. RPC URL 也要小心

```ts
function maskUrl(value: string): string {
  const url = new URL(value);

  return `${url.protocol}//${url.host}/...`;
}
```

复习点：很多 RPC URL 里面会带项目 ID 或 key，所以也不要完整打印。只显示协议和 host 通常就够了。

### 这段示例代码在模拟什么

```ts
const parsed = EnvSchema.safeParse(process.env);
```

它在模拟一个程序启动时的配置安全门：

```text
读取 .env
  -> 放进 process.env
  -> zod 校验
  -> 成功：得到可信 config
  -> 失败：输出 INVALID_ENV，exit code = 1
```

真实项目里，这一层会保护后面的能力：

- 连接链上 RPC 前，确认 `RPC_URL` 合法。
- 发交易前，确认 `CHAIN_ID` 是预期网络。
- 调外部服务前，确认 API key 存在。
- 打日志时，不泄漏 key、token、私钥、RPC 项目 ID。

所以模块 4 不是在做业务功能，而是在给项目加“启动前检查”。

### 易错点

- 不要把 `.env` 提交到仓库。
- 不要把私钥、API key、RPC key 写死在 TypeScript 文件里。
- 不要直接使用 `process.env.RPC_URL` 进入业务逻辑，先用 schema 校验。

## 模块 5：zod / JSON Schema 作为工具边界

模块 5 的核心目标是：**把所有外部输入和结构化输出统一成 schema，再从 schema 推导类型。**

这节是前面几节的收口。模块 1 先手写 type guard，模块 4 先校验 `.env`，到了模块 5，你要开始把 CLI 参数、任务意图、工具输入、agent 输出都放进 schema 里管。

### 这节学了什么

- `z.object()` 定义对象 schema。
- `z.enum()` 定义枚举。
- `z.string().min(1)` 或 `z.string().trim().min(1)` 定义非空文本。
- `z.coerce.number()` 把字符串输入转成数字再校验。
- `safeParse()` 返回成功/失败结果，不会直接抛异常。
- `z.infer<typeof Schema>` 可以从 schema 推导 TypeScript 类型。
- `strict()` 可以禁止多余字段进入对象。
- `z.toJSONSchema()` 可以把 Zod schema 导成 JSON Schema。
- 错误应该格式化成可读的 `path + message`。

### 复习时重点看

1. 基础复用字段

```ts
const NonEmptyText = z.string().trim().min(1);
const ActionSchema = z.enum(["quote", "execute", "status"]);
const RiskLevelSchema = z.enum(["low", "medium", "high"]);
const ToolNameSchema = z.enum(["mockChainTool", "mockHttpTool"]);
```

复习点：先把常复用的小 piece 拆出来，后面的 schema 才不会一大坨。

2. `EnvSchema`

```ts
const EnvSchema = z.object({
  RPC_URL: z.httpUrl(),
  CHAIN_ID: z.coerce.number().int().positive(),
  LOG_LEVEL: z.enum(["debug", "info", "warn", "error"]).default("info"),
  SERVICE_API_KEY: z.string().min(1).optional(),
});
```

复习点：这和模块 4 的思路一样，但现在它不只是“能读配置”，而是“配置本身也变成可复用 schema”。

3. `TaskIntentSchema`

```ts
const TaskIntentSchema = z
  .object({
    taskId: NonEmptyText,
    action: ActionSchema,
    service: NonEmptyText,
    maxAmount: z.coerce.number().positive(),
    expiresAt: z.coerce.number().int().positive(),
    nonce: NonEmptyText,
  })
  .strict();
```

复习点：这是任务意图的边界。任务 ID、动作、服务名、金额、过期时间、nonce 都要过 schema，不能乱塞字段。

4. `ToolInputSchema`

```ts
const ToolInputSchema = z
  .object({
    requestId: NonEmptyText,
    tool: ToolNameSchema,
    input: z
      .object({
        service: NonEmptyText,
        maxAmount: z.coerce.number().positive(),
      })
      .strict(),
  })
  .strict();
```

复习点：工具输入经常是嵌套对象。这里演示的是 MCP tool / HTTP tool 的典型边界写法。

5. `AgentOutputSchema`

```ts
const AgentOutputSchema = z
  .object({
    ok: z.boolean(),
    action: ActionSchema,
    riskLevel: RiskLevelSchema,
    confidence: z.coerce.number().min(0).max(1),
    requiresApproval: z.boolean(),
    reasons: z.array(NonEmptyText),
  })
  .strict();
```

复习点：这就是结构化 agent 输出的雏形。模型可以提建议，但最后要靠 schema 把它变成可验证的数据。

6. `z.infer`

```ts
type TaskIntent = z.infer<typeof TaskIntentSchema>;
```

复习点：类型不再手写，直接从 schema 推出来。这样 schema 改了，类型跟着改，不容易不一致。

7. `safeParse` + `report`

```ts
const taskIntentResult = TaskIntentSchema.safeParse(taskIntentCandidate);

if (taskIntentResult.success) {
  const taskIntent: TaskIntent = taskIntentResult.data;
}
```

复习点：`safeParse()` 成功后，`data` 才能安全使用。失败时不要硬取字段。

8. `z.toJSONSchema`

```ts
showJsonSchema("ToolInputSchema", ToolInputSchema);
```

复习点：这一步把 Zod schema 导出成 JSON Schema，后面可以给 API 文档、工具协议、前后端共享校验用。

### 这段示例代码在模拟什么

这个脚本在模拟四类边界：

- `action` 填成了 `pay`。
- `tool` 填成了未知名字。
- `confidence` 超出 0 到 1。
- `reasons` 里塞了空字符串和数字。

目的很直接：**让你看到 schema 怎么挡住脏输入。**

### 易错点

- 不要把 schema 和类型分开手写两份，容易漂移。

## 模块 6：异步与错误处理

模块 6 的核心目标是：**外部调用失败时，不要直接炸栈，而是把错误分成可定位、可处理的结构化结果。**

这节开始接近真实工具调用场景：HTTP、RPC、MCP tool、队列任务、链上查询，都可能慢、失败、偶发抖动。通用机制尽量用成熟库，这里用 `p-retry` 和 `p-timeout`。

### 这节学了什么

- `async/await`：把异步流程写得更直观。
- `Promise.all`：并发等待多个异步任务。
- `p-timeout`：给某个异步调用加超时边界。
- `p-retry`：对临时失败做自动重试。
- `AppResult<T>`：把成功和失败统一成一种返回结构。
- 错误分类：把超时、工具失败、输入错误分开。
- 先校验输入，再调用外部工具，避免无意义重试。

### 复习时重点看

1. `AppResult<T>`

```ts
type AppResult<T> = { ok: true; value: T } | { ok: false; error: AppError };
```

复习点：成功时有 `value`，失败时有 `error`。`result.ok` 是判断成功/失败的开关。

2. 先校验输入

```ts
function validateQuoteRequest(request: QuoteRequest): AppResult<QuoteRequest> {
  if (request.amount <= 0) {
    return fail({
      code: "INVALID_INPUT",
      message: "amount must be greater than 0",
      retryable: false,
    });
  }

  return ok(request);
}
```

复习点：输入错了就直接返回 `INVALID_INPUT`，不要进入 retry。重试是给临时外部失败用的，不是给错误输入用的。

3. 用 `p-timeout` 做超时

```ts
function callQuoteToolWithTimeout(request: QuoteRequest, attempt: number): Promise<Quote> {
  return pTimeout(mockQuoteApi(request, attempt), {
    milliseconds: request.timeoutMs,
    message: `operation timed out after ${request.timeoutMs}ms`,
  });
}
```

复习点：超时控制是通用基础设施，工程里优先用成熟库。这里你要理解的是“外部调用必须有时间边界”。

4. 用 `p-retry` 做重试

```ts
const quote = await pRetry(
  (attemptNumber) => callQuoteToolWithTimeout(validation.value, attemptNumber),
  {
    retries: attempts - 1,
    minTimeout: 25,
    factor: 1,
  },
);
```

复习点：`retries` 是“失败后再试几次”，不是“总共试几次”。所以想总共尝试 `attempts` 次，要写 `retries: attempts - 1`。

5. 把库异常转成自己的错误结构

```ts
function classifyToolFailure(cause: unknown, attempts: number): AppError {
  if (cause instanceof PTimeoutError) {
    return {
      code: "TIMEOUT",
      message: `failed after ${attempts} attempt(s)`,
      retryable: false,
      cause: errorMessage(cause),
    };
  }

  return {
    code: cause instanceof Error ? "TOOL_FAILED" : "UNKNOWN_ERROR",
    message: `failed after ${attempts} attempt(s)`,
    retryable: false,
    cause: errorMessage(cause),
  };
}
```

复习点：库可以抛自己的错误，但项目最好输出自己的错误语义。这样 CLI、测试、日志、上层 agent 都更容易处理。

### 这段示例代码在模拟什么

它在模拟一个“外部工具调用包装器”：

- 先校验请求参数。
- 再调用一个 mock 外部工具。
- 工具太慢就超时。
- 临时失败就重试。
- 最后输出结构化 JSON。

对应到后面的真实项目，`mockQuoteApi` 可以逐步替换成真实 HTTP API、RPC 调用、MCP tool、x402 服务调用或链上 SDK 调用。

### 易错点

- 不要把 `p-retry` / `p-timeout` 这种基础工具再手写一遍。
- 不要让输入错误进入重试流程。
- 不要把 `retries` 理解成总尝试次数。
- 不要只输出一句 `failed`，要带 `code`、`message`、`cause`。
- 不要把超时错误和普通工具失败混成一类。

## 模块 7：日志与可观测性基础

模块 7 的核心目标是：**把一次任务做了什么，按步骤、按上下文、按级别记录下来。**

这节不是教你“多打点字”，而是教你把日志写成能查、能筛、能串联的一份运行记录。真实项目里，CLI、HTTP、MCP tool、链上调用都需要这套东西。

### 这节学了什么

- 结构化日志：不要只拼字符串，要输出对象。
- `requestId` / `taskId`：把同一次任务的日志串起来。
- `logger.child(context)`：让同一任务自动带上下文字段。
- 日志级别：`debug` / `info` / `warn` / `error`。
- 脱敏：API key、RPC URL 不能原样打印。
- 错误日志：失败时记录 `err`，方便排查。

### 复习时重点看

1. `logger.child(context)`

```ts
const taskLogger = logger.child(context);
```

复习点：这是最重要的地方。它会把 `requestId`、`taskId`、`action` 绑定到这一次任务的所有日志上。后面你只要打 `event`，上下文会自动跟上。

2. 结构化日志

```ts
taskLogger.info({
  event: "task.started",
  service: input.service,
  amount: input.amount,
});
```

复习点：日志不是一句话，而是一个对象。这样机器才能按字段查日志，比如查某个 `taskId` 的所有记录。

3. 日志级别

```ts
taskLogger.debug(...);
taskLogger.info(...);
taskLogger.warn(...);
taskLogger.error(...);
```

复习点：

- `debug`：细节。
- `info`：正常步骤。
- `warn`：有风险，但还没失败。
- `error`：真的失败了。

4. 脱敏

```ts
apiKey: maskSecret(input.apiKey),
rpcUrl: maskUrl(input.rpcUrl),
```

复习点：日志会被保存、搜索、上传，所以不能直接把密钥和带 token 的 URL 打出来。脱敏后还能看出是什么类型的值，但不会泄漏完整内容。

5. 错误日志

```ts
taskLogger.error({
  event: "task.failed",
  status: "failed",
  err: error,
});
```

复习点：失败时要留足信息。`err` 让 logger 把错误对象、message、stack 记下来，后面才好排查。

### 这段示例代码在模拟什么

它在模拟一次任务执行：

- 开始任务。
- 准备工具输入。
- 调用工具。
- 工具成功就记成功日志。
- 金额超阈值就记审批提醒。
- 工具失败就记错误日志。

`requestId` / `taskId` 只是追踪编号，不是业务请求本身。

### 易错点

- 不要把日志写成普通字符串。
- 不要让所有日志都一个 level。
- 不要忘记 `requestId` / `taskId`。
- 不要直接打印 API key、私钥、RPC URL。
- 不要只记“失败了”，要把 `err` 记录下来。

1. 三个 schema

```ts
export const EnvSchema = z
  .object({
    SERVICE_BASE_URL: z.string().url(),
    SERVICE_API_KEY: z.string().min(1),
  })
  .strict();
```

```ts
export const QuoteInputSchema = z
  .object({
    service: z.string().min(1),
    amount: z.number().positive(),
  })
  .strict();
```

```ts
export const QuoteResponseSchema = z
  .object({
    price: z.string().min(1),
    currency: z.literal("USDC"),
    requiresApproval: z.boolean(),
  })
  .strict();
```

复习点：配置、输入、HTTP 响应都不能直接信。先用 schema 挡住脏数据，后面的测试才有明确边界。

2. `AppResult<T>`

```ts
export type AppResult<T> =
  | { ok: true; value: T }
  | { ok: false; error: { code: string; message: string } };
```

复习点：成功和失败都用统一结构返回。测试里就可以稳定判断 `result.ok` 和 `result.error.code`。

3. `loadEnv`

```ts
export function loadEnv(source: NodeJS.ProcessEnv): AppResult<Env> {
  const parsed = EnvSchema.safeParse(source);

  if (!parsed.success) {
    return {
      ok: false,
      error: {
        code: "INVALID_ENV",
        message: parsed.error.issues.map((issue) => issue.path.join(".")).join(", "),
      },
    };
  }

  return { ok: true, value: parsed.data };
}
```

复习点：`.env` / `process.env` 本质上是不可信输入。`safeParse` 失败时不抛异常，而是返回错误结果。

4. `fetchQuote`

```ts
const inputResult = QuoteInputSchema.safeParse(input);

if (!inputResult.success) {
  return {
    ok: false,
    error: {
      code: "INVALID_INPUT",
      message: inputResult.error.issues.map((issue) => issue.path.join(".")).join(", "),
    },
  };
}
```

复习点：先校验输入，再发 HTTP。输入错了就直接返回 `INVALID_INPUT`，不应该真的请求外部服务。

```ts
const response = await fetch(new URL("/quote", env.SERVICE_BASE_URL), {
  method: "POST",
  headers: {
    authorization: `Bearer ${env.SERVICE_API_KEY}`,
    "content-type": "application/json",
  },
  body: JSON.stringify(inputResult.data),
});
```

复习点：这里演示的是一个可测试的 HTTP 边界。真实项目里可以替换成实际 API、RPC、MCP tool 或链上 SDK。

```ts
const body = (await response.json()) as unknown;
const outputResult = QuoteResponseSchema.safeParse(body);
```

复习点：HTTP 返回也不能直接信。返回体先当作 `unknown`，再用 `QuoteResponseSchema` 校验。

5. `nock` mock HTTP

```ts
const scope = nock("https://quote.example")
  .post("/quote", { service: "risk-check", amount: 5 })
  .reply(200, {
    price: "0.10",
    currency: "USDC",
    requiresApproval: false,
  });
```

复习点：`nock` 会拦截 HTTP 请求并返回假响应。测试不用真的访问网络，也不会因为外部服务不稳定而失败。

6. Vitest 测试结构

```ts
describe("module8-quality", () => {
  it("fetches quote successfully", async () => {
    const result = await fetchQuote(...);

    expect(result.ok).toBe(true);
  });
});
```

复习点：`describe` 是一组测试，`it` 是一个测试用例，`expect` 是断言。

7. 清理 mock

```ts
afterEach(() => {
  nock.cleanAll();
});
```

复习点：每个测试跑完要清理 HTTP mock，避免上一个测试影响下一个测试。

8. 质量命令

```json
{
  "scripts": {
    "typecheck": "tsc --noEmit",
    "test": "vitest run",
    "lint": "eslint .",
    "format:check": "prettier . --check"
  }
}
```

复习点：模块 8 的最终目标是形成固定检查命令。改代码后至少跑 `typecheck` 和 `test`，提交前再跑 `lint` 和 `format:check`。

### 这段示例代码在模拟什么

它在模拟一个最小 HTTP 工具调用：

- 读取配置。
- 校验输入。
- 调用 `/quote` API。
- 检查 HTTP 状态。
- 校验响应 JSON。
- 返回统一的 `AppResult`。

测试文件则模拟这些情况：

- 配置正确。
- 配置错误。
- HTTP 成功。
- 输入非法。
- HTTP 响应结构错误。

### 易错点

- 不要只测成功路径，失败路径更容易藏 bug。
- 不要让测试真的访问外网，HTTP 用 `nock` mock。
- 不要把 HTTP 响应直接当可信对象用。
- 不要忘记 `afterEach(() => nock.cleanAll())`。
- 不要把 `safeParse` 和 `parse` 混淆；`safeParse` 失败不会抛异常。
- 不要把 `ESLint` 和 `Prettier` 混为一谈：前者查问题，后者管格式。
