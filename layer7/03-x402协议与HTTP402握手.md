# 模块 2：x402 协议与 HTTP 402 握手

上一模块我们说清了分工：**x402 负责结算（钱真付了），AP2 负责授权（这笔钱被批准了）**。这一模块把 x402 的核心机制拆开——**一次 HTTP 往返里，机器怎么把钱付掉**。要回答的问题很具体：

> Agent A 请求 Agent B 的服务，B 怎么在一个 HTTP 响应里说清"要付多少、付给谁、付什么币、在哪条链"，A 又怎么在同一条 HTTP 通道里把钱付了、拿到资源？没有账号、没有 API key、没有月结账单。

答案藏在一个沉睡多年的 HTTP 状态码里：**402 Payment Required**。

> 目标：能讲清 x402 的 402 握手全流程，能读懂 `PaymentRequirements` 的每个字段（`scheme` / `network` / `amount` / `asset` / `payTo` / `maxTimeoutSeconds`），能说清 client 怎么拼 `PaymentPayload`（proof），并能亲手用 `pnpm inspect` 看到一个**真实服务返回的 402 + payment-required header**。

## 2.1 先讲清楚：402 状态码睡了 25 年，x402 把它叫醒了

HTTP 状态码里，`200 OK`、`404 Not Found`、`500` 你天天见。但 `402 Payment Required` 从 HTTP/1.1 时代就写在规范里，注释是一句"**Reserved for future use（保留待用）**"——整整二十多年，没有任何标准约定"收到 402 之后到底该怎么付"。它是一扇画在墙上的门。

为什么一直没用起来？因为**网上付款从来不是"一次 HTTP 往返"能搞定的事**：

```text
传统 API 付费的路径（对人友好，对 agent 极不友好）：
  注册账号 → 绑信用卡 → 拿 API key → 每月对账 → 人工处理失败扣款
  ↑ 每一步都要一个"人"和一个"账户关系"，都要提前建立、长期维持
```

对**人**来说这套流程能忍——你一年注册几十个账号。但对 **agent** 来说是灾难：一个 agent 可能要临时调用几百个它从没打过交道的服务，每个只用一次、每次只花几厘钱。为"调一次、花 $0.001 的服务"去注册账号、绑卡、等月结，成本比服务本身高几个数量级。

x402 做的事，就是把 402 从"保留待用"激活成一个**真正的机器对机器结算协议**：

```text
x402 的路径（对 agent 友好）：
  client 请求资源 → server 回 402 + "怎么付" → client 当场签一笔链上授权、带着重试
                 → facilitator 验证并结算 → server 放行返回资源
  ↑ 一次交互内完成，无账号、无 API key、无月结；付的是链上稳定币（USDC）
```

关键差别：**支付凭据不是"账户关系"，而是一笔当场签出的链上授权**。谁都能付、付完即走、不留账户。这正是 agent 经济需要的基础设施。

## 2.2 类比：x402 = 自动贩卖机，不是会员店

把传统 API 付费想成**会员制超市**：你得先办卡（注册）、绑定支付方式（信用卡）、每月结账（billing）。没有会员卡进不了门。这套东西为"长期、反复、有身份"的关系设计。

x402 更像**自动贩卖机**：

> 你走到贩卖机前，屏幕显示"这瓶水 3 元"（**402 + PaymentRequirements**）。你投币或刷一下（**签一笔授权、带着重试**）。机器确认收到钱（**facilitator 验证结算**），掉出一瓶水（**server 返回资源**）。全程不需要办会员卡、不需要留电话、不需要月底对账。你和这台贩卖机可能这辈子只打这一次交道。

这个类比里有三个要点，后面反复用到：

1. **报价写在响应里，不在合同里**。贩卖机把价格显示在屏幕上（402 响应携带 `PaymentRequirements`），不是让你先签一份服务合同。每次请求都能带全新的报价。
2. **一次交互闭环**。投币—出货是一个连续动作，不是"先建立账户、以后再扣款"。x402 的付款也发生在同一条 HTTP 请求-重试的链路里。
3. **付的是真钱、当场结清**。贩卖机收的是真硬币，不是欠条。x402 结算的是链上真 USDC（通过 EIP-3009 gasless 授权，下一模块细讲），facilitator 当场验证、上链。

## 2.3 402 握手全流程：ASCII 全景 + 每一步在做什么

x402 的一次完整握手是**四方五步**——client、server、facilitator，外加底层的链：

```text
  ┌────────┐                    ┌────────┐                 ┌─────────────┐
  │ client │                    │ server │                 │ facilitator │
  │(Agent A)│                   │(Agent B)│                │ (公共)       │
  └───┬────┘                    └───┬────┘                 └──────┬──────┘
      │                             │                             │
   ①  │ GET /premium （不带付款）    │                             │
      │────────────────────────────▶│                             │
      │                             │                             │
   ②  │ 402 Payment Required        │                             │
      │ + payment-required header   │  ← header 里是 b64 编码的     │
      │   (PaymentRequirements)     │     PaymentRequirements       │
      │◀────────────────────────────│    "要付多少/给谁/什么币/哪条链"│
      │                             │                             │
   ③  │ 从 requirements 里选一个，   │                             │
      │ 签一笔 EIP-3009 USDC 授权，   │                             │
      │ 拼成 PaymentPayload(proof)   │                             │
      │                             │                             │
   ④  │ GET /premium（带 proof 重试）│                             │
      │────────────────────────────▶│ 把 proof 交给 facilitator 验证│
      │                             │────────────────────────────▶│
      │                             │                        验证+结算(上链)
      │                             │◀────────────────────────────│
   ⑤  │ 200 OK + 付费资源            │  facilitator 说"钱到位" → 放行 │
      │◀────────────────────────────│                             │
      │                             │                             │
```

一步一步拆：

- **① 请求**：client 像访问普通 API 一样 `GET /premium`，不带任何付款。它可能根本不知道这是付费资源。
- **② 402 报价**：server 发现资源要收费，回 `402 Payment Required`，并在 **`payment-required` 响应头**（base64 编码）里附上一份 `PaymentRequirements`——这是整个协议的核心数据结构，明确告诉 client："付 `amount`，付给 `payTo`，用 `asset` 这个币，在 `network` 这条链，`scheme` 用 exact，`maxTimeoutSeconds` 秒内有效"。
- **③ client 拼 proof**：client 读懂 requirements，从中选一个它能满足的（一次报价可能给多个可选方案），用自己的钱包对一笔 **EIP-3009 授权**做 EIP-712 签名，拼成一个 `PaymentPayload`——这就是 x402 里说的 **proof（付款凭据）**。
- **④ 带 proof 重试 + facilitator 验证**：client 重新请求同一个资源，这次把 proof 放进请求头。server 不自己验，而是把 proof 交给 **facilitator**（公共的第三方验证/结算服务），由它校验签名、金额、防重放，并**把这笔授权上链结算**。
- **⑤ 放行**：facilitator 确认钱到位，server 返回 `200 OK` 和真正的付费资源。

> 一句话记住：**402 是"报价单"，proof 是"当场签的付款授权"，facilitator 是"收银台"，200 是"出货"。**

## 2.4 密集真实代码：PaymentRequirements 与拼 PaymentPayload

下面全是脚手架里**实测跑通**的代码，不是伪码。先看 server 侧在 402 里放的那份报价——`PaymentRequirements` 的类型定义（`payment-lab/src/shared.ts`，字段对齐 x402 v2 的 exact-evm scheme）：

```ts
// payment-lab/src/shared.ts
export interface PaymentRequirements {
  scheme: "exact";
  network: string; // CAIP-2，如 "eip155:84532" = Base Sepolia
  /** 原子单位金额（USDC 6 位小数：1000 = $0.001）。 */
  amount: string;
  /** 结算资产合约地址。 */
  asset: string;
  /** 收款地址。 */
  payTo: string;
  /** 授权有效秒数（quote 的时间窗）。 */
  maxTimeoutSeconds: number;
  extra?: { name?: string; version?: string; resourceUrl?: string };
}
```

server 用一个 fixture 造出一份具体报价（Base Sepolia + 测网 USDC + $0.001）：

```ts
// payment-lab/src/shared.ts
export function makeRequirements(overrides = {}): PaymentRequirements {
  return {
    scheme: "exact",
    network: NETWORK_BASE_SEPOLIA,        // "eip155:84532"
    amount: "1000",                        // $0.001（USDC 6 位小数）
    asset: USDC_BASE_SEPOLIA,              // 0x036CbD...F3dCF7e
    payTo: "0x1c47E9C085c2B7458F5b6C16cCBD65A65255a9f6",
    maxTimeoutSeconds: 300,
    extra: { name: "USDC", version: "2", resourceUrl: "https://svc.example/premium" },
    ...overrides,
  };
}
```

client 侧拿到这份 requirements，**选中它、签一笔 EIP-3009 授权、拼成 `PaymentPayload`**（`payment-lab/src/01-x402-handshake.ts`）——这个 `PaymentPayload` 就是要带回去的 proof：

```ts
// payment-lab/src/01-x402-handshake.ts
// client 侧：拿到一组 requirements，选一个，拼出 payment payload
function buildPayment(req: PaymentRequirements, from: string, now: number): PaymentPayload {
  const auth: Eip3009Authorization = {
    from,
    to: req.payTo,                          // ★ 收款方必须 = requirements 里的 payTo
    value: req.amount,                      // ★ 恰好付要求的金额
    validAfter: now - 1,
    validBefore: now + req.maxTimeoutSeconds, // ★ 用 quote 的时间窗算过期点
    nonce: "0x" + "a1".repeat(32),          // 教学固定 nonce；真实里是 32 字节随机，防重放
  };
  return {
    scheme: req.scheme,                     // "exact"，和 requirements 一致
    network: req.network,                   // 链一致
    asset: req.asset,                       // 币一致
    authorization: auth,
    signature: "0xEIP712_SIGNATURE_PLACEHOLDER", // 真实里是对 EIP-3009 结构的 EIP-712 签名
  };
}
```

注意 proof 的每个关键字段都**从 requirements 里"照抄"过来**：`to` 抄 `payTo`、`value` 抄 `amount`、`network`/`asset`/`scheme` 全对齐。这不是巧合——facilitator 验证时逐字段比对，任何一处不符（收款方被改、金额不足、链/币不对）都会被拒。把 demo 跑起来看这一步：

```bash
pnpm demo:handshake
```

它会打印 server 的 402 报价、client 拼出的 payment，并断言"requirements ↔ payment 一一对应"。

> **诚实标注**：`payment-lab` 是**本地不联网、不花钱、无私钥**的原理演示——`signature` 是占位字符串，EIP-3009 的 EIP-712 签名和真实结算不在这里发生。它的价值是把"402 里放什么、proof 怎么拼、字段怎么对齐"讲透。真正的签名和链上结算，由下一模块（EIP-3009）和模块 4 的 `paywalled-service`（真 SDK、真连 facilitator）完成。

## 2.5 真实体现：pnpm inspect 看到活的 402

上面是本地拼装的。x402 的 402 到底长什么样？`paywalled-service/src/inspect-402.ts` 是一个**不需要私钥、不花钱**的探针进程——它裸 `fetch` 一个真实 x402 服务的付费资源，把返回的 402 原样打印出来：

```ts
// paywalled-service/src/inspect-402.ts
const res = await fetch(`${SERVICE_URL}/premium`);   // 裸请求，不带任何付款
console.log(`HTTP 状态：${res.status} ${res.statusText}`);

if (res.status === 402) {
  console.log("✅ 如预期返回 402 Payment Required —— 这就是 x402 握手的第一步。\n");
  // x402 v2 把付款要求放在响应头 / body 里
  const relevantHeaders = ["payment-required", "www-authenticate", "content-type"];
  for (const h of relevantHeaders) {
    const v = res.headers.get(h);
    if (v) console.log(`  ${h}: ${v.length > 80 ? v.slice(0, 80) + "..." : v}`);
  }
  const body = await res.text();
  console.log("\n响应 body（付款要求，PaymentRequirements）：");
  console.log(body.slice(0, 600));
}
```

server 侧只用官方中间件挂一个付费闸门就能产出这个 402（`paywalled-service/src/server.ts`，模块 4 会细讲）：

```ts
// paywalled-service/src/server.ts
app.use(
  paymentMiddleware(
    { "GET /premium": { accepts: { scheme: "exact", price: "$0.001", network: NETWORK, payTo: PAY_TO } } },
    resourceServer,
  ),
);
```

跑起来（只需要 server 在跑，两条命令两个终端）：

```bash
pnpm start:server   # 终端 1：起付费服务
pnpm inspect        # 终端 2：探针，无需私钥
```

**实测**：`pnpm inspect` 真实返回 `402 Payment Required`，并带一个 `payment-required` 响应头——它是 base64 编码的 `PaymentRequirements`。把那段 base64 解出来，是这样的 JSON（`x402Version: 2` 明确标出这是 v2 协议）：

```json
{
  "x402Version": 2,
  "accepts": [
    {
      "scheme": "exact",
      "network": "eip155:84532",
      "maxAmountRequired": "1000",
      "asset": "0x036CbD53842c5426634e7929541eC2318f3dCF7e",
      "payTo": "0x....（你在 .env 里填的收款钱包）",
      "maxTimeoutSeconds": 300
    }
  ]
}
```

这就是握手的第一步落到实处：一个**真实、活的、可以自己复现**的 402。你看到的 `accepts` 是一个数组——server 可以一次给多个可选方案（不同链/币/价），client 挑一个能满足的。

> **诚实标注**：`pnpm inspect` 只跑到"看见真实 402"这一步——它**不需要私钥、不花钱**。握手后半段（选 requirement → 签 EIP-3009 授权 → 带 proof 重试 → facilitator 真结算）需要你在 `.env` 里填一个 **Base Sepolia 测试钱包私钥**、并领一点测网 USDC（<https://faucet.circle.com>），跑 `pnpm start:client` 才会真付一笔。这和 Layer 4"看得见、要真跑得自己填钱包"的边界一致。

## 2.6 对照表：PaymentRequirements（报价）与 PaymentPayload（proof）字段

| 字段 | 在 PaymentRequirements（server 报价） | 在 PaymentPayload（client proof） | 关系 / facilitator 怎么验 |
| --- | --- | --- | --- |
| `scheme` | `"exact"`：付准确金额的方案 | 照抄 `"exact"` | 必须相等，否则拒 |
| `network` | CAIP-2 链标识，如 `eip155:84532` | 照抄 | 必须相等（防跨链错付） |
| `amount` / `maxAmountRequired` | 要收的原子单位金额（`"1000"` = $0.001） | proof 的 `authorization.value` ≥ 它 | 金额不足即拒 |
| `asset` | 结算币合约地址（Base Sepolia USDC） | 照抄 | 必须相等（防用错币） |
| `payTo` | 收款地址 | proof 的 `authorization.to` = 它 | 收款方被改即拒（防改单） |
| `maxTimeoutSeconds` | 授权有效秒数（quote 时间窗） | 用它算 `validBefore = now + 它` | 过期即拒 |
| `nonce` | —（不在报价里） | proof 里 32 字节随机，防重放 | 重复 nonce 即拒（防重放） |
| `signature` | —（不在报价里） | 对 EIP-3009 结构的 EIP-712 签名 | 签名无效 = 伪造 proof，拒 |

一句话读表：**报价方（server）说条件，付款方（client）在 proof 里逐条满足，facilitator 逐条核对。** 这张表就是下一模块（EIP-3009 结算）和模块 5（校验/对账）反复要用的验证清单。

【学习提示】**别把 `amount` 当成"美元数"。** x402 报价里的金额是**原子单位**：USDC 是 6 位小数，`"1000"` 是 $0.001，`"1000000"` 才是 $1。server 端的中间件配置写的是人类友好的 `price: "$0.001"`（见 `server.ts`），SDK 会自动换算成 `"1000"` 这个原子单位塞进 `PaymentRequirements`。你在 lab 里直接看到的是换算后的 `amount: "1000"`。金额字段用**字符串**而非数字，是为了避免 JS 大整数精度丢失——`BigInt` 比较，一分不能差。

## 版本核验与边界（务必读）

- **x402 用 V2 的 scoped 包 `@x402/*`（实测 v2.17.0），不是旧的扁平 `x402-express` v1.x。** 分包是：`@x402/core`、`@x402/evm`、`@x402/express`（server 中间件）、`@x402/fetch`（client 包裹 fetch）。旧教程里的 `import { paymentMiddleware } from "x402-express"` 是 v1 写法，v2 已不适用。动手前以当前 npm 包为准。
- **network 一律用 CAIP-2**：Base Sepolia 是 `"eip155:84532"`，不是链名字符串 `"base-sepolia"`、也不是裸 chainId 数字。`inspect` 解出的真实 402 里 `network` 就是 `"eip155:84532"`，`x402Version` 是 `2`——这是判断"我连的是不是 v2 服务"的最直接证据。
- **公共 facilitator 不自建**：本层全程用公共测试 facilitator `https://x402.org/facilitator`（支持 Base Sepolia + Solana devnet），MVP 阶段不自建 facilitator。server 通过 `HTTPFacilitatorClient({ url })` 连它。
- **`payment-lab` 是原理演示，不是真结算**：本地不联网、`signature` 是占位、`nonce` 是固定值。真实签名/结算看模块 3（EIP-3009）和模块 4（`paywalled-service`）。
- **真付款要读者自己填钱包**：`pnpm inspect` 看 402 不需要私钥；`pnpm start:client` 真付一笔需要在 `.env` 填 Base Sepolia 测试钱包私钥 + 领测网 USDC。私钥只进 `.env`（已 `.gitignore`），只用测试网，绝不进前端/日志/仓库。
- **x402 治理与演进**：x402 V2 于 2025-12 发布（模块化拆包、多链、wallet sessions），2026-04 归入 Linux Foundation 下的 x402 Foundation（Google/Visa/Stripe/Mastercard/Circle 等 20+ 成员）。协议在快速演进，字段名/包结构以你动手时的当前版本为准。

## 本模块小结

- HTTP `402 Payment Required` 被保留了二十多年没启用，因为传统付费要"账号 + API key + 月结"，对"调一次、花几厘钱"的 agent 极不友好。x402 把 402 激活成**机器对机器的一次性结算协议**。
- 类比自动贩卖机：**报价写在响应里（402 + PaymentRequirements）、一次交互闭环、付的是链上真 USDC**——不办会员卡、不留账户。
- 402 握手四方五步：**① client 请求 → ② server 回 402 + payment-required header（PaymentRequirements）→ ③ client 选一个、签 EIP-3009 授权拼成 proof → ④ 带 proof 重试、facilitator 验证并上链结算 → ⑤ server 放行返回资源。**
- `PaymentRequirements` 是报价单：`scheme` / `network`(CAIP-2) / `amount`(原子单位) / `asset` / `payTo` / `maxTimeoutSeconds`。`PaymentPayload` 是 proof，字段从报价里"照抄"并逐条满足；facilitator 逐条核对（金额、收款方、链、币、时间窗、防重放 nonce、签名）。
- `pnpm inspect` 能**不花钱、无私钥**看到一个真实服务返回的 402 + `payment-required` header（b64 的 `PaymentRequirements`，解出 `x402Version: 2`）；真付款半段需自己填钱包 + 测网 USDC。
- 边界诚实标注：x402 是 v2 的 `@x402/*` 包不是 v1；`payment-lab` 是原理演示非真结算；network 用 CAIP-2；公共 facilitator 不自建；真付款要读者填测试钱包。

## 复习题
