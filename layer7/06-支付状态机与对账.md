# 模块 5：支付状态机与对账

上一模块把"付一次钱"跑通了。但产品级的 agent 支付不能停在"付没付"这个粗糙的抽象上——真实系统必须能回答：**这笔支付卡在哪一步？为什么失败？能不能重试或退款？这笔链上 tx 到底对应哪个任务、哪份授权？**

> 目标：把支付建模成一个可审计的**状态机**，并做一本**对账簿**，让 task / mandate / tx / 金额 相互印证；能处理重复通知、迟到回执、失败恢复。

## 6.1 先讲清楚：为什么"付没付"不够

设想一个只有布尔值 `paid: true/false` 的支付系统。第一天就会被这些问题打穿：

- 用户点了付款，但还没签授权——这算 `paid` 吗？
- 签了授权、提交了，但 facilitator 还没确认上链——算吗？
- 结算失败了，钱没动，但状态卡在中间——怎么让用户重试？
- 一笔付款结算成功了，但服务没交付——要退款，`paid` 这时候是 true 还是 false？

"付没付"表达不了这些。支付天然是**一串有序状态**，每一步都要能落到日志和 UI。

## 6.2 类比：快递物流状态

把支付想成**快递**：你不会只关心"到没到"，你关心"揽收 / 运输中 / 派送中 / 已签收 / 退回"。每个状态都有意义，且状态之间的跃迁是**有规则的**——不可能从"揽收"直接跳到"已签收"。支付状态机也一样。

## 6.3 支付状态机：合法路径与非法跃迁

看脚手架 `payment-lab/src/03-state-machine.ts` 的状态定义（实测跑通）：

```ts
type State =
  | "quote_pending"      // 拿到报价、还没授权
  | "awaiting_mandate"   // 高风险，等 AP2 mandate 签名
  | "authorized"         // 已授权（签了 EIP-3009 / 拿到 mandate）
  | "submitted"          // 已提交给 facilitator
  | "settled"            // 链上结算成功
  | "failed"             // 失败
  | "expired"            // quote / 授权过期
  | "refunded";          // 已退款

// 允许的状态跃迁（从 → 能到哪些）
const TRANSITIONS: Record<State, State[]> = {
  quote_pending:    ["awaiting_mandate", "authorized", "expired", "failed"],
  awaiting_mandate: ["authorized", "expired", "failed"],
  authorized:       ["submitted", "expired", "failed"],
  submitted:        ["settled", "failed"],
  settled:          ["refunded"],
  failed:           [],   // 终态
  expired:          [],   // 终态
  refunded:         [],   // 终态
};

function canTransition(from: State, to: State): boolean {
  return TRANSITIONS[from].includes(to);
}
```

ASCII 状态图：

```text
quote_pending ──┬─▶ awaiting_mandate ──▶ authorized ──▶ submitted ──▶ settled ──▶ refunded
                └──────────────────────▶ authorized                    │
   （任意非终态都可能）─────────────────────────────▶ expired / failed
```

小额走上面那条直线（`quote_pending → authorized → submitted → settled`）；高风险多一步 `awaiting_mandate`（等模块 6 的 AP2 mandate）。运行 `pnpm demo:state` 会看到合法路径 `✅` 通过、非法跃迁 `⛔` 被拒：

```text
✅ 通过 | quote_pending → authorized
✅ 通过 | authorized → submitted
⛔ 拒绝 | quote_pending → settled     （没结算就想 settled）
⛔ 拒绝 | submitted → refunded         （没 settled 不能退款）
⛔ 拒绝 | failed → settled             （终态不能复活）
```

**核心心智：** 状态机不是为了好看，是为了让"这笔支付现在能做什么、不能做什么"有唯一答案。非法跃迁被拒，等于把一整类 bug（比如"没结算却标记成功"）挡在门外。

## 6.4 对账：把 task / mandate / tx / 金额 串起来

支付成功后还有一件大事：**对账**。一条对账记录要能把这几样相互印证：

```text
task id（谁的任务）
  ↔ mandate id（哪份授权批的）
  ↔ tx hash（链上哪笔结算）
  ↔ 收款方 / 金额 / 币 / 链（对不对得上）
  ↔ 服务响应（真交付了吗）
```

看 `payment-lab/src/04-reconciliation.ts` 的对账簿（实测跑通）：

```ts
class Ledger {
  private entries = new Map<string, LedgerEntry>();
  private seenTx = new Set<string>();

  open(e: LedgerEntry): void {
    this.entries.set(e.taskId, e);
  }

  /** 收到一个结算回执（可能重复、可能迟到）。 */
  onReceipt(taskId: string, txHash: string): { ok: true; dup: boolean } | { ok: false; reason: string } {
    const e = this.entries.get(taskId);
    if (!e) return { ok: false, reason: `未知 taskId：${taskId}（迷失回执）` };
    if (this.seenTx.has(txHash)) return { ok: true, dup: true }; // 重复通知：幂等，不重复入账
    this.seenTx.add(txHash);
    e.txHash = txHash;
    e.state = "settled";
    return { ok: true, dup: false };
  }
}
```

## 6.5 三个必须处理的现实情况

对账最难的不是"正常那条路"，而是这些异常：

| 情况 | 问题 | 对账簿怎么处理 |
| --- | --- | --- |
| **重复通知** | facilitator / webhook 把同一笔回执发了多次 | 用 `seenTx` 去重，**幂等**——重复的识别出来、不重复入账 |
| **迟到回执** | 结算确认晚到（网络/区块延迟） | 状态机允许 `submitted` 停留，回执到了再 → `settled` |
| **迷失回执** | 收到一个不认识的 `taskId` 的回执 | 拒绝并告警，不能凭空入账 |

运行 `pnpm demo:recon` 会看到这三种都被正确处理：正常回执入账、同一 tx 重复到达识别为幂等、不认识的 taskId 被拒。

**要点：** 重复提交防护（幂等）不是可选项。支付回执的通道（webhook、轮询）天然会重复、会乱序、会迟到——对账簿必须假设这些都会发生。

## 6.6 收费模型与高频优化

一个付费服务至少要支持一种最小计费单位。常见模型：

| 模型 | 说明 | 适合 |
| --- | --- | --- |
| 按次 | 每次调用固定价（如 $0.001/次） | API、数据查询 |
| 按流量 | 按返回数据量计价 | 文件、大响应 |
| 按任务 | 一个复合任务一口价 | agent 服务 |
| 按成功结果 | 只在成功时收费 | 高不确定性任务 |

**高频优化：** 每次调用都上链结算，对高频 agent 太贵。x402 V2 引入了 **wallet sessions**——认证一次后用 session token，服务端**周期性批量结算**，不必每次调用都上链。高频场景（比如 agent 每秒调几十次推理 API）应该用它，而不是每笔都单独结算。

## 6.7 失败恢复与退款

产品级支付还要能处理"钱付了但服务没交付"：

- **失败重试**：`submitted → failed` 后，允许用一个新 quote 重新走一遍（旧的 nonce 已作废，不会重复扣钱）。
- **退款/争议**：`settled → refunded` 是唯一合法的退款路径——必须先确认结算成功，才谈退款。争议要留记录（task id / tx / 原因），可审计。

## 本模块小结

- 支付是**状态机**不是布尔值：`quote_pending → (awaiting_mandate) → authorized → submitted → settled`，可 `failed/expired/refunded`；非法跃迁必须被拒。
- 对账把 **task / mandate / tx / 金额 / 收款方 / 服务响应** 串成一条可审计记录。
- 重复通知要**幂等**、迟到回执要能等、迷失回执要拒绝并告警——这些异常是常态不是意外。
- 收费模型至少支持一种最小计费单位；高频调用用 x402 V2 的 wallet session 降低上链成本。
- 退款只能从 `settled` 出发；失败重试用新 quote，不重复扣钱。
