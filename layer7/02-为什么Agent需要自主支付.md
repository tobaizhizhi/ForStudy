# 模块 1：为什么 Agent 需要自主支付

Layer 6 结束时，我们的 agent 已经能做一件很像样的事：一个 agent 发现另一个 agent、验完对方的 Signed Agent Card、把任务交接过去、拿回结果。协作这一环打通了。

但有个洞一直没补上：

> Agent A 用了 Agent B 的服务，这一步"该付钱了"怎么办？

Layer 6 里我们默认了服务是白嫖的。可真实世界里，B 的服务背后是算力、是 API 配额、是数据订阅——**用了就得付钱**。而今天付钱这件事的每一步都是给**人**设计的：注册账号、绑信用卡、走月结账单、财务对账。让一个 agent 去做这些，要么根本走不通（没有人类的信用卡和身份），要么就得"每笔调用都停下来等人转一次账"——这**根本不 scale**。一个 agent 一天可能调上千次别的 agent，每次几厘钱，人工转账的方案在第一次就崩了。

> 目标：能说清 **AP2=授权（这笔钱被批准了）** 和 **x402=结算（钱真的付了）** 的分工，能讲清为什么 agent 支付**不是"转一次账"**而是一条产品级闭环，并知道本层为什么以 x402 为主、AP2 轻量处理。

## 1.1 先讲清楚：协作打通了，但"付钱"这一步是空的

回到北极星业务的场景：Agent A（你的助理 agent）需要一份链上数据分析，它通过 Layer 6 发现了 Agent B（一个数据分析 agent），验了 B 的卡、把任务派过去。B 干完活，返回结果。

然后呢？B 的这次服务标价 $0.001。谁来付、怎么付、付了怎么证明、A 的老板事后怎么查这笔钱花在哪了？

传统方案套不进来，因为它们都假设背后站着一个**人**：

```text
注册账号 / 绑卡        →  agent 没有身份证、没有信用卡，绑不了
API key + 月结账单     →  要人类事先开通、事后对账，agent 高频调用时账单是一团乱麻
人工审批每一笔转账      →  一天上千次调用、每次几厘钱，人根本审不过来，也没人愿意审
```

问题的本质是：**agent 之间的价值转移，必须由 agent 自己在协作的当下完成，且全程可审计。** 不能停下来等人，但也不能完全脱离人的授权失控地花钱。这就引出了两个必须分清的东西——**授权**和**结算**。

## 1.2 类比：收银台刷卡 vs 信用卡授权 / 老板签字

把一次线下消费拆开看，你就会发现"付钱"从来不是一个动作，而是**两个**：

> **x402 = 收银台一次刷卡结算。** 你在收银台把卡一刷，钱**真的**从你账户划到了商家账户——这一刻价值发生了转移，商家拿到了钱，交易完成。x402 干的就是这件事：一次 HTTP 往返里，稳定币**真的**从付款 agent 转到收款 agent。
>
> **AP2 = 信用卡授权 / 老板签字批准。** 一笔大额报销，不是财务想付就付——得先有老板在单子上**签字批准**："这笔钱，我批了，可以付。"签字本身**不动钱**，它只是一张"被授权"的凭证；真正划钱是后面收银台/银行做的事。AP2 干的就是签字这件事：它签发一张 **mandate（授权书）**，说明"谁授权付给谁、多少、什么币、哪条链、为哪个任务"。

两者是**接力**关系，不是二选一：老板先签字（AP2 授权）→ 财务拿着签过字的单子去收银台刷卡（x402 结算）。小额买瓶水不用老板签字，直接刷卡；大额报销才必须先拿到签字。agent 支付完全对应：小额自动走 x402，高风险大额先拿 AP2 mandate 再走 x402。

## 1.3 对照表：AP2 管"谁批准"，x402 管"钱真付了"

| | AP2（授权层） | x402（结算层） |
| --- | --- | --- |
| 一句话 | 这笔钱**被批准了** | 钱**真的付了** |
| 生活类比 | 老板签字 / 信用卡授权 | 收银台刷卡 / 银行划账 |
| 动不动钱 | **不动钱**，只签一张授权凭证 | **真的转账**，稳定币到账 |
| 产物 | 一张签名的 mandate（intent/cart/payment） | 一笔链上结算 + 交易回执（tx hash） |
| 绑定到 | 绑定到**用户**（不绑定具体 agent） | 绑定到一次具体的**付款**（金额/链/币/收款方） |
| 密码学 | ECDSA over P-256 + SHA-256，W3C VC 风格 | EIP-3009 `transferWithAuthorization`（USDC gasless） |
| 什么时候用 | **高风险 / 大额**，需要人类批准时 | **每一笔**结算都要它，钱靠它才真的动 |
| 本层处理 | **轻量**：概念 + 本地 mandate 演示（非真实 SDK） | **核心**：真连公共 facilitator、真跑 |

一句话记死："**AP2 决定这笔钱该不该付，x402 负责把钱真的付出去。**" 授权对了但没结算，钱没动；结算了但没授权，就是失控花钱——两者都要，且必须**绑定同一笔**（同一 task id、金额、币种、链、收款方，任一不符就拒绝，这条安全线在模块 5、6 会反复出现）。

## 1.4 为什么 agent 支付是产品级闭环，不是"转一次账"

新手最容易的误解是把 agent 支付想成"调个转账函数、把钱打过去"。真到了机器对机器、要可审计、能失败恢复的场景，一笔支付是**一条状态清晰、每步可查的闭环**：

```text
① 报价 quote      收款方回一张"怎么付"的单子（金额/链/币/收款方/有效期）
② 授权 authorize  付款方签一笔授权（小额=EIP-3009；高风险=先拿 AP2 mandate）
③ 结算 settle     facilitator 验证并上链，稳定币真的到账
④ 回执 receipt    拿到 tx hash / 结算凭证，证明"这笔真的付了"
⑤ 对账 reconcile  把 task ↔ mandate ↔ tx ↔ 金额 串起来，重复通知要幂等
⑥ 失败恢复        过期了能重新报价、验不过能拒、付错了能标记退款
```

这正好对应本层脚手架 `payment-lab/` 的六个练习，一个练习一环：`01-x402-handshake`（报价+拼授权）→ `02-payment-verify`（结算前的验证）→ `03-state-machine`（状态机流转）→ `04-reconciliation`（对账+幂等）→ `05-ap2-mandate`（高风险授权）→ `06-interop-rejection`（拒绝矩阵）。

其中"支付是**状态机**、不是一个布尔值"是本层反复强调的心智。看 `payment-lab/src/03-state-machine.ts` 里定义的合法状态：

```ts
// payment-lab/src/03-state-machine.ts
type State =
  | "quote_pending"     // 收到 402 报价，等付款
  | "awaiting_mandate"  // 高风险：等 AP2 mandate 签字
  | "authorized"        // 已授权，可以提交
  | "submitted"         // 已带 proof 提交给 facilitator
  | "settled"           // 上链结算成功
  | "failed"            // 失败（终态）
  | "expired"           // quote 过期（终态）
  | "refunded";         // settled 之后退款
```

"付了没付"是最粗的抽象。产品级支付要能回答的是"**这笔卡在哪一步、为什么、能不能重试或退款**"——`quote_pending` 卡住是对方还没付，`submitted` 卡住是 facilitator 还没结算完，`expired` 是报价超时了要重新报价。每个状态都要能映射到日志和前端 UI，这才是"可审计"的含义。

## 1.5 ASCII：一次完整付费调用的全流程

把上面六环画成一次真实调用（对应里程碑 Paywalled Agent Service）：

```text
        Agent A（付款）                              Agent B（收款/付费服务）
             │                                              │
             │  ① 请求 B 的付费服务 GET /premium            │
             │ ───────────────────────────────────────────▶│
             │                                              │  没带 payment
             │  ② 402 Payment Required                      │
             │     + PaymentRequirements                    │
             │     {scheme, network=eip155:84532,           │
             │      amount=1000, asset=USDC, payTo, 有效期}  │
             │ ◀───────────────────────────────────────────│
             │                                              │
   ┌─────────┴──────────┐                                   │
   │ 高风险大额？        │  是 → ③ 先拿一张签过的 AP2 mandate │
   │ (超阈值)           │       （老板签字：这笔我批了）     │
   └─────────┬──────────┘                                   │
             │  小额 → 跳过 mandate，直接授权                │
             │                                              │
             │  ④ 签 EIP-3009 USDC 授权（gasless）          │
             │     带 proof 重试 GET /premium               │
             │ ───────────────────────────────────────────▶│
             │                                    ┌─────────┴─────────┐
             │                                    │ ⑤ 交 facilitator  │
             │                                    │   验证：金额/链/币/ │
             │                                    │   收款方/时间窗/nonce│
             │                                    │   → 上链结算 USDC   │
             │                                    └─────────┬─────────┘
             │  ⑥ 200 OK + 付费内容 + 回执(tx hash)          │  验过才放行
             │ ◀───────────────────────────────────────────│
             │                                              │
      ⑦ 对账：task ↔ mandate ↔ tx ↔ 金额 串起来、幂等落账      │
```

一次刷卡的钱真的动了，一张签字的单子被归档，一条可查的记录进了对账簿——**这才叫一笔 agent 支付**，而不只是"调了个 transfer"。

【学习提示】**别把 x402 和 AP2 当成"两个可以二选一的支付方案"**——它们在不同的层，是接力不是竞争。x402 在**结算层**（HTTP 402 + 稳定币真转账），AP2 在**授权层**（签一张不动钱的 mandate）。90% 的小额调用**只需要 x402**（agent 用受限 session key 钱包直接自动付，见 Layer 4）；只有当金额/风险超过阈值，才在 x402 前面加一道 AP2 授权闸，让人类"签字批准"。把两者搞混，就会问出"到底该用 x402 还是 AP2"这种伪问题——答案是**小额只用 x402，大额 AP2 授权 + x402 结算**。

## 1.6 本层的取舍：x402 主线真跑，AP2 轻量演示

诚实说清本层怎么处理这两层，免得你对着脚手架产生错误预期：

- **x402 是本层核心，真连、真跑。** 隔壁 `paywalled-service/` 用真实的 `@x402/*` v2 SDK（server 侧 `@x402/express` 的 `paymentMiddleware`、client 侧 `@x402/fetch` 的 `wrapFetchWithPayment`），真连公共 facilitator `https://x402.org/facilitator`、真走 Base Sepolia 测网 USDC。实测 `pnpm inspect`（无需私钥）就能看到服务器返回真实的 `402 Payment Required` + `payment-required` header。**真付款那半段**（签 EIP-3009、facilitator 结算）需要你自己在 `.env` 填一个 Base Sepolia 测试钱包私钥 + 领测网 USDC（<https://faucet.circle.com>）才跑得通——这点和 Layer 4 一致。

- **AP2 是本层轻量处理，只到"概念 + 本地演示"。** 本课**不接真实 AP2 SDK**。`payment-lab/src/05-ap2-mandate.ts` 用 Node 内置 `crypto` 的 P-256（ECDSA + SHA-256）**手写演示** mandate 的签发与验证原理（对应 AP2 的真实密码学），让你理解"签一张不动钱的授权、验一张授权、高风险超阈值才要"的心智——但它**不是**真实的 JSON-LD Verifiable Credentials，也不是生产密码库。看 `shared.ts` 里的 `Mandate` 类型和 `signMandate` / `verifyMandate` 就能明白结构。

- **两者靠"绑定"接起来。** AP2 mandate 授权的金额/收款方/任务，必须和 x402 实际要结算的那笔**逐字段一致**，不符就拒绝（`verifyMandate` 的 `expect` 参数做的就是这件事）。真实世界里把 AP2 接到 x402 结算的是 **a2a-x402 extension**（Google + Coinbase + Ethereum Foundation + MetaMask 在做，已 production-ready），本课不展开它的 SDK，但心智一致。

接下来模块 2 就钻进 x402 的 402 握手——一次 HTTP 往返到底怎么完成机器对机器的支付。

## 版本核验（2026-07）

- **x402 用 V2 的 scoped 包 `@x402/*`（实测 v2.17.0），不是旧的扁平 `x402-express` v1.x。** 拆成 `@x402/core`、`@x402/evm`、`@x402/express`、`@x402/fetch` 等；network 用 **CAIP-2**（`eip155:84532` = Base Sepolia）。x402 V2（2025-12 发布）支持多链（Base / Solana / ETH / Polygon 等）、wallet sessions（认证一次拿 session token、周期结算，优化高频调用）、可插拔 legacy rails。**动手前以当前 npm 包为准。**
- **x402 治理**：2026-04 归入 Linux Foundation 下的 **x402 Foundation**，20+ 成员（Google / Visa / Stripe / Mastercard / Circle 等）。
- **结算底层是 EIP-3009 `transferWithAuthorization`**：稳定币 gasless 授权转账。USDC 是 6 位小数（`1000` 原子单位 = $0.001）；Base Sepolia USDC 合约 `0x036CbD53842c5426634e7929541eC2318f3dCF7e`；领测网 USDC：<https://faucet.circle.com>。
- **AP2 边界**：真实 AP2 的 mandate 是 **JSON-LD 的 W3C Verifiable Credentials**，用 **ECDSA over P-256 + SHA-256** 签名，分 Intent / Cart / Payment 三种；AP2 **只授权不动钱**，且**绑定到用户、不绑定 agent**。本课 `payment-lab` 里是**本地手写演示，不是真实 AP2 SDK**。
- **真付款需读者填钱包**：脚手架实测只保证到"服务器真实返回 402"（`pnpm inspect` 无需私钥）；付款结算半段需要你自己填 `.env` 测试钱包私钥 + 测网 USDC 才能跑通。

## 本模块小结

- Layer 6 打通了协作，但"用了服务要付钱"这一步是空的；传统的注册/绑卡/月结/人工审批都假设背后有个人，**agent 高频小额调用时人工转账根本不 scale**。
- **x402 = 结算层，钱真的付了**（收银台刷卡）；**AP2 = 授权层，这笔钱被批准了**（老板签字，不动钱）。两者是**接力**关系，不是二选一。
- 记死一句："AP2 决定这笔钱该不该付，x402 负责把钱真的付出去。" 授权和结算必须**绑定同一笔**（task id / 金额 / 币 / 链 / 收款方）。
- agent 支付**不是"转一次账"**，是一条闭环：**报价 → 授权 → 结算 → 回执 → 对账 → 失败恢复**，对应 `payment-lab` 的六个练习。
- **支付是状态机不是布尔值**：要能回答"卡在哪、为什么、能不能重试/退款"，每个状态映射到日志和 UI。
- 本层取舍：**x402 主线真跑**（`paywalled-service/` 真连公共 facilitator + Base Sepolia USDC），**AP2 轻量演示**（`payment-lab/` 本地 P-256 手写 mandate，非真实 SDK）；真付款需读者自填测试钱包。
