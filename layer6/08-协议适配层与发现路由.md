# 模块 7：协议适配层与发现路由

前面几章我们把「一张 Agent Card 怎么发现、怎么验签、怎么协商版本、怎么发任务」都拆开讲透了。这一章把它们**串成一条流水线**，并回答一个工程上绕不过去的问题：

> 现实里，同一个能力（比如「查 Base Sepolia 余额」）可能同时能用 **A2A**（对方是个 agent）、**MCP**（对方是个工具服务）、甚至**裸 HTTP**（对方是个老接口）触达。上层业务代码**不该硬编码**「我就走 A2A」——它应该把「用哪种协议」这个决策，交给一个专门的**路由器**。

这章的产出：一个统一的 **adapter 接口**（`discover / healthCheck / invoke`），A2A / MCP / HTTP 各实现一版；一个**路由器**，按「目标是工具还是 agent + 支持哪种协议」选出 adapter；再把它接进前面几章的**发现→验签→协商→撮合→派任务**全链路。

## 7.1 为什么需要一个「协议适配层」

先看没有适配层会怎样。假设你的网关代码里到处是这种硬编码：

```text
if (要查余额) {
  const client = new ClientFactory(...).createFromUrl(a2aUrl);  // 焊死 A2A
  ...
}
```

问题立刻来了：

- 那个「查余额」的能力**换了个供应方**，新供应方只暴露 MCP 工具、没有 A2A agent——你得改业务代码。
- 一个老系统只有裸 HTTP 接口，没上 A2A/MCP——你又得写一条 `if`。
- 你想加一条「A2A 连不上就退到 HTTP」的**降级**逻辑——散落在各处的 `if` 根本没法统一加。

**适配层的价值，就是把「协议差异」关进一个统一接口后面。** 上层只说「我要触达目标 T、干件事 X」，适配层负责「T 支持哪些协议、该选哪个、怎么按那个协议的方式把 X 发出去」。这和操作系统用「文件」这个统一抽象盖住「磁盘/网络/管道」是同一种思路。

```text
硬编码协议  =  业务代码里到处 if 协议；换供应方 / 加降级都要动核心逻辑  -> 脆、乱、不可审计
协议适配层  =  统一 adapter 接口 + 一个路由器；协议差异被封装，降级可集中加  -> 可换、可降级、可审计
```

## 7.2 统一 adapter 接口：三个方法

一个 adapter 要覆盖「触达一个目标」的三件事，正好三个方法：

| 方法 | 干什么 | 对应前面章节 | A2A 实现 | MCP 实现 | HTTP 实现 |
| --- | --- | --- | --- | --- | --- |
| `discover()` | 发现 + 拿到目标的描述（能力、端点、协议版本） | 模块 4/5 的 Agent Card 发现 | 抓 `/.well-known/agent-card.json` | `tools/list` 拿工具清单 | 约定的 `GET /manifest` 或直接已知 |
| `healthCheck()` | 目标现在活着吗、可达吗 | 模块 8 目录里的 `healthy` 位 | 卡能拉到即可达 | MCP `initialize` 握手成功 | `GET /health` 或 `HEAD` 返回 2xx |
| `invoke()` | 真正把一次调用/任务发出去，拿回结果 | 模块 6 的 `message/send` | `client.sendMessage(...)` | `client.callTool(...)` | `fetch(POST ...)` |

用 TypeScript 把接口定下来（这就是里程碑网关背后那层抽象的样子）：

```ts
type Protocol = "a2a" | "mcp" | "http";

/** 一次发现拿到的目标描述：三种 adapter 归一化到同一个形状。 */
interface Discovered {
  name: string;
  protocol: Protocol;
  endpoint: string;
  /** 能力标签，用于后续目录撮合（onchain-read 等）。 */
  capabilities: string[];
  /** A2A 才有：协议版本 + 是否带签名，交给验签/协商闸。 */
  protocolVersion?: string;
  raw?: unknown; // 原始描述（如 AgentCard），留给验签用
}

interface ProtocolAdapter {
  readonly protocol: Protocol;
  /** 发现：拿到目标的归一化描述。 */
  discover(endpoint: string): Promise<Discovered>;
  /** 健康检查：目标此刻是否可达。 */
  healthCheck(endpoint: string): Promise<boolean>;
  /** 调用：把一次请求发出去，拿回文本结果。 */
  invoke(endpoint: string, payload: { text: string }): Promise<string>;
}
```

三版实现的**核心动作**都来自你已经跑通的脚手架，这里只是把它们塞进统一接口里。A2A 版：

```ts
import { ClientFactory, JsonRpcTransportFactory } from "@a2a-js/sdk/client";
import { fetchAgentCard } from "./client.js"; // agent-network 里已有
import { v4 as uuidv4 } from "./uuid.js";

const a2aAdapter: ProtocolAdapter = {
  protocol: "a2a",
  async discover(endpoint) {
    const card = await fetchAgentCard(endpoint); // 抓 /.well-known/agent-card.json
    return {
      name: card.name,
      protocol: "a2a",
      endpoint,
      capabilities: card.skills.map((s) => s.name),
      protocolVersion: card.protocolVersion, // SDK 里是 "0.3.0"
      raw: card, // 原始卡留给验签闸
    };
  },
  async healthCheck(endpoint) {
    try {
      await fetchAgentCard(endpoint);
      return true;
    } catch {
      return false;
    }
  },
  async invoke(endpoint, { text }) {
    const client = await new ClientFactory({ transports: [new JsonRpcTransportFactory()] }).createFromUrl(endpoint);
    const result = await client.sendMessage({
      message: { kind: "message", messageId: uuidv4(), role: "user", parts: [{ kind: "text", text }] },
    });
    const reply = result as { parts?: { kind: string; text?: string }[] };
    return reply.parts?.filter((p) => p.kind === "text").map((p) => p.text).join("") ?? JSON.stringify(result);
  },
};
```

MCP 版把「握手 + 列工具 + 调工具」封进同样三个方法（动作来自 `a2a-agent.ts` 里的 `callMcpTool`）：

```ts
import { Client } from "@modelcontextprotocol/sdk/client/index.js";
import { StreamableHTTPClientTransport } from "@modelcontextprotocol/sdk/client/streamableHttp.js";

async function withMcp<T>(endpoint: string, fn: (c: Client) => Promise<T>): Promise<T> {
  const client = new Client({ name: "gateway-adapter", version: "0.1.0" });
  await client.connect(new StreamableHTTPClientTransport(new URL(endpoint)));
  try {
    return await fn(client);
  } finally {
    await client.close();
  }
}

const mcpAdapter: ProtocolAdapter = {
  protocol: "mcp",
  async discover(endpoint) {
    return withMcp(endpoint, async (c) => {
      const { tools } = await c.listTools(); // tools/list
      return { name: "mcp-server", protocol: "mcp", endpoint, capabilities: tools.map((t) => t.name) };
    });
  },
  async healthCheck(endpoint) {
    try {
      // connect() 内部就完成了 MCP initialize 握手，握上手即视为健康
      return await withMcp(endpoint, async () => true);
    } catch {
      return false;
    }
  },
  async invoke(endpoint, { text }) {
    // 这里为演示直接把 text 当成工具名+参数的约定；真实网关会先做意图解析
    return withMcp(endpoint, async (c) => {
      const result = await c.callTool({ name: "getBlockNumber", arguments: {} });
      const parts = (result.content ?? []) as { type: string; text?: string }[];
      return parts.filter((p) => p.type === "text").map((p) => p.text).join("");
    });
  },
};
```

HTTP 版是**兜底**：没有协议描述、没有握手，只有约定的路径：

```ts
const httpAdapter: ProtocolAdapter = {
  protocol: "http",
  async discover(endpoint) {
    // 裸 HTTP 没有标准发现口，退而求其次：约定一个 manifest，或直接返回最小描述
    return { name: endpoint, protocol: "http", endpoint, capabilities: [] };
  },
  async healthCheck(endpoint) {
    try {
      const res = await fetch(`${endpoint}/health`, { method: "GET" });
      return res.ok;
    } catch {
      return false;
    }
  },
  async invoke(endpoint, { text }) {
    const res = await fetch(endpoint, {
      method: "POST",
      headers: { "content-type": "application/json" },
      body: JSON.stringify({ text }),
    });
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return await res.text();
  },
};
```

注意三版的**能力差异是诚实的**：A2A 的 `discover` 能带回签名和版本（能进验签/协商闸），MCP 能带回工具清单，HTTP 几乎啥都带不回来——所以 HTTP 只配当 fallback，这一点在下一节的路由偏好里会直接体现出来。

## 7.3 路由器：目标是 agent 还是工具，支持哪些协议

有了三个 adapter，**谁来决定用哪个？** 这就是路由器。它的输入是「目标的画像」——**是 agent 还是 tool、实际支持哪些协议**；输出是「选中的协议 / 或拒绝」。

路由偏好一句话：**agent 优先 A2A，工具优先 MCP，都不行退到 HTTP fallback。** 这段纯逻辑就是 `protocol-lab/src/04-protocol-router.ts`，已带自断言跑通：

```ts
type Protocol = "a2a" | "mcp" | "http";

interface Target {
  name: string;
  kind: "agent" | "tool";       // 目标是另一个 agent（用 A2A）还是工具服务（用 MCP）
  supports: Protocol[];         // 这个目标实际支持哪些协议
}

// 路由偏好：agent 优先 A2A、工具优先 MCP，都不行退 HTTP。
function route(target: Target): { ok: true; protocol: Protocol } | { ok: false; reason: string } {
  const preference: Protocol[] =
    target.kind === "agent" ? ["a2a", "http"] : ["mcp", "http"];
  for (const p of preference) {
    if (target.supports.includes(p)) return { ok: true, protocol: p };
  }
  return { ok: false, reason: `目标 ${target.name} 不支持任何可用协议（${target.supports.join(", ") || "无"}）` };
}
```

对照它在 `runDemo()` 里那四个目标，路由结果一目了然：

| 目标 | kind | 实际支持 | 偏好序 | 选中 | 说明 |
| --- | --- | --- | --- | --- | --- |
| `chain-reader-agent` | agent | a2a, http | a2a → http | **A2A** | agent 首选 A2A，命中 |
| `mcp-chain-server` | tool | mcp | mcp → http | **MCP** | 工具首选 MCP，命中 |
| `legacy-price-agent` | agent | http | a2a → http | **HTTP** | 是 agent 但不支持 A2A，**降级到 fallback** |
| `broken-service` | tool | （无） | mcp → http | **⛔拒绝** | 一个可用协议都没有 |

关键点：**偏好列表里 HTTP 永远在最后**。agent 的偏好是 `["a2a","http"]`——先试 A2A，A2A 不支持才退 HTTP；工具的偏好是 `["mcp","http"]`——先试 MCP。HTTP 永远是「实在没别的了」的兜底，绝不会抢在 A2A/MCP 前面。这正对应上一节说的：HTTP adapter 能力最弱，只配兜底。

运行本地 lab 看它跑：

```bash
pnpm demo:router
```

## 7.4 发现路由全链路：把前几章串起来

现在把「选协议」放回**完整决策链**里。一次「请求方想安全地把任务派给某个目标」，要顺次过这些闸——每一闸都在前面章节讲过，这里是它们的**装配顺序**：

```text
                    ┌─────────────────────────────────────────────┐
   一个请求进来 ──▶  │  路由器：目标是 agent 还是 tool？支持哪些协议？ │
                    └───────────────┬─────────────────────────────┘
                                    │ 选出 adapter（agent→A2A / tool→MCP / 兜底→HTTP）
                                    ▼
              ┌──────────────── adapter.discover() ────────────────┐
              │  拿回归一化描述：能力 / 端点 / protocolVersion / raw │
              └───────────────┬────────────────────────────────────┘
                              ▼
   闸①  Card 结构校验 ── 字段齐不齐（模块 4）
                              ▼
   闸②  JWS 验签 ────────── verifySignedCard：签名对不对、卡有没有被篡改（模块 5）
                              ▼
   闸③  公钥可信 ────────── 这把公钥是不是我认的（模块 5；教学里内联，生产查注册表）
                              ▼
   闸④  版本协商 ────────── protocolVersion ∈ 支持集？不兼容则降级或拒绝（模块 6）
                              ▼
   闸⑤  健康检查 ────────── adapter.healthCheck(endpoint)：端点此刻可达吗
                              ▼
   闸⑥  能力匹配 ────────── 目录撮合：能力 / 链 / 预算 / 信誉（模块 8）
                              ▼
   闸⑦  任务路由 ────────── adapter.invoke(endpoint, payload)：真正派任务（模块 6）
                              ▼
                          拿回结果
```

**顺序不是随便排的**，几条硬规则：

- **验签（闸②③）必须在派任务（闸⑦）之前**。没验签就 `invoke`，等于把任务交给一个可能冒充的端点——这是全层的安全灵魂。
- **健康检查（闸⑤）在能力匹配（闸⑥）之前**过滤。一个挂掉的 agent，信誉再高也不该被撮合进来（对应 `05-service-directory.ts` 里那个 `chain-reader-C-down`：`healthy:false`，信誉 99 也被筛掉）。
- **任何一闸失败都不进入下一闸**。坏卡在闸②就被拦下，根本走不到 `invoke`。

## 7.5 真实体现：网关里的 discoverAndVerify + HTTP fallback

上面这条链，在里程碑网关 `agent-network/src/gateway.ts` 里是**真跑通**的。它的 `discoverAndVerify` 把「版本协商 + JWS 验签」两闸压在派任务之前，任何一闸不过就**返回 null → 触发 HTTP fallback 或拒绝**：

```ts
const SUPPORTED_VERSIONS = ["0.3.0", "1.0"]; // SDK 实现 0.3.x；同时声明能协商到 1.0 概念

async function discoverAndVerify(baseUrl: string): Promise<AgentCard | null> {
  const card = await fetchAgentCard(baseUrl);

  // 闸 1：版本协商 —— 不兼容就不硬闯 A2A，交给上层 HTTP fallback
  if (!SUPPORTED_VERSIONS.includes(card.protocolVersion)) {
    console.log(`[gateway] ⛔ 版本不兼容：${card.protocolVersion} → 尝试 HTTP fallback`);
    return null;
  }

  // 闸 2：JWS 验签 —— 验不过一律拒绝，绝不派任务
  const v = verifySignedCard(card);
  if (!v.ok) {
    console.log(`[gateway] ⛔ Agent Card 验签失败：${v.reason} → 拒绝，不派任务`);
    return null;
  }
  console.log(`[gateway] ✅ ${card.name} 验签通过、版本 ${card.protocolVersion} 可协商`);
  return card;
}
```

注意这里的 **fallback 决策是显式、可见、可审计的**：版本不兼容时，网关不是默默失败，而是打印一条「→ 尝试 HTTP fallback」，把「我从 A2A 降级到了裸 HTTP」这件事**记进日志**。这正是安全验收里那条硬要求——「协议降级（A2A→HTTP fallback）要显式、可见、可审计」。降级不是丢人的事，**偷偷降级**才是。

验签通过后，网关才继续做目录撮合（能力 / 链 / 信誉，其中 `reputation` 来自 `getReputation()` 这个 **ERC-8004 可插拔口**），再 `dispatch` 派任务。而错误回放那段——把一张篡改了 `url` 的卡喂进 `verifySignedCard` 会 ⛔ 拒绝——恰恰证明了「验签必须在派任务之前」这条闸是真的拦得住冒充端点。

跑一次完整链路（需先起好 mcp-server + a2a-agent）：

```bash
pnpm start:gateway
```

【学习提示】**路由是纯逻辑，最适合先在本地 lab 跑通，再接真进程。** `route()` 不依赖网络、不依赖 SDK、不需要 API key——它就是「输入目标画像、输出选中协议」的一个纯函数。先用 `pnpm demo:router` 把「agent→A2A / tool→MCP / 兜底→HTTP / 全不支持→拒绝」这四种情况的断言全跑绿，你就把**路由决策**这块心智焊死了；这时再去看 `gateway.ts` 里 `discoverAndVerify` 怎么把它接到真实的验签、真实的 A2A 派任务上，就只剩「把纯逻辑接上 IO」这一步，不会被网络问题和协议逻辑搅在一起。**先纯逻辑、后接 IO**，是调这类协议代码最省心的顺序。

## 7.6 版本核验（2026-07）

- **A2A adapter 的 `invoke` 用 `ClientFactory + createFromUrl + client.sendMessage`**，这是 `@a2a-js/sdk`（实测 **0.3.14**，实现 spec **v0.3.0**）当前的 API。传输工厂是 `JsonRpcTransportFactory`；Agent Card 默认路径 `/.well-known/agent-card.json`；卡里 `protocolVersion` 是字符串 `"0.3.0"`。**概念上 A2A 协议已发 v1.0**（11 个 JSON-RPC 方法、三种 binding、任务状态枚举改 SCREAMING_SNAKE_CASE 等破坏性变更），但**写代码以 SDK v0.3.x 为准**——这是本层反复强调的「概念领先、实现追赶」落差，路由到 A2A 时按 SDK 的方法名写。
- **MCP adapter 的 `Client / StreamableHTTPClientTransport / connect / listTools / callTool`** 来自 `@modelcontextprotocol/sdk`（实测 **1.29.0**）。导入路径带 `.js`：`@modelcontextprotocol/sdk/client/index.js`、`.../client/streamableHttp.js`。传输统一用 **StreamableHTTP**（SSE 已废弃，别用）。`connect()` 内部完成 MCP `initialize` 握手，所以「握上手」可直接当健康信号。
- **路由器本身不依赖任何 SDK**——`route()` 是 `protocol-lab/src/04-protocol-router.ts` 里的纯函数，不联网、无网络调用。它不会随 SDK 升级而变，是本章最稳的一块。
- **HTTP adapter 的 `/health`、`/manifest` 路径是本教学的约定**，不是任何标准强制的。裸 HTTP 本就没有 A2A/MCP 那样的标准发现口，所以它能力最弱、只当 fallback；接真实老系统时按对方实际暴露的路径改。

## 本模块小结

- 现实里一个能力可能同时能用 A2A / MCP / 裸 HTTP 触达，上层业务**不该硬编码协议**；协议差异应关进统一 adapter 接口后面。
- 统一 adapter 三方法：`discover()`（发现 + 归一化描述）、`healthCheck()`（此刻可达吗）、`invoke()`（真正派一次调用）；A2A / MCP / HTTP 各实现一版，动作分别来自 `client.ts` 的发现验签、`a2a-agent.ts` 的 MCP 调工具、和裸 `fetch`。
- 三版能力不对等：A2A 能带回签名+版本，MCP 能带回工具清单，HTTP 几乎啥都带不回——所以 HTTP 只配兜底。
- 路由偏好：**agent 优先 A2A、工具优先 MCP，都不行退 HTTP fallback**；偏好列表里 HTTP 永远在最后（`protocol-lab/src/04-protocol-router.ts`）。
- 发现路由全链路 = 路由选 adapter → discover → 结构校验 → JWS 验签 → 公钥可信 → 版本协商 → 健康检查 → 能力匹配 → 任务路由；**验签必须在派任务之前**，健康检查在能力匹配之前。
- 里程碑网关 `gateway.ts` 的 `discoverAndVerify` 真跑通了「版本+验签两闸不过 → 返回 null → HTTP fallback / 拒绝」；**降级必须显式、可见、可审计**，偷偷降级才是问题。
- 路由是纯逻辑，先本地 lab（`pnpm demo:router`）跑通断言，再接真进程（`pnpm start:gateway`）接 IO。
