# 模块 1：为什么 Agent 需要通信协议 —— MCP vs A2A 的心智

这一模块不写代码，先把心智建起来。因为 Layer 6 后面所有名词（Agent Card、JWS 验签、任务/消息、服务目录）都挂在一个前提上：**agent 之间要通信，就必须有共同语言。** 想清楚「为什么要有协议」「MCP 和 A2A 各管什么」，后面读规范会轻很多。

> 目标：能一句话说清 MCP 和 A2A 的分工，能画出「人→agent→工具/链」和「agentA⇄agentB」两张不同的图，能解释为什么 agent 协作不能靠「每对 agent 私下约定接口」。

## 1.1 先讲清楚为什么：手、工具，然后是共同语言

回顾一下 agent 到这一步长成了什么样：

- Layer 4 给了它一只**受限的手**：session key 钱包，能自主花钱，但花得住手（白名单、限额、可撤销）。
- Layer 5 给了它**大脑和工具**：用 MCP 把链上能力封装成 tool，用 workflow 编排决策。

到这里，一个 agent 已经能**独立完成一整件事**了。但它是**一座孤岛**：它会用自己的工具，却不会跟**别的 agent** 打交道。而真实的 agent 经济，价值恰恰在协作——一个擅长查链上数据的 agent，一个擅长做市场分析的 agent，一个擅长执行交易的 agent，应该能**互相发现、把任务交接给对方**，而不是每个 agent 都把所有能力重造一遍。

那「让 A 把任务交给 B」难在哪？难在**没有共同语言时，协作是 O(n²) 的**：

```text
没有协议：每一对 agent 都要私下约定一套接口
   A↔B 写一套胶水   A↔C 写一套胶水   B↔C 写一套胶水  ...
   n 个 agent = n×(n-1)/2 套私有胶水，任何一方改字段全线崩
   新来的 agent D 想接入 = 要跟已有每个 agent 分别对接

有协议：所有 agent 说同一套话
   A / B / C / D 都遵守 A2A，任意两方按标准发现、验签、派任务
   新来的 D 只要发布一张标准 Agent Card，全网都能跟它协作
```

这就是**为什么要有通信协议**：协议把 O(n²) 的私有胶水，压成 O(n) 的「大家都遵守同一份标准」。互联网能长成今天这样，靠的就是 TCP/IP、HTTP、TLS 这些谁都不用私下商量的公共协议。Agent 网络要长大，同样需要它自己的那几份协议——这一层要学的 **MCP**、**A2A**、**ERC-8004** 就是。

## 1.2 银行卡类比：MCP 是功能菜单，A2A 是跨行清算

这一层最容易混的，是 MCP 和 A2A **都叫「协议」，但管的完全是两件事**。用银行来类比一次，就分清了：

> **MCP 像你和自己银行 App 之间的「功能菜单」。** 你（agent）打开自己银行的 App（工具服务），菜单上列着「查余额、转账、看流水」这些**功能**。你点一个功能，App 执行、把结果给你。这是**一个主体调用它自己能用的一批工具**——`agent ↔ 工具`。

> **A2A 像银行之间的跨行清算网络（SWIFT）。** 当**你的银行**要把一笔钱汇到**另一家银行**，两家银行是**对等的机构**，谁也不是谁的「功能菜单」。它们需要一套**跨机构的标准**：怎么互相识别（这家银行是真的吗）、用什么报文格式、怎么确认到账。这是**两个对等主体之间的协作**——`agent ↔ agent`。

一句话钉死：

```text
MCP：agent 调它的工具   —— 单向、我用你，像点 App 菜单
A2A：agent 找 agent 协作 —— 对等、你我平级，像跨行清算
```

**为什么非要分成两个协议、不能用一个？** 因为这两件事的性质根本不同。你调自己银行 App 的功能，不需要「验证 App 是不是冒充的、跟它协商协议版本、把它当成一个能拒绝你的独立主体」——它就是你的工具。但要把钱汇给**另一家银行**，你必须先确认对方**不是冒充的**（验签）、双方**说得上话**（版本协商）、对方**可能拒绝或收费**（对等性）。A2A 天生要背这些「跨主体信任」的负担，MCP 不用。硬把两者合成一个，要么 MCP 被塞进一堆它不需要的信任机制，要么 A2A 缺了跨主体协作的关键部件。

## 1.3 两张图：谁调谁，一眼看清

把上面那句话画出来。第一张是 MCP 的世界——**人在顶上，agent 用工具，工具碰链**：

```text
        人（下指令 / 审批高风险动作）
          │
          ▼
   ┌──────────────┐
   │    agent     │  有大脑(L5) + 受限的手(L4)
   └──────┬───────┘
          │  MCP（agent↔工具，单向调用）
          ▼
   ┌──────────────┐
   │  MCP server  │  getBalance / getBlockNumber / getTxCount
   └──────┬───────┘
          │  viem 只读
          ▼
     Base Sepolia 链
```

这条链是纵向的、有层级的：人指挥 agent，agent 调工具，工具读链。每一层都「用」下一层。第二张是 A2A 的世界——**两个 agent 平级，互相发现、协作**：

```text
   ┌──────────────┐                    ┌──────────────┐
   │   agent A    │  ⇄  A2A（对等） ⇄  │   agent B    │
   │  （请求方）   │                    │  （提供方）   │
   └──────────────┘                    └──────┬───────┘
        发现 B 的 Agent Card                   │
        验 B 的签名（不是冒充的）                │ B 内部再用 MCP
        协商版本、派任务                        ▼
                                        ┌──────────────┐
                                        │  MCP server  │
                                        └──────────────┘
```

注意第二张图的关键：**A 和 B 之间是双向箭头（对等）**，但 **B 拿到任务后，转身又用 MCP 去调自己的工具**。这正是本层里程碑要跑通的真实链路——`agent-network/src/a2a-agent.ts` 里那个 `ChainReaderExecutor`，对外用 A2A 接任务，对内用 MCP client 调 `mcp-server` 的工具：

```text
client ──A2A派任务──▶ agent B.execute() ──MCP──▶ mcp-server ──viem──▶ Base Sepolia
                              │                                          │
       真实链上余额 ◀─────────┴──────────  原路返回  ────────────────────┘
```

**同一个 agent，对外是 A2A 的对等节点，对内是 MCP 的工具使用者。** 这就是两个协议「各司其职、一起用」的样子。

## 1.4 MCP vs A2A 边界对照表

把区别摊开成一张表，这是本模块要记住的核心：

| 维度 | MCP（Model Context Protocol） | A2A（Agent-to-Agent） |
| --- | --- | --- |
| 连接谁和谁 | agent ↔ 工具 / 数据源 | agent ↔ agent |
| 谁调谁 | agent 调工具（单向：我用你） | 两个 agent 对等协作（你我平级） |
| 有无对等性 | 无。工具是被调用方，不会「反过来找 agent」 | 有。对方是独立主体，可拒绝、可收费、可反向发起 |
| 发现方式 | 连上 server 后 `tools/list` 列它的工具 | 读对方的 **Agent Card**（`/.well-known/agent-card.json`）+ **验签** |
| 信任模型 | 你信任你连的这个 server | 必须**验证对方身份**（JWS 签名），默认不信任 |
| 版本协商 | 传输/协议层握手 | **A2A-Version** header 协商，不兼容用新 URI |
| 有无长任务状态 | 一次 tool 调用，请求—响应 | 有**任务生命周期**（提交/进行中/完成/失败），可流式 |
| 本层对应 SDK | `@modelcontextprotocol/sdk`（实测 v1.29.0） | `@a2a-js/sdk`（实测 v0.3.14，实现 spec v0.3.0） |
| 类比 | 你和自己银行 App 的功能菜单 | 银行之间的跨行清算（SWIFT） |

【学习提示】最常见的两个用反：**别拿 A2A 当工具协议**（给 agent 装个能力，用 MCP，不用给它发 Agent Card、验签）；**也别拿 MCP 当 agent 发现协议**（要找一个能协作的对等 agent、要验它身份，用 A2A，MCP 里没有「验对方是不是冒充」这回事）。判断口诀：**目标是一个「工具/数据源」还是一个「能拒绝你的对等 agent」？** 前者 MCP，后者 A2A。

## 1.5 埋一颗种子：上层不该硬编码协议，而是「选协议」

上面那个口诀，在本层的 `protocol-lab/src/04-protocol-router.ts` 里被写成了一段能跑的路由逻辑。它做的正是「看目标是什么，选 MCP 还是 A2A 还是退到 HTTP」：

```ts
type Protocol = "a2a" | "mcp" | "http";

interface Target {
  name: string;
  /** 目标是另一个 agent（用 A2A）还是一个工具服务（用 MCP）。 */
  kind: "agent" | "tool";
  supports: Protocol[]; // 这个目标实际支持哪些协议
}

// 路由偏好：agent 优先走 A2A，工具优先走 MCP，都不行退到 HTTP。
function route(target: Target): { ok: true; protocol: Protocol } | { ok: false; reason: string } {
  const preference: Protocol[] =
    target.kind === "agent" ? ["a2a", "http"] : ["mcp", "http"];
  for (const p of preference) {
    if (target.supports.includes(p)) return { ok: true, protocol: p };
  }
  return { ok: false, reason: `目标 ${target.name} 不支持任何可用协议` };
}
```

跑一下看它怎么判：

```bash
cd /home/lenovo/solidity-course/ata/layer6/protocol-lab
pnpm demo:router
```

你会看到「agent 目标 → 选 A2A」「tool 目标 → 选 MCP」「只支持 HTTP 的旧 agent → 退到 HTTP fallback」「无可用协议 → ⛔拒绝」。**这段代码就是本模块口诀的可执行版本**，也是里程碑网关 `gateway.ts` 的核心决策逻辑。现在只要建立心智即可，具体路由/降级到模块 7 再展开。

## 1.6 本层三件套：MCP + A2A + ERC-8004

MCP 和 A2A 解决了「怎么调工具」和「怎么找对等 agent」，但还差一块：**A 凭什么信任 B？** 验签能证明「这张卡确实是持有某把私钥的人签的」，但「这把私钥背后的 agent 靠不靠谱、历史信誉如何」，签名本身答不了。这就是第三件套 **ERC-8004** 的位置——把 agent 的**链上身份和可携带信誉**接进来。三者分工：

```text
MCP       —— agent 怎么用工具（接工具 / 数据源）
A2A       —— agent 怎么找 agent（发现 + 验签 + 版本协商 + 任务交接）
ERC-8004  —— agent 是谁、信誉如何（链上身份 + 可携带信誉，可插拔）
```

本层 **MCP + A2A 是必学核心**；**ERC-8004 本层只讲概念和接入点**（在服务目录里作为可插拔的信誉来源，见 `gateway.ts` 里 `getReputation()` 那个口子），真正落到链上留到 Capstone。这是一个诚实边界：别把项目核心逻辑焊死在 ERC-8004 当前的注册表接口上——它还是 draft。

## 1.7 本层不替换 L4/L5，是在它们之上加一层「对外协作」

最后钉一个容易误解的点：**Layer 6 不推翻前面两层，是在它们之上加了一层「对外协作层」。**

```text
       ┌─────────────────────────────────────────────┐
Layer6 │  对外协作：A2A 发现/验签/交接 · MCP 深化 · ERC-8004 身份  │  ← 本层新增
       ├─────────────────────────────────────────────┤
Layer5 │  大脑和工具：MCP 造工具 · workflow 编排决策            │
       ├─────────────────────────────────────────────┤
Layer4 │  受限的手：session key 钱包（自主花钱 · 花得住手）       │
       └─────────────────────────────────────────────┘
```

L4 的钱包、L5 的工具和大脑一个都不变，agent 内部照旧用它们干活。Layer 6 加的是**朝外的那一面**：让这个已经会独立干活的 agent，能被别的 agent 发现、验证、并安全地交接任务。一句话——**前面几层让 agent 强，这一层让 agent 们连成网。**

## 1.8 一个诚实边界：A2A 概念已 v1.0，SDK 还在 v0.3.x

本模块讲的是心智，还没碰代码，但有个贯穿全层的落差必须先亮出来：

- **A2A 协议已经发布 v1.0**，但官方 JS SDK（`@a2a-js/sdk`，实测 v0.3.14）当前实现的还是 **spec v0.3.0**。
- 所以本层的规矩是：**讲概念按 v1.0**（Signed Agent Card、11 个 JSON-RPC 方法、三种 binding、破坏性变更），**写代码按 SDK 当前的 v0.3.x API**（`agent-network` 里卡的 `protocolVersion` 就是 `"0.3.0"`）。
- MCP 侧相对稳：`@modelcontextprotocol/sdk` 实测 v1.29.0（v1.x 生产支持版），传输统一用 **StreamableHTTP**（SSE 传输已废弃）。

这不是 bug，是「协议跑在 SDK 前面」的真实状态。每章涉及处都会标注，动手前以当前 npm 包为准。这种「概念领先、实现追赶」的落差，本身就是 2026 年做 agent 协议要有的心理准备。

## 本模块小结

- **为什么要协议**：agent 有了手（L4）和工具（L5）后，协作是价值所在；没有共同语言，每对 agent 都要写私有胶水，是 O(n²)；有了标准协议，压成 O(n)。
- **MCP = 功能菜单**（agent↔工具，单向、我用你，像点自己银行 App）；**A2A = 跨行清算**（agent↔agent，对等、可拒绝可收费，像 SWIFT）。
- 分成两个协议是因为**性质不同**：A2A 天生要背「验对方身份、协商版本、对等协作」的信任负担，MCP 不用。
- 判断口诀：目标是「工具/数据源」用 MCP，是「能拒绝你的对等 agent」用 A2A；`04-protocol-router.ts` 是这句口诀的可执行版。
- **同一个 agent 可以两面用**：对外是 A2A 对等节点，对内是 MCP 工具使用者（里程碑的 `ChainReaderExecutor` 就是）。
- 本层三件套：**MCP**（接工具）+ **A2A**（找 agent，核心）+ **ERC-8004**（身份/信誉，可插拔、本层只讲概念）。
- Layer 6 **不替换** L4/L5，是在其上加「对外协作层」。
- 诚实边界：A2A 概念按 v1.0、代码按 SDK v0.3.x；MCP 传输用 StreamableHTTP、SSE 已废弃。
