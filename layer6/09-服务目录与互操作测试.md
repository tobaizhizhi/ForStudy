# 模块 8：服务目录与互操作测试

前一模块我们把「拿到一个目标，按协议路由到 MCP / A2A / HTTP」这条 adapter 讲通了。但那假设了一个前提：**你已经知道要找哪个 agent**。真实的 agent 网络不是这样——上线的 agent 越来越多，请求方手里往往只有一句「我要一个能读 Base Sepolia 余额、免费、靠谱的 agent」，它得先**找到人**，才谈得上路由和派任务。

这一模块解决两件事：

> 1. **服务目录（service directory / registry）**：agent 世界的「黄页」，按能力 + 链 + 预算 + 健康 + 信誉把候选撮合出来。
> 2. **互操作测试（interop test）**：用一批坏输入压测发现链路的健壮性——未签名 / 验签失败 / 公钥不可信 / 错版本 / 过期 / 伪造 endpoint，**全部要被拒绝**。这是本层安全验收的收口。

## 8.1 为什么需要「黄页」+「压测」

先讲清动机，两条线各自的痛点：

**动机一：多 agent 上线后要有撮合。** 一个能力（比如「链上只读」）可能有五个 agent 都声称会做，它们的链、收费、健康状态、信誉各不相同。请求方不该硬编码一个 URL，而应该**声明需求，让目录撮合**：

```text
请求方说的是「需求」不是「地址」：
  需要能力 = onchain-read
  需要链   = base-sepolia
  预算     = 免费（0）
       │
       ▼
  服务目录：过滤 + 排序 → 撮合出一个（或几个）候选 agent
```

**动机二：撮合出来的候选，卡是真的吗？** 目录里那条记录的 endpoint、能力、信誉，都是 agent **自报**的。真正派任务前，必须回到发现闸门重新验一遍卡。而「验得够不够严」这件事，不能靠人肉审，要用一张**拒绝矩阵**自动压测：喂各种坏卡进去，断言系统全部⛔。

> 类比：服务目录像点评网站的「按菜系 + 价位 + 距离 + 评分」筛餐厅；互操作测试像食品安全抽检——你不能因为它评分高就信它，得拿标准坏样本去测它守不守规矩。**评分高但健康检查挂了的 agent，照样不派任务给它**——这条在下面的 `search()` 里就是硬规则。

## 8.2 轻量 registry 的数据模型

一条目录记录就是一个 agent 的「名片摘要」。看 `protocol-lab/src/05-service-directory.ts` 里的类型定义：

```ts
interface DirectoryEntry {
  name: string;
  url: string;
  capabilityTags: string[];   // 能力标签，如 ["onchain-read"]
  chains: string[];           // 支持的链，如 ["base-sepolia"]
  /** 每次调用报价（教学字段）；0 = 免费。 */
  pricePerCall: number;
  healthy: boolean;           // 健康检查结果（来自模块 7 的 health check）
  /** 信誉分 0–100。生产里应来自 ERC-8004 Reputation 注册表，而非 registry 自报。 */
  reputation: number;
}
```

每条记录 = **agent（谁）+ 能力（会什么）+ 链（在哪条链上）+ 收费（多少钱）+ 健康（活着吗）+ 信誉（靠谱吗）**。这六个维度正好对应请求方会问的六个问题。

对照到本层其它模块，这些字段各有来源，不是凭空填的：

| 字段 | 来源 | 谁说了算 |
| --- | --- | --- |
| `name` / `url` | Agent Card 的 `name` / `url`（模块 4） | agent 自报，**派任务前要验签** |
| `capabilityTags` | Agent Card `skills[].tags`（模块 4） | agent 自报 |
| `chains` | Agent Card `chains`（模块 4 教学字段） | agent 自报 |
| `pricePerCall` | Agent Card `pricing`（教学字段） | agent 自报 |
| `healthy` | 模块 7 的 health check 结果 | **目录方实测**，不是自报 |
| `reputation` | 现在是 fixture；生产来自 ERC-8004（模块 9） | **不该由被查方自吹** |

【学习提示】注意 `healthy` 和 `reputation` 这两列的「谁说了算」——它们**不能由 agent 自报**。`healthy` 必须是目录方主动探活（模块 7 的 health check）得来的实测值；`reputation` 自报就等于「让考生自己填分」，生产里必须换成可携带、可验证的外部信誉源（ERC-8004）。这条区分是目录能不能防作弊的关键。

## 8.3 撮合逻辑：过滤 + 排序

`search()` 是整个目录的核心，就一条流水线——**先用硬条件过滤，再按信誉软排序**：

```ts
interface Query {
  needCapability: string;
  needChain: string;
  maxPrice: number;
}

function search(dir: DirectoryEntry[], q: Query): DirectoryEntry[] {
  return dir
    .filter((e) => e.healthy)                              // 健康过滤：挂了的直接淘汰
    .filter((e) => e.capabilityTags.includes(q.needCapability)) // 能力匹配
    .filter((e) => e.chains.includes(q.needChain))        // 链匹配
    .filter((e) => e.pricePerCall <= q.maxPrice)          // 预算匹配（价 ≤ 预算）
    .sort((a, b) => b.reputation - a.reputation);         // 信誉高的排前面
}
```

流水线的顺序不是随意的，健康过滤放最前：

```text
全部候选
   │  .filter(healthy)          ← 硬淘汰：健康检查挂的，信誉再高也出局
   ▼
活着的
   │  .filter(能力匹配)          ← 硬淘汰：不会这个能力的出局
   ▼
会这个能力的
   │  .filter(链匹配)            ← 硬淘汰：链不对的出局
   ▼
链也对的
   │  .filter(价 ≤ 预算)         ← 硬淘汰：超预算的出局
   ▼
预算内的
   │  .sort(信誉降序)            ← 软排序：留下来的按信誉排，取头部
   ▼
撮合结果（信誉高的在前）
```

**「过滤」和「排序」是两种性质完全不同的操作**：过滤是「达不到硬门槛就出局」（能力、链、预算、健康都是一票否决）；排序是「达标者之间比高下」（信誉只在幸存者里决定先后）。别把信誉也做成过滤门槛——那样会把「信誉 80、免费、健康」的 agent 挡在「信誉 99 但挂了」后面，逻辑就反了。

跑 `pnpm demo:directory`，用一份带各种「陷阱」的 fixture 演示每条淘汰规则：

```text
查询：能力=onchain-read，链=base-sepolia，预算=0（免费）

撮合出 2 个候选（按信誉降序）：
✅ 通过 | chain-reader-A  信誉=92  价=0
✅ 通过 | chain-reader-B  信誉=78  价=0

被筛掉的：
  chain-reader-C-down —— 健康检查失败（信誉=99 也不派任务给挂掉的 agent）
  eth-mainnet-reader  —— 链不匹配（要 base-sepolia，它是 ethereum）
  premium-analytics   —— 超预算（要免费，它要 50）
```

看清楚那条 `chain-reader-C-down`：它信誉 99（全场最高），但 `healthy: false`，被健康过滤在第一步就淘汰了。**「不给挂掉的 agent 派任务」比「优先高信誉」优先级更高**——这正是把健康过滤放在流水线最前、且做成硬淘汰的原因。

## 8.4 互操作测试矩阵：坏卡一律拒绝

目录只负责「撮合出候选」，它**不替你验卡**。撮合命中后、派任务前，必须回到模块 5 的发现闸门，把整条链路走一遍。`protocol-lab/src/06-interop-rejection.ts` 就是这条完整闸门 + 一张自断言的拒绝矩阵。

先看完整闸门顺序（和本层安全验收逐条对齐）：

```ts
function fullDiscoveryCheck(card, ctx, requestedVersion) {
  // 闸 1：结构校验（缺字段 / url 非法 → 拦）
  const shape = validateCardShape(card);
  if (!shape.ok) return shape;

  // 闸 2：JWS 验签（卡被篡改 / 未签名 → 挂）
  const sig = verifyCard(card);
  if (!sig.ok) return sig;

  // 闸 3：公钥可信（挡住「攻击者用自己 key 自签」的有效 JWS）
  if (!ctx.trustedPublicKeys.has(card.signature.publicKeyPem)) {
    return { ok: false, reason: "签名公钥不在可信名单（自签卡：JWS 有效但签发者不可信）" };
  }

  // 闸 4：版本协商（用 client 请求的版本，不是卡里印的版本）
  const ver = negotiateVersion(requestedVersion);
  if (!ver.ok) return ver;

  // 闸 5：过期检查
  if (card.expiresAt !== undefined && card.expiresAt < ctx.now) {
    return { ok: false, reason: "Agent Card 已过期" };
  }

  return { ok: true };
}
```

这五道闸缺一不可，尤其**闸 3 公钥可信**最容易被漏掉：攻击者完全可以用**自己的** key 生成一张 JWS 完全合法的卡——闸 2 会放行，因为签名和内容确实自洽。挡住它的唯一办法是把公钥拿去和可信名单（DID / JWKS / 未来的 ERC-8004 身份注册表）比对。**「验签通过」不等于「签发者可信」，这是初学者最常见的安全误区。**

`06-interop-rejection.ts` 把 8 条测试用例喂进这套闸门，每条都带 `expectOk`，**跑完自己断言实际结果和预期一致**——这就是这套 lab 的「测试」。8 条对照表：

| # | 坏输入（喂进去的卡） | 期望结果 | 被哪道闸拦下 | 对应安全验收 |
| --- | --- | --- | --- | --- |
| 1 | 合法卡（结构全 + owner 签 + 版本 1.0 + 未过期） | ✅ 放行 | 全过 | 合法卡必须能协作 |
| 2 | 缺字段（`url` 置 undefined） | ⛔ 拒绝 | 闸 1 结构 | 结构非法一律拒绝 |
| 3 | 被篡改（`url` 改成 evil.example） | ⛔ 拒绝 | 闸 2 验签 | 验签失败必须拒绝 |
| 4 | 未签名卡（没有 signature） | ⛔ 拒绝 | 闸 2 验签 | 未签名必须拒绝 |
| 5 | 攻击者自签（JWS 有效但公钥不可信） | ⛔ 拒绝 | 闸 3 公钥可信 | 签发者不可信必须拒绝 |
| 6 | 版本不兼容（请求 0.3，server 只支持 1.0） | ⛔ 拒绝 | 闸 4 版本协商 | 版本不兼容必须拒绝 |
| 7 | 过期卡（`expiresAt < now`） | ⛔ 拒绝 | 闸 5 过期 | 过期必须拒绝 |
| 8 | 伪造 endpoint（`url` 指向 169.254.169.254 内网元数据） | ⛔ 拒绝 | 闸 1 结构 | 伪造 endpoint 必须拒绝 |

跑 `pnpm demo:interop`，末尾会打印断言汇总：

```text
断言：8/8 条与预期一致
全部符合预期：合法卡放行，坏卡（缺字段/篡改/未签/自签/错版本/过期/伪造端点）全部拒绝。
这正对应本层安全验收：未签名 / 验签失败 / 错版本 / 过期 / 伪造 endpoint 一律拒绝。
```

**这个 `8/8` 才是重点。** 它把「安全验收」从一句口号变成了可执行的断言：任何一条闸门逻辑被改坏（比如某天有人手滑注释掉了闸 3），这个断言立刻从 8/8 掉下来，`process.exitCode = 1`，CI 直接红。**互操作测试的价值，是让「坏卡一律拒绝」这条纪律变成回归测试，而不是靠人记得住。**

【学习提示】第 8 条那个 `169.254.169.254` 不是随便挑的地址——它是云环境（AWS / GCP 等）的**元数据服务地址**，SSRF 攻击的经典目标。如果目录不校验 endpoint 就让 agent 去连它自报的 url，一张伪造卡就能诱导你的服务去打内网元数据接口偷凭证。本 lab 里它被结构闸的 url 规则拦下；生产还要再加**出站地址白名单 / 内网段黑名单**，这是纵深防御。

## 8.5 真实体现：gateway 里的目录撮合

前面两节是本地 lab（`protocol-lab/`，不联网、纯断言）。到了真实三进程的里程碑网关 `agent-network/src/gateway.ts`，目录撮合和坏卡拒绝**合到了一条端到端决策链**里。看它的正常路径：

```ts
// 闸门：先发现 + 验签 + 版本协商，卡不合格直接返回 null
const card = await discoverAndVerify(A2A_BASE_URL);
if (card) {
  // 轻量服务目录撮合（按能力 + 链 + 信誉）
  const entry: DirectoryEntry = {
    name: card.name,
    baseUrl: A2A_BASE_URL,
    capability: "onchain-read",
    chain: CHAIN,
    reputation: getReputation(A2A_BASE_URL), // ← ERC-8004 可插拔口
  };
  console.log(`[gateway] 目录撮合命中：${entry.name}（能力=${entry.capability} 链=${entry.chain} 信誉=${entry.reputation}）`);
  await dispatch(A2A_BASE_URL, `查一下 ${DEMO_ADDRESS} 在 Base Sepolia 的余额`);
}
```

注意顺序：**先 `discoverAndVerify`（发现 + 验签 + 版本协商），验过了才组装 `DirectoryEntry` 去撮合**。目录记录里的 `name`、`url` 全都取自**已验签**的 card——绝不用未验证的自报数据去派任务。

`reputation` 这个字段网关用了一个可插拔的口子：

```ts
/** ERC-8004 可插拔口：现在读 fixture，未来换成链上 Reputation 注册表查询。 */
function getReputation(_agentUrl: string): number {
  return 88; // fixture
}
```

这一层抽象很关键：**今天 `getReputation` 返回一个写死的 fixture，明天换成查 ERC-8004 Reputation 注册表，`gateway` 主流程一行都不用动。** 这就是大纲里「身份信誉可插拔」原则的落地——别把核心撮合逻辑焊死在信誉源的当前实现上。

网关的错误回放段，则把 8.4 的拒绝矩阵浓缩成一条真实断言——喂一张篡改过 `url` 的卡，验签必须⛔：

```ts
const good = await fetchAgentCard(A2A_BASE_URL);
const tampered: AgentCard = { ...good, url: "http://attacker.example/a2a" };
const v = verifySignedCard(tampered);
console.log(`[gateway] 篡改 url 后验签：${v.ok ? "✅（不该通过！）" : "⛔ 拒绝 —— " + v.reason}`);
console.log(`[gateway] 决策：拒绝该卡，不把任务派给可能是伪造端点的 agent。`);
```

跑 `pnpm start:gateway`（需要 `start:mcp` 和 `start:agent` 已在跑），能看到网关：发现真实 A2A agent → 验签✅ → 目录撮合命中 → 派任务、agent 内部调 MCP 查到 Base Sepolia 真实余额并返回 → 最后把篡改卡⛔拒绝。**撮合、验签、错误回放，在一条链上跑完，可审计。**

## 8.6 边界与诚实标注

这套目录 + 测试是**教学实现**，几处边界必须说清，别当生产架构用：

**（1）目录是内存态，生产要持久化。** `protocol-lab` 和 `gateway` 里的目录都是进程内的数组 / fixture，进程一重启就没了。生产的服务目录要落到数据库 / 分布式注册表，还要考虑一致性、并发写、TTL 过期清理。本课程只做到「看懂撮合逻辑」这一层。

**（2）注册需要反女巫（anti-Sybil）。** 内存 lab 里 `DirectoryEntry` 是我们手填的。真实目录允许 agent 自助注册时，攻击者可以批量注册一堆高信誉马甲刷榜、或注册同名 agent 钓鱼。生产要有准入成本（质押 / 身份绑定）、去重、签名注册，这些都超出本层范围。

**（3）`reputation` 现在是 fixture，生产必须换可信信誉源。** 这是最重要的一条诚实标注，`05-service-directory.ts` 和 `gateway.ts` 都在注释里写死了同一句话：

```text
reputation 目前是 registry 自报的 fixture。
生产里它应来自 ERC-8004 Reputation 注册表——可携带、可验证、不由被查方自吹。
```

**「让被查方自己报信誉分」在安全上等于零**——谁都会给自己打 100。信誉必须来自一个**外部的、可验证的、可携带的**源：由第三方基于真实交互反馈累积，签名背书，任何人都能独立核验。这正是模块 9（ERC-8004）要接的活。`getReputation()` 那个可插拔口，就是为这次替换预留的接缝：函数签名不变，实现从「返回 fixture」换成「查链上注册表」。

**（4）撮合维度是精简版。** 真实目录还会考虑延迟、地理位置、SLA、并发容量、历史成功率、协议版本兼容度等。本课程取「能力 / 链 / 预算 / 健康 / 信誉」五维，是为了把「过滤 + 排序」的骨架讲清楚，不是维度上限。

## 版本核验（2026-07）

- 本模块的目录撮合和拒绝矩阵**全部在 `protocol-lab/` 本地跑**（`pnpm demo:directory` / `pnpm demo:interop`），不联网、不调 SDK 网络接口、无 API key——`search()` 和 `fullDiscoveryCheck()` 是纯函数逻辑，不依赖任何协议 SDK 版本。
- `gateway.ts` 用到的 A2A 客户端是 `@a2a-js/sdk@0.3.14`（实现 A2A spec **v0.3.0**）；`ClientFactory` + `createFromUrl`、`sendMessage`、Agent Card 默认路径 `/.well-known/agent-card.json` 都是这个版本的 API。**A2A 协议本身已发 v1.0，SDK 还停在 v0.3.x**——这是本层反复强调的落差：讲发现闸门、拒绝矩阵这些**概念按 v1.0**（`protocolVersion` 概念、11 方法、破坏性变更都是 v1.0 的），**写代码按 SDK v0.3.x**（网关里 `SUPPORTED_VERSIONS = ["0.3.0", "1.0"]` 就是这个妥协：既接受 SDK 真实产出的 `"0.3.0"` 卡，也声明能协商到 `"1.0"` 概念）。
- `reputation` 字段的可信来源 **ERC-8004 仍是 draft**，接口可能变。所以 `getReputation()` 做成可插拔口，正是为了不把架构押死在它当前的注册表接口上。模块 9 讲它的三注册表和接入点。
- `169.254.169.254` 是云元数据服务地址（AWS/GCP 等长期稳定使用），拿它做 SSRF 反例不受版本影响。

## 本模块小结

- 多 agent 上线后要有「黄页」：请求方声明**需求**（能力 + 链 + 预算），目录**撮合**出候选，而不是硬编码 URL。
- `DirectoryEntry` 六维：agent / 能力 / 链 / 收费 / 健康 / 信誉。其中 `healthy` 是目录方**实测**、`reputation` 生产**不该自报**。
- `search()` = **硬过滤（能力 / 链 / 预算 / 健康，一票否决）+ 软排序（信誉降序，只在幸存者间比高下）**；健康过滤放最前，信誉再高也救不了挂掉的 agent。
- 撮合出候选 ≠ 可信。派任务前必走完整发现闸门：**结构 → 验签 → 公钥可信 → 版本协商 → 过期**。
- 互操作测试矩阵（8 条自断言）把「坏卡一律拒绝」变成**回归测试**：任一闸门被改坏，断言从 8/8 掉下来、CI 变红。
- 闸 3「公钥可信」最易漏：**验签通过 ≠ 签发者可信**，攻击者能用自己 key 签一张 JWS 合法的卡。
- `gateway.ts` 把撮合 + 验签 + 错误回放串成一条可审计决策链；`reputation` 走 `getReputation()` 可插拔口。
- 边界：目录内存态（生产要持久化 + 反女巫）；`reputation` 现为 fixture，生产必须换成 ERC-8004 这类**可携带、可验证、非自报**的信誉源。
