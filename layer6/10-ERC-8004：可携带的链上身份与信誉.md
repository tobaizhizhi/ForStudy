# 模块 9（轻量）：ERC-8004 —— 可携带的链上身份与信誉

前面八个模块，我们把 A2A 发现、JWS 验签、版本协商、服务目录都跑通了。但目录里有一个字段我们一直在“作弊”：**reputation（信誉）**。

看 `agent-network/src/gateway.ts` 里目录撮合那段——它按能力、链、信誉给 agent 打分。可这个 `reputation: 88` 是哪来的？答案是写死的 fixture：

```ts
/** ERC-8004 可插拔口：现在读 fixture，未来换成链上 Reputation 注册表查询。 */
function getReputation(_agentUrl: string): number {
  return 88; // fixture
}
```

这一模块要回答的就是：**这个 88 应该从哪来，能不能跨平台带走。** 这正是 ERC-8004 要解决的问题。

> 本模块是**轻量概念章**：只讲 ERC-8004 是什么、三个注册表各存什么、它怎么作为可插拔身份/信誉层接回本层目录。真正的合约实现与链上落地留到 **Capstone**。

## 9.1 为什么：信誉不该锁死在单个平台里

先想清楚今天的痛点。一个 agent 在平台 A 上干了 100 单活、拿了 4.9 分好评。现在它想去平台 B 接活——它的 100 单历史、4.9 分，能带走吗？

```text
今天（信誉锁在平台数据库里）：
  agent 在平台 A 攒的信誉  ──X──▶  换到平台 B = 从零开始
  平台 A 说了算、平台 A 能改、平台 A 关门信誉就没了

ERC-8004（信誉上链）：
  agent 的身份/信誉/验证记录都在链上注册表
  换平台不清零、平台改不了、跨链能迁（以太坊 → Base / Optimism / Gnosis…）
  任何撮合方（比如我们的 gateway）都能直接查、无需信任某个平台
```

一句话：**A2A 和 MCP 解决“agent 怎么说话”，ERC-8004 解决“凭什么信这个 agent”。** 它是 A2A 刻意留白的那一层——身份、信誉、验证。

## 9.2 三个注册表：各存什么

ERC-8004 把“信任”拆成三个链上单例注册表（每条链各部署一份）。Identity 是锚，另外两个都引用它、写入前先问它“调用者真有权代表这个 agent 吗”。

| 注册表 | 存什么 | 回答的问题 | 底层机制 |
| --- | --- | --- | --- |
| **Identity Registry**（身份） | 每个 agent = 一个 ERC-721 NFT，`tokenURI` 指向 Agent Card / 注册文件 | 这个 agent **是谁**？ | ERC-721 + URIStorage；`agentId` = tokenId |
| **Reputation Registry**（信誉） | 交互后由客户/其他 agent 提交的评分、反馈，链上可查、可组合 | 凭什么**信**它的产出？ | 按 reviewer / tag 过滤；复杂聚合放链下 |
| **Validation Registry**（验证） | agent 声称“我干了 X”，validator 合约给出可追踪的验证结论 | 它**真的做**了它声称的事吗？ | stake 重跑推理 / zkML / TEE 预言机 |

【学习提示】三者的成熟度不一样。Identity 最稳（就是标准的 ERC-721 扩展）；Reputation 定的是“怎么把反馈写上链”，聚合算法留给链下；**Validation Registry 最不定型**——它还在和 TEE 社区讨论，后续 spec 会改。写代码时优先只依赖 Identity + Reputation 的读接口，别把架构押在 Validation 的当前形状上。

## 9.3 核心机制：agent = 一个 NFT

ERC-8004 最妙的一步，是把 agent 的身份做成 **ERC-721 NFT**（Identity Registry 继承 ERC-721 + URIStorage）。这一个决定带来三个白捡的好处：

```text
每个 agent = Identity Registry 里的一个 ERC-721 tokenId
        │
        ├─ tokenURI(tokenId) ──▶ 一个 JSON（agent 的注册文件 / Agent Card）
        │                         里面有 name / description / endpoints / 钱包地址…
        │
        ├─ 天然兼容整个 NFT 生态：现成的钱包、浏览器、indexer 直接能列出、能看
        │
        └─ 天然兼容 token-bound account（ERC-6551）：
             NFT 自己可以“拥有”一个账户 → agent 的身份直接绑定一个可持资产的钱包
```

**可携带 = 信誉不清零、能跨链迁。** 因为身份是链上 NFT、信誉是链上记录，它们不属于任何单个平台。agent 从以太坊主网迁到 Base / Optimism，或者同时在多条链上被撮合，历史都还在。这就是“**portable reputation**（可携带信誉）”这个词的分量。

注意一个关键的呼应：ERC-8004 的 `tokenURI` 指向的那个 JSON，和我们模块 4 讲的 **A2A Agent Card 是同一类东西**——都是“描述这个 agent 是谁、能干什么、在哪找它”的元数据。ERC-8004 正是构建在 A2A 之上的：A2A 的 Agent Card 负责“能力和消息”，ERC-8004 在它下面补上“身份和信任”。

## 9.4 从 tokenId 到 Agent Card 到 A2A 发现

把这条链路画出来，你就看清 ERC-8004 是怎么“喂”给我们已经写好的 A2A 发现流程的：

```text
   ┌─────────────────────┐
   │ Identity Registry   │  (链上 ERC-721 单例)
   │   tokenId = 42      │
   └──────────┬──────────┘
              │ tokenURI(42)
              ▼
   ┌─────────────────────┐        ┌─────────────────────────┐
   │  https://.../42.json│        │ Reputation Registry     │
   │  = Agent 注册文件   │◀──关联──│  tokenId 42 的评分/反馈 │
   │  ≈ A2A Agent Card   │        └─────────────────────────┘
   │  { name, url,       │
   │    endpoints, … }   │
   └──────────┬──────────┘
              │ 拿到 card.url
              ▼
   ┌─────────────────────────────────────────────┐
   │  本层已实现的 A2A 发现流程（模块 4–8）：     │
   │  fetchAgentCard → JWS 验签 → 版本协商 → 派任务 │
   └─────────────────────────────────────────────┘
```

链上注册表给出“谁可信、信誉多少”，链下 A2A 给出“怎么发现、怎么派任务”。**两者接缝就在那个 JSON——它既是 NFT 的 `tokenURI`，也是 A2A 的 Agent Card。**

一个只读示意：用 viem 读某个 agent 的 `tokenURI`（概念示意，注册表地址与 ABI 以 Capstone / 官方参考实现为准，这里不接进脚手架跑）：

```ts
import { createPublicClient, http } from "viem";
import { baseSepolia } from "viem/chains";
import { RPC_URL } from "./config.js";

// ⚠️ 概念示意：Identity Registry 是标准 ERC-721，读法就是 tokenURI(tokenId)。
//    真实地址 / 完整 ABI 见官方参考实现（github.com/erc-8004/erc-8004-contracts）。
const IDENTITY_REGISTRY = "0x...";           // Capstone 里填真实部署地址
const erc721TokenUriAbi = [
  {
    type: "function",
    name: "tokenURI",
    stateMutability: "view",
    inputs: [{ name: "tokenId", type: "uint256" }],
    outputs: [{ name: "", type: "string" }],
  },
] as const;

const publicClient = createPublicClient({ chain: baseSepolia, transport: http(RPC_URL) });

// 只读、不写链、不碰私钥 —— 和本层所有链交互一个原则
const agentCardUri = await publicClient.readContract({
  address: IDENTITY_REGISTRY,
  abi: erc721TokenUriAbi,
  functionName: "tokenURI",
  args: [42n],                                // 某个 agent 的 tokenId
});
// agentCardUri 就是那张 Agent Card 的 URL，接下来交给已有的 fetchAgentCard 流程
```

拿到 `agentCardUri` 之后，后面全是我们已经写好的东西：`fetchAgentCard(url)` → `verifySignedCard(card)` → 版本协商 → 派任务。ERC-8004 只是在最前面加了一个“**链上身份来源**”。

## 9.5 接回本层目录：把 fixture 换成链上查询

现在回到开头那个写死的 `getReputation()`。ERC-8004 的接入点就在这里——我们在 `gateway.ts` 里**特意留了一个可插拔口**，现在读 fixture，将来把函数体换成一次链上 Reputation Registry 查询即可，调用方一行都不用改：

```ts
// 现在（gateway.ts 实际代码）：读 fixture
function getReputation(_agentUrl: string): number {
  return 88; // fixture
}

// 未来（Capstone 里换实现，签名不变）：查链上 Reputation Registry
async function getReputation(agentId: bigint): Promise<number> {
  // 1) 从 Reputation Registry 读该 agentId 的反馈条目
  // 2) 链下按 reviewer / tag 过滤 + 聚合成一个分数
  // 3) 返回给目录撮合逻辑 —— 上层 DirectoryEntry.reputation 照常用
  return await queryOnchainReputation(agentId);
}
```

这正是本层大纲里那条“**身份信誉可插拔**”原则的落地：

> 别把项目核心逻辑焊死在 ERC-8004 当前的注册表接口上。

为什么强调“可插拔”而不是“焊死”？因为 ERC-8004 还是 **draft ERC**，接口细节仍在变（尤其 Validation Registry）。我们把它做成一个**可替换的信誉来源**：目录撮合只依赖“给我一个 0–100 的分数”这个契约，至于分数来自 fixture、来自 ERC-8004、还是来自别的信誉系统，都能换，核心架构不动。

对照一下 `DirectoryEntry` 的 `reputation` 字段——注释早就写明了它的归宿：

```ts
interface DirectoryEntry {
  name: string;
  baseUrl: string;
  capability: string;
  chain: string;
  /** 可插拔：真实系统里应来自 ERC-8004 Reputation 注册表，而非本地写死。 */
  reputation: number;
}
```

## 9.6 边界与版本核验（重要）

这一模块只讲到“概念 + 接入点”，下面这些边界必须诚实标注：

- **本章不做深度合约实现。** agent = NFT、三注册表、tokenURI→Agent Card 都是概念讲解 + 一个只读 `tokenURI` 示意；真正的合约部署、注册、写信誉、跨链迁移，全部留到 **Capstone**。上面那段 viem 代码不接进 `agent-network` 脚手架跑（脚手架主线是 MCP + A2A 三进程 + gateway，已实测跑通）。
- **ERC-8004 状态（截至 2026-07）**：它是 **draft ERC（EIP-8004 “Trustless Agents”）**，由 MetaMask、以太坊基金会、Google、Coinbase 等贡献者共同起草；已在 2026-01-29 前后有主网参考实现部署，多链（Base / Gnosis / BNB 等）开始起量、已注册的 agent 身份在增长。但它**仍是 draft，接口细节可能变**——尤其 **Validation Registry 最不定型**，还在和 TEE 社区讨论、后续 spec 会修订。
- **依赖顺序建议**：优先只依赖 Identity（标准 ERC-721 的 `tokenURI`）+ Reputation 的**读**接口。写接口、Validation 的形状变动概率更高。
- 动手前以官方为准：[EIP-8004 规范](https://eips.ethereum.org/EIPS/eip-8004)、[参考合约仓库](https://github.com/erc-8004/erc-8004-contracts)。

## 本模块小结

- 目录里的 `reputation` 字段今天是 fixture；ERC-8004 要让它变成**链上、可携带、跨平台带得走**的真信誉。
- ERC-8004 = 三个链上单例注册表：**Identity**（agent 是谁）、**Reputation**（凭什么信）、**Validation**（真做了吗）；Identity 是锚，另两个引用它。
- 核心机制：**每个 agent = 一个 ERC-721 NFT**，`tokenURI` 指向 Agent Card JSON——天然兼容 NFT 生态 + token-bound account；信誉因此**可携带**（换平台不清零、能从以太坊迁到 Base/Optimism）。
- 那个 `tokenURI` 指向的 JSON，就是我们模块 4 的 **A2A Agent Card**：链上给身份，链下 A2A 给发现和派任务，接缝在同一张卡。
- 接入本层目录：`gateway.ts` 的 `getReputation()` 是**可插拔口**，现在读 fixture，未来把函数体换成链上 Reputation Registry 查询，调用方不变。
- 边界：本章只讲概念 + 接入点，合约实现与链上落地留到 **Capstone**；ERC-8004 是 draft ERC，已有多链主网参考实现但接口仍可能变（Validation 最不定型）。
