# Layer 6 总结与复习 —— Agent 通信协议

走到这里，Layer 6 的两条主线（MCP 深化、A2A 从发现到交接）和一条辅线（ERC-8004 可插拔身份/信誉）已经全部铺完。这一章把它们收成一张网、一套自测题、一份可跑清单，最后指向下一步。

## 一句话贯穿全层

```text
MCP 接工具、A2A 接 agent；
发现一张卡先验签 + 协商版本，再交给目录做撮合；
ERC-8004 让身份和信誉可携带，作为可插拔层挂在撮合背后。
```

把这句话拆开，就是本层做的全部事情：**工具接入走 MCP，agent 协作走 A2A，信任的入口是「验签 + 版本协商」，选谁干活是「目录撮合」，而「这个 agent 是谁、口碑几何」由 ERC-8004 在链上兜底、但不焊死进核心逻辑。** 少任何一环，agent 网络要么冒充横行（不验签），要么鸡同鸭讲（不协商版本），要么找不到对的人（没目录）。

## 分模块回顾

| 模块 | 一句话记住它 |
| --- | --- |
| 1 为什么需要通信协议 | agent 走出孤岛要开放标准：**MCP 让 agent 用工具，A2A 让 agent 找 agent**，两者分工不重叠。 |
| 2 MCP 深化：三原语 | 用 `@modelcontextprotocol/sdk` 的 `registerTool` 把链上只读能力封成 **tool**；tools/resources/prompts 三原语各司其职。 |
| 3 MCP 传输层 | 远程多客户端用 **StreamableHTTP**（`sessionIdGenerator: undefined` 起无状态端点）；**SSE 传输已废弃**，别再用。 |
| 4 A2A 导论 + Agent Card | Agent Card 是 agent 的「名片 + 说明书」：`protocolVersion` / `url` / `skills` / `capabilities`，默认挂在 `/.well-known/agent-card.json`。 |
| 5 A2A 签名验证（核心） | **Signed Agent Card = 先 JCS 规范化、再 JWS 签名**；验签时用当前卡重算 JCS，篡改任何字段都会失败——这是本层的安全灵魂。 |
| 6 A2A 任务与消息 | 用 `ClientFactory.createFromUrl` + `client.sendMessage` 发一次任务；11 个 JSON-RPC 方法、三种 binding、v1.0 破坏性变更（枚举改大写等）要心里有数。 |
| 7 协议适配层与路由 | 一个统一 adapter 按「目标 + 支持协议」把请求路由到 MCP / A2A / HTTP，带 health check，降级要显式可审计。 |
| 8 服务目录与互操作测试 | 内存 registry 按能力/链/预算/信誉/健康撮合；喂一批坏卡，**拒绝矩阵全部 ⛔**。 |
| 9 ERC-8004（轻量） | 三注册表（Identity / Reputation / Validation），agent≈NFT→指向 Agent Card；信誉可携带，作为**可插拔**接口接入，不押核心架构。 |
| 10 里程碑网关 | Agent Protocol Gateway 把发现 + 验签 + 版本协商 + 目录撮合 + 错误回放串成一条可审计的决策链。 |

## 概念对照表

一页速查，考前扫一眼就能对上号：

| 概念 | 一句话含义 |
| --- | --- |
| **MCP** | Model Context Protocol，agent↔**工具**的协议：把链上/外部能力封成 tool 给 agent 调。 |
| **A2A** | Agent-to-Agent，agent↔**agent**的协议：发现、验证、派任务、交接结果。 |
| **Agent Card** | agent 的机器可读名片：声明 `protocolVersion` / `url` / `skills` / `capabilities`，发现的第一手信息。 |
| **Signed Agent Card** | 带 JWS 签名的 Agent Card，证明「这张卡确实由持钥者签发、未被篡改」。 |
| **JWS** | JSON Web Signature（RFC 7515），本层用 EdDSA(Ed25519) 对卡做分离签名。 |
| **JCS** | JSON Canonicalization Scheme（RFC 8785），签名前把 JSON 规范化（key 升序、去空白），**签名双方字节必须一致**。 |
| **binding** | A2A 的传输绑定：同一套语义可跑在 **JSON-RPC 2.0 / gRPC / HTTP+REST** 三种上。 |
| **版本协商** | 用 `A2A-Version` header 对齐协议版本；不兼容则走新 URI 或降级，绝不静默乱跑。 |
| **adapter（协议适配层）** | 统一入口，按目标 + 协议把请求路由到 MCP/A2A/HTTP，并挂认证、额度、计费、日志。 |
| **registry（服务目录）** | 可查询的 agent 目录，按能力/链/收费/信誉/健康做撮合，选出该派任务的 agent。 |
| **ERC-8004** | 链上「可信 agent」标准草案：Identity/Reputation/Validation 三注册表，让**身份与信誉可携带、可插拔**。 |

## 综合自测

覆盖 10 个模块，能不看书答完，本层就过关了：

1. 一句话说清 **MCP 和 A2A 的分工**。为什么不能拿 A2A 当工具协议、也不能拿 MCP 当 agent 发现协议？
2. MCP 三原语 **tools / resources / prompts** 各是什么？本层把「查 Base Sepolia 余额」封成了哪一种？
3. MCP 远程传输为什么选 **StreamableHTTP** 而不是 SSE？`StreamableHTTPServerTransport` 的 `sessionIdGenerator: undefined` 意味着什么？
4. 为什么脚手架里连本机服务用 **`127.0.0.1` 而不是 `localhost`**？在 WSL2 下不这么做会踩什么坑？
5. 一张 Agent Card 至少要有哪些**必填字段**？默认发现路径是什么？
6. **JCS 为什么必须在 JWS 之前**做？如果签名方和验证方对同一张卡算出的规范化字节差一个空格，会发生什么？
7. 验签时为什么要**用「当前收到的卡」去掉 signature 后重新算 JCS**，而不是信任卡里自带的 payload？这条设计怎么挡住「篡改 url」攻击？
8. 列出**坏卡拒绝矩阵**至少 4 类：分别是哪几种卡、各因为什么被 ⛔ 拒绝？
9. 派任务前的**信任闸完整顺序**是什么（结构校验 → … → 版本协商）？漏掉「验签」这一步，最坏会发生什么？
10. 用 `@a2a-js/sdk` 发一次任务的**最小调用链**是什么（从 `ClientFactory` 到拿到回复）？消息体里 `kind` / `role` / `parts` 各填什么？
11. A2A 有 **11 个 JSON-RPC 方法**和**三种 binding**，请各举两个例子。`message/send` 和 `message/stream` 差在哪？
12. A2A **协议已 v1.0、SDK 还在 v0.3.x**，这个落差本层怎么处理？卡里的 `protocolVersion` 该按哪个版本写？（讲概念/写代码分别按哪个）
13. 协议**降级（A2A → HTTP fallback）**为什么必须「显式、可见、可审计」？adapter 层有哪些东西**绝不能被绕过**？
14. 服务目录**撮合**用到哪些维度？其中「信誉」这一维在教学里来自哪里、生产里应该来自哪里？
15. ERC-8004 的**三注册表**分别管什么？为什么本层强调把它做成**可插拔**、而不是把核心逻辑焊死在它当前接口上？
16. 里程碑网关跑「错误回放」时，把一张卡的 `url` 改成 `http://attacker.example/a2a`，验签结果应该是什么？网关据此做什么决策？

## 可运行产物清单

两套脚手架都已【实测跑通】。以下命令与预期以真实运行为准。

### protocol-lab（本地、不联网、无 API key）

在 `layer6/protocol-lab/` 里 `pnpm install` 后：

| 命令 | 干什么 | 预期 |
| --- | --- | --- |
| `pnpm demo:card` | 构造 + 解析 Agent Card | 全字段卡 ✅ 通过；缺字段/空 skills/坏 url 的卡 ⛔ 拒绝并打原因 |
| `pnpm demo:jws` | JCS 规范化 + JWS 签名/验签 | 原卡验签 ✅；改任一字段后重验 ⛔（"卡被篡改或非该公钥所签"） |
| `pnpm demo:version` | `A2A-Version` 协商 + v1.0 枚举迁移 | 支持的版本 ✅；不支持的版本 ⛔；旧枚举→v1.0 大写枚举映射打印 |
| `pnpm demo:router` | 协议路由闸门 | 按目标 + 支持协议路由到 MCP / A2A / HTTP；不支持组合 ⛔ |
| `pnpm demo:directory` | 内存 registry 撮合 | 按能力/链/预算/信誉/健康筛出候选，落选项打拒因 |
| `pnpm demo:interop` | 互操作拒绝矩阵 | 8 条自断言全过：坏卡全 ⛔、好卡 ✅，任一断言不符即非零退出 |
| `pnpm demo:all` | 依次跑上面全部 | 六个 demo 顺序输出，用来一键回归 |

### agent-network（真实三进程、真连 Base Sepolia 只读 RPC）

在 `layer6/agent-network/` 里 `pnpm install` 后，开**三个终端**依次起：

| 终端 | 命令 | 预期 |
| --- | --- | --- |
| ① MCP server | `pnpm start:mcp` | 监听 `:41241`，注册 `getBalance` / `getBlockNumber` / `getTxCount`（viem 连 `https://sepolia.base.org`） |
| ② A2A agent | `pnpm start:agent` | 监听 `:41242`，发布 Signed Agent Card 到 `/.well-known/agent-card.json`；executor 内部用 MCP Client 连①调工具 |
| ③ client | `pnpm start:client` | 发现 agent → **验签 ✅ 通过** → 派任务 → agent 内部调 MCP 查到**真实链上余额** → 回真数据 |

里程碑网关（起好①②后单开一个终端）：

| 命令 | 预期 |
| --- | --- |
| `pnpm start:gateway` | ① 正常交接：发现 → 验签 ✅ → 目录撮合命中（打印能力/链/信誉）→ 派任务拿真数据；② 错误回放：把卡 `url` 篡改后验签 **⛔ 拒绝**，决策为「不把任务派给可能伪造端点的 agent」 |

> 关掉签名开关跑负例：`SIGN_AGENT_CARD=false pnpm start:agent`，此时 client 会因「未签名卡」**⛔ 拒绝派任务**——亲手验证「不验签 = 谁都能冒充」这条安全灵魂。

## 下一步怎么接

Layer 6 交付后，多个 agent 已经能「互相发现、验证、协作、交接任务」。但它们之间**还不能自主付款**，链上信誉也还停在可插拔接口上。这两件事留给后面的层收口：

```text
Layer 6（已完成）：A2A 发现 + MCP 工具 + ERC-8004 身份（agent 能互相找到、信任、协作）
下一步 · 支付层     ：x402 / AP2 让 agent 之间能自主结算；高金额动作引入 mandate 人工审批
下一步 · Capstone   ：把 ERC-8004 三注册表真正落到链上，身份/信誉从 fixture 变成可携带的链上事实
```

到 Capstone，你会把 Layer 4 的受限钱包、Layer 5 的 MCP 工具、Layer 6 的 A2A 协议、以及支付层的自主结算，串成一个**能自己找活、能验证对方、能安全付款、留得下审计**的 agent 网络——那才是「agent-to-agent」这个名字的完整落地。

## 本模块小结

- 一句话贯穿：**MCP 接工具、A2A 接 agent；发现先验签 + 协商版本，再让目录撮合；ERC-8004 让身份信誉可携带且可插拔。**
- 安全灵魂只有一条：**坏卡一律拒绝**——未签名 / 验签失败 / 版本不兼容 / 篡改 endpoint，全部 ⛔。验签靠「用当前卡重算 JCS」挡篡改。
- 概念/实现有落差要诚实标注：**A2A 概念按 v1.0，代码按 SDK v0.3.x**；MCP 传输只用 StreamableHTTP（SSE 已废弃）。
- 两套脚手架都可跑：`protocol-lab` 拆原理（六个 demo），`agent-network` 跑真实三进程（连 Base Sepolia 只读）。
- 下一步是支付层（x402 / AP2）和 Capstone（ERC-8004 落链），把协作网络补上「自主付款」和「链上信誉」。
