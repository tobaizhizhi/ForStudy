# Layer 1 总结与复习

## 模块 1：Layer 1 心智模型与工具准备

这一模块不是让你立刻写合约，而是先搞清楚：你用什么工具连接链、查链、验证交易结果。后面所有读链、发交易、查 receipt、看 logs，都会建立在这里。

## 1. 这一模块要记住什么

最核心的关系：

```text
人 / 脚本
  -> 钱包 / cast / viem
  -> RPC 节点
  -> 区块链网络
```

区块浏览器是旁边的“链上数据查看器”：你用钱包、`cast` 或 `viem` 发起操作后，可以去区块浏览器核对链上到底发生了什么。

## 2. 五个角色分别是什么

| 角色 | 负责什么 | 你现在怎么用 |
| --- | --- | --- |
| 钱包 | 管理私钥、签名、发交易 | 用测试钱包领取测试币、手动签名、手动发交易 |
| RPC | 让工具连接到某条链 | `https://sepolia.base.org` 连接 Base Sepolia |
| 区块浏览器 | 用网页查看链上数据 | 查地址余额、交易状态、区块、receipt、logs |
| `cast` | Foundry 的命令行链工具 | 查 chainId、余额、区块、交易，也能发简单交易 |
| `viem` | TypeScript 链交互库 | 后面用代码读链、写链、解析 ABI / logs |

一句话记法：

```text
钱包管钥匙，RPC 是通道，区块浏览器负责看结果，cast 是命令行工具，viem 是代码里的链工具。
```

## 3. Base Sepolia 基本信息

```text
Network Name: Base Sepolia
RPC URL: https://sepolia.base.org
Chain ID: 84532
Currency Symbol: ETH
Block Explorer: https://sepolia.basescan.org
```

注意：Base Sepolia 的 gas 代币是测试网 ETH，不是 `BASE` 代币。

## 4. `.env.example` 和 `.env` 的区别

`.env.example` 是模板，告诉你项目需要哪些配置。它通常可以提交到 Git。

`.env` 是真实配置，里面会放 RPC、测试私钥、测试地址。它不能提交到 Git。

模块 1 需要这几个配置：

```env
BASE_SEPOLIA_RPC_URL=https://sepolia.base.org
SEPOLIA_RPC_URL=https://ethereum-sepolia.publicnode.com
TEST_PRIVATE_KEY=你的测试钱包私钥
TEST_ADDRESS=你的测试钱包地址
```

如果你只练 Base Sepolia，`SEPOLIA_RPC_URL` 可以暂时不填。

## 5. 加载 `.env`

如果你把配置写在 `.env`，终端不会自动读取。需要加载：

```bash
set -a
source .env
set +a
```

意思是：

```text
set -a      开启自动 export
source .env 读取 .env 里的变量
set +a      关闭自动 export
```

这样后面的 `cast` 命令才能读到 `$BASE_SEPOLIA_RPC_URL`、`$TEST_ADDRESS` 等变量。

## 7. 常见报错

报错：

```text
error: a value is required for '--rpc-url <URL>' but none was supplied
```

意思是：你传给 `--rpc-url` 的变量是空的。

常见原因：

```bash
cast chain-id --rpc-url $BASE_SEPOLIA_RPC_URL
```

但 `$BASE_SEPOLIA_RPC_URL` 没有值。

排查：

```bash
echo "$BASE_SEPOLIA_RPC_URL"
```

如果没有输出，说明你还没加载 `.env`，或者 `.env` 里没填。

临时测试可以直接写：

```bash
cast chain-id --rpc-url https://sepolia.base.org
```

## 模块 2：账户、地址、私钥与签名

这一模块是给后面的 EIP-712、EIP-1271、ERC-4337 打底。你要先分清楚：谁在控制账户、什么是地址、签名到底证明了什么。

## 1. 这一模块要记住什么

最短关系链：

```text
private key
  -> public key
  -> address
```

这里最容易记混的点是：

- `public key` 和 `address` 不是一个东西。
- `address` 是从公钥加工出来的短标识。
- `signature` 是私钥对一段内容的授权证明。

一句话总结：

```text
私钥负责签名，公钥负责验证，地址负责表示账户。
```

## 2. EOA 和合约账户

| 类型 | 简单理解 | 你要记住什么 |
| --- | --- | --- |
| EOA | 外部账户，普通钱包 | 有私钥，能主动发交易 |
| Contract Account | 合约账户 | 由合约代码控制，通常没有普通私钥 |

怎么看一个地址大概率是不是合约账户：

```bash
cast code $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL
```

- 如果输出是 `0x`，通常说明这是 EOA。
- 如果输出是一长串 bytecode，说明这是合约账户。

## 3. 三种常见签名

| 签名类型 | 用来做什么 | 会不会花 gas |
| --- | --- | --- |
| 交易签名 | 转账、调用合约、改变链上状态 | 会 |
| `personal_sign` | 普通消息签名，例如登录证明 | 不会 |
| EIP-712 | 结构化数据签名，例如任务意图、授权单 | 不会，签完之后如果拿去发交易才会进入链上流程 |

你现在先记住：

```text
签消息 != 发交易
```

签消息只是证明“这个地址同意了这段内容”；发交易才会真的改链上状态。

## 4. 模块 2 必会命令

生成一个新测试账户：

```bash
cast wallet new
```

从私钥推导地址：

```bash
cast wallet address --private-key $TEST_PRIVATE_KEY
```

从私钥推导公钥：

```bash
cast wallet public-key --raw-private-key $TEST_PRIVATE_KEY
```

给一段消息签名：

```bash
MESSAGE="hello layer1"
SIGNATURE=$(cast wallet sign "$MESSAGE" --private-key $TEST_PRIVATE_KEY)
```

验证签名：

```bash
cast wallet verify --address $TEST_ADDRESS "$MESSAGE" "$SIGNATURE"
```

查地址余额，确认这个账户能连到 Base Sepolia：

```bash
cast balance $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL
```

## 5. 最容易混淆的地方

- 公钥不是地址，地址只是从公钥派生出来的短标识。
- 签名不是消息本身，签名是对消息的数学证明。
- `cast wallet sign` 不会发交易，也不会花 gas。
- `cast wallet verify` 只是在检查签名是否匹配地址。
- 能验证签名，不代表签名人真的看懂了内容。

最后一点很重要：签名验证通过，只能说明“是这个地址背后的私钥签的”，不能说明“签的人理解了后果”。

## 6. 模块 2 的安全底线

- 不要把测试私钥提交到 Git。
- 不要在不理解内容时乱签名。
- 不要把“签消息”和“发交易”混为一谈。
- 不要把地址和公钥当成一个东西。

## 模块 3：交易生命周期与 gas

这一模块要解决的是：一笔交易从你本地发出去，到链上执行完成，中间到底经历了什么。后面学合约调用、EIP-712 授权、x402 支付、ERC-4337 UserOperation，都会反复用到这里的概念。

## 1. 这一模块要记住什么

一笔普通交易的大致流程：

```text
本地构造交易
  -> 私钥签名
  -> 通过 RPC 广播
  -> 进入 mempool / 等待区
  -> 被区块生产者或 L2 sequencer 打包
  -> EVM 执行
  -> 生成 receipt
  -> 前端 / 脚本 / 浏览器查询结果
```

最短记法：

```text
tx = 我请求链做什么
receipt = 链执行后的结果
```

## 2. 交易里最重要的字段

| 字段 | 含义 | 你要怎么理解 |
| --- | --- | --- |
| `from` | 发送者地址 | 谁发起交易 |
| `to` | 接收方地址 | 发给谁；如果是合约调用，这里是合约地址 |
| `nonce` | 账户交易序号 | 同一个地址的交易顺序号，不能跳乱 |
| `value` | 原生币数量 | 转多少 ETH；调用合约时也可以带 value |
| `data` | 调用数据 | 普通 ETH 转账通常为空；合约调用会有 calldata |
| `chainId` | 链 ID | 防止交易被拿去别的链上重放 |
| `gas limit` | 最大 gas 上限 | 你最多允许这笔交易消耗多少 gas |
| `gasUsed` | 实际消耗 gas | receipt 里看到的实际消耗量 |
| `effectiveGasPrice` | 实际 gas 单价 | 交易打包后最终按多少 wei / gas 结算 |

手续费计算：

```text
实际手续费 = gasUsed × effectiveGasPrice
```

## 3. mempool 是什么

`mempool` 可以理解成：

```text
已广播、未上链、等待被打包的交易池。
```

交易进入 mempool 不等于成功。它可能：

- 很快被打包。
- 因为 gas 太低而卡住。
- 因为 nonce 问题被后续交易挡住。
- 在某些情况下被节点丢弃。

在 Base 这类 L2 上，具体机制会有 sequencer，但你现在可以先用“等待打包区”来理解。

## 4. nonce 是什么

`nonce` 是一个账户发交易的顺序号。

比如某个地址当前 nonce 是 `7`，下一笔交易就应该用：

```text
nonce = 7
```

这笔交易成功后，下一个 nonce 变成：

```text
nonce = 8
```

为什么重要：

- 同一个账户的交易按 nonce 顺序执行。
- `nonce = 8` 的交易不能越过卡住的 `nonce = 7`。
- 所以一笔低 gas 或异常交易可能把后续交易一起卡住。

查 nonce：

```bash
cast nonce $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL
```

## 5. gas、gasUsed、effectiveGasPrice

`gas` 是 EVM 执行操作消耗的计算单位。

你可以这样区分：

| 名称 | 什么时候知道 | 含义 |
| --- | --- | --- |
| `gas limit` | 发交易前设置或估算 | 最多愿意给这笔交易多少 gas |
| `gasUsed` | 交易执行后，在 receipt 里看到 | 实际用了多少 gas |
| `gasPrice` / `maxFeePerGas` | 发交易前看到或设置 | 预估或上限单价 |
| `effectiveGasPrice` | 交易执行后，在 receipt 里看到 | 最终实际 gas 单价 |

重点：

```text
最终成本看 receipt 里的 gasUsed 和 effectiveGasPrice。
```

## 6. 模块 3 必会命令

查当前 nonce：

```bash
cast nonce $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL
```

查当前 gas price：

```bash
cast gas-price --rpc-url $BASE_SEPOLIA_RPC_URL
```

查最新区块：

```bash
cast block latest --rpc-url $BASE_SEPOLIA_RPC_URL
```

查余额：

```bash
cast balance $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL --ether
```

估算一笔极小额转账需要多少 gas：

```bash
TO_ADDRESS=$TEST_ADDRESS
cast estimate "$TO_ADDRESS" --from $TEST_ADDRESS --value 0.000001ether --rpc-url $BASE_SEPOLIA_RPC_URL
```

估算总成本：

```bash
cast estimate "$TO_ADDRESS" --from $TEST_ADDRESS --value 0.000001ether --cost --rpc-url $BASE_SEPOLIA_RPC_URL
```

发送一笔测试交易，先只拿交易 hash：

```bash
TX_HASH=$(cast send "$TO_ADDRESS" \
  --value 0.000001ether \
  --private-key $TEST_PRIVATE_KEY \
  --rpc-url $BASE_SEPOLIA_RPC_URL \
  --async)

echo "$TX_HASH"
```

查看交易请求：

```bash
cast tx "$TX_HASH" --rpc-url $BASE_SEPOLIA_RPC_URL
```

查看执行结果：

```bash
cast receipt "$TX_HASH" --rpc-url $BASE_SEPOLIA_RPC_URL
```

用浏览器交叉核对：

```text
https://sepolia.basescan.org/tx/<TX_HASH>
```

## 7. 最容易混淆的地方

- 交易 hash 出来了，不代表交易已经成功。
- 交易进了 mempool，不代表一定会上链。
- `cast tx` 看的是交易本身，`cast receipt` 看的是执行结果。
- 失败交易也会消耗 gas。
- `gas limit` 是上限，`gasUsed` 才是实际用量。
- `effectiveGasPrice` 是实际 gas 单价，不是你发交易前随便看到的预估值。
- nonce 卡住时，后面的交易也可能被卡住。

## 8. 模块 3 的安全底线

- 发交易前先确认 `chainId` 和 RPC。
- 发交易前先查余额，避免余额不足。
- 先用 `cast estimate` 估算，再 `cast send`。
- 初学时先转给自己的测试地址，金额用极小值。
- 不要在主网用练习命令。
- 私钥仍然不能出现在日志、截图、Git 提交里。

## 模块 4：读链、RPC 与区块浏览器

模块 4 和模块 3 会有一些重叠，但重点不同：

```text
模块 3：交易怎么发生
模块 4：交易发生后，怎么从链上读取事实并交叉验证
```

所以模块 4 的重点不是再次讲 gas 和 nonce，而是学会稳定读取这些链上对象：

```text
block
account state
transaction
receipt
logs
```

## 1. 这一模块要记住什么

读链不是“相信某个页面”，而是用多个工具确认同一个事实。

最核心的读链路径：

```text
cast / viem / 区块浏览器
  -> RPC
  -> 区块链节点
  -> 返回 block / state / tx / receipt / logs
```

一句话：

```text
模块 4 学的是怎么查链上事实。
```

## 2. 几个链上对象

| 对象 | 是什么 | 常用命令 |
| --- | --- | --- |
| block | 一批已经打包的交易 | `cast block latest` |
| account state | 某个地址的余额、nonce、code | `cast balance`、`cast nonce`、`cast code` |
| transaction | 交易请求本身 | `cast tx <TX_HASH>` |
| receipt | 交易执行结果 | `cast receipt <TX_HASH>` |
| logs | 合约执行时发出的事件 | receipt 里的 `logs`、`cast logs` |

模块 3 里已经讲过 `transaction` 和 `receipt`，这里再用一句话复习：

```text
transaction = 请求
receipt = 结果
```

## 3. RPC 是什么

RPC 是你连接区块链节点的接口。

钱包、`cast`、`viem`、前端、后端脚本，本质上都要通过 RPC 向节点问问题或广播交易。

例如：

```bash
cast chain-id --rpc-url $BASE_SEPOLIA_RPC_URL
```

这不是在本地算，而是在问 RPC 后面的节点：

```text
你是哪条链？
```

## 4. block tag

常见 block tag：

| tag | 简单理解 | 你现在怎么用 |
| --- | --- | --- |
| `latest` | 当前节点认为的最新区块 | 最常用 |
| `pending` | 包含待处理交易的状态 | 初学少用 |
| `safe` | 相对安全的区块 | 了解即可 |
| `finalized` | 最终确认的区块 | 后面学 finality 再深入 |

现在优先掌握：

```text
latest = 查当前最新状态
```

## 5. 模块 4 必会命令

确认 RPC 正常、链也对：

```bash
cast chain-id --rpc-url $BASE_SEPOLIA_RPC_URL
cast client --rpc-url $BASE_SEPOLIA_RPC_URL
cast block-number --rpc-url $BASE_SEPOLIA_RPC_URL
```

查最新区块：

```bash
cast block latest --rpc-url $BASE_SEPOLIA_RPC_URL
```

查地址状态：

```bash
cast balance $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL --ether
cast nonce $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL
cast code $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL
```

查交易请求和执行结果：

```bash
cast tx "$TX_HASH" --rpc-url $BASE_SEPOLIA_RPC_URL
cast receipt "$TX_HASH" --rpc-url $BASE_SEPOLIA_RPC_URL
```

看一眼原始 JSON-RPC：

```bash
cast rpc eth_blockNumber --rpc-url $BASE_SEPOLIA_RPC_URL
cast rpc eth_getBalance "[\"$TEST_ADDRESS\",\"latest\"]" --raw --rpc-url $BASE_SEPOLIA_RPC_URL
cast rpc eth_getTransactionByHash "[\"$TX_HASH\"]" --raw --rpc-url $BASE_SEPOLIA_RPC_URL
```

## 模块 5：ABI、calldata 与合约调用

模块 5 解决的是：你调用合约时，真正发给合约的 `data` 到底是什么。

前面模块 3 / 4 讲的是交易和读链；模块 5 开始进入“合约函数调用”本身。

## 1. 这一模块要记住什么

最核心关系：

```text
function signature
  -> selector
参数
  -> ABI encode
selector + ABI 编码后的参数
  -> calldata
```

一句话：

```text
calldata = 合约调用的机器格式参数包。
```

比如 ERC-20 转账：

```text
transfer(address,uint256)
  -> selector: 0xa9059cbb
  -> calldata: 0xa9059cbb + to 地址 + amount
```

## 2. 几个核心概念

| 概念 | 是什么 | 你要怎么理解 |
| --- | --- | --- |
| ABI | 合约接口说明 | 告诉工具函数名、参数、返回值、事件 |
| function signature | 函数字符串 | 例如 `transfer(address,uint256)` |
| selector | 函数选择器 | 对 signature 做 keccak 后取前 4 bytes |
| calldata | 合约调用原始数据 | `selector + ABI 编码后的参数` |
| read call | 只读调用 | 不改状态、不花 gas、没有 tx hash |
| write tx | 写交易 | 改状态、要签名、要上链、会花 gas |

注意：这里的 function signature 不是私钥签名，不是模块 2 里的 signature。它只是函数的标准文本表示。

## 3. `USDC_BASE_SEPOLIA` 是什么

这是一个 shell 变量，用来保存 Base Sepolia 测试 USDC 合约地址：

```bash
USDC_BASE_SEPOLIA=0x036CbD53842c5426634e7929541eC2318f3dCF7e
```

后面命令里的：

```bash
$USDC_BASE_SEPOLIA
```

会被替换成这个合约地址。

这样做是为了少复制长地址，降低输错概率。

## 4. 模块 5 必会命令

查看函数 selector：

```bash
cast sig "transfer(address,uint256)"
```

输出：

```text
0xa9059cbb
```

反查 selector：

```bash
cast 4byte 0xa9059cbb
```

编码 `transfer` calldata：

```bash
TO_ADDRESS=0x1111111111111111111111111111111111111111
TRANSFER_CALLDATA=$(cast calldata "transfer(address,uint256)" "$TO_ADDRESS" 1000000)
echo "$TRANSFER_CALLDATA"
```

解码 calldata：

```bash
cast decode-calldata "transfer(address,uint256)" "$TRANSFER_CALLDATA"
```

读取 ERC-20 信息：

```bash
cast call "$USDC_BASE_SEPOLIA" "symbol()(string)" --rpc-url $BASE_SEPOLIA_RPC_URL
cast call "$USDC_BASE_SEPOLIA" "decimals()(uint8)" --rpc-url $BASE_SEPOLIA_RPC_URL
cast call "$USDC_BASE_SEPOLIA" "balanceOf(address)(uint256)" $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL
```

用原始 calldata 做只读调用：

```bash
BALANCE_CALLDATA=$(cast calldata "balanceOf(address)" "$TEST_ADDRESS")
cast call "$USDC_BASE_SEPOLIA" --data "$BALANCE_CALLDATA" --rpc-url $BASE_SEPOLIA_RPC_URL
```

这两种写法本质等价：

```bash
cast call "$USDC_BASE_SEPOLIA" "balanceOf(address)(uint256)" $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL
```

和：

```bash
BALANCE_CALLDATA=$(cast calldata "balanceOf(address)" "$TEST_ADDRESS")
cast call "$USDC_BASE_SEPOLIA" --data "$BALANCE_CALLDATA" --rpc-url $BASE_SEPOLIA_RPC_URL
```

区别只是：

```text
第一种：cast 帮你编码 calldata
第二种：你先用 cast calldata 编好，再用 --data 直接发出去
```

## 5. read call 和 write tx 的区别

| 类型 | 命令 | 改状态吗 | 花 gas 吗 | 有 tx hash 吗 |
| --- | --- | --- | --- | --- |
| read call | `cast call` | 不改 | 不花 | 没有 |
| write tx | `cast send` | 会改 | 会花 | 有 |

读调用例子：

```bash
cast call "$USDC_BASE_SEPOLIA" "balanceOf(address)(uint256)" $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL
```

写交易例子：

```bash
cast send "$USDC_BASE_SEPOLIA" "transfer(address,uint256)" "$TO_ADDRESS" 1000000 \
  --private-key $TEST_PRIVATE_KEY \
  --rpc-url $BASE_SEPOLIA_RPC_URL
```

注意：写交易这条不要随便跑，除非你确认测试账户有 Base Sepolia USDC。

## 6. 最容易混淆的地方

- function signature 不是私钥签名。
- selector 是函数短编号，不是地址。
- calldata 不是随便的乱码，它是 `selector + 编码参数`。
- `--data` 是直接把原始 calldata 发给合约。
- `cast call` 是模拟读，不上链、不花 gas。
- `cast send` 是写交易，要签名、上链、花 gas。
- 普通 ETH 转账通常 data 为空；合约调用通常 data 里就是 calldata。
- 项目里不要手写 ABI 编码，优先用 `cast` / `viem` / `ethers`。

## 7. BaseScan 里怎么看

找一笔 ERC-20 转账交易，重点看：

- `Method`：通常会显示 `transfer`。
- `Input Data`：通常以 `0xa9059cbb` 开头。
- `Logs`：通常有 `Transfer` event。

你要能把这三者对应起来：

```text
Method: transfer
Input Data: 0xa9059cbb...
Logs: Transfer event
```

## 模块 6：EVM 执行模型入门

模块 6 讲的是：EVM 收到 calldata 之后，大概怎么执行。

模块 5 偏“输入是什么”，模块 6 偏“执行过程是什么”。

## 1. 这一模块要记住什么

EVM 可以先理解成一个状态机：

```text
当前链上状态
  + 交易 / call 输入
  + 合约 bytecode
  -> EVM 执行
  -> return / revert
  -> 新状态 + logs + receipt
```

你不需要手写 EVM，也不用背完 opcode。你要做的是：看到执行 trace 时，能认出几种常见词。

最核心的三个数据区域：

| 区域 | 简单理解 | 生命周期 | 常见用途 |
| --- | --- | --- | --- |
| `storage` | 合约的链上硬盘 | 持久保存 | token 余额、owner、配置 |
| `memory` | 本次调用的临时内存 | 调用结束就没了 | 临时计算、拼返回值 |
| `calldata` | 外部传进来的只读参数 | 本次调用期间存在 | 函数 selector、函数参数 |

你现在先记：

```text
storage 持久且贵；memory 临时；calldata 是外部输入。
```

## 2. 常见 opcode 先认识这几个

| opcode | 大概做什么 |
| --- | --- |
| `PUSH` | 把值压到 stack |
| `CALLDATALOAD` | 从 calldata 读参数 |
| `MSTORE` / `MLOAD` | 写 / 读 memory |
| `SSTORE` / `SLOAD` | 写 / 读 storage |
| `JUMP` / `JUMPI` | 跳转、分支 |
| `CALL` / `DELEGATECALL` | 调用其他合约 / 代理逻辑 |
| `LOG` | 发事件 |
| `REVERT` | 回滚执行 |
| `RETURN` | 返回结果 |

先不用逐个背，只要看到这些词，知道它们是“在操作哪种数据”和“执行在做什么”。

## 3. `delegatecall` 是什么

在 trace 里你可能会看到：

```text
[9750] 0x036C...dCF7e::balanceOf(...)
  ├─ [2553] 0xd74c...C5B5::balanceOf(...) [delegatecall]
  └─ ← [Return] ...
```

这通常表示：

```text
代理合约把逻辑委托给实现合约执行。
```

你现在先这样理解：

- 外层地址是代理入口。
- 内层地址是真正执行逻辑的实现合约。
- `delegatecall` 是很多可升级合约常见的模式。

## 4. 模块 6 必会命令

看合约有没有 bytecode：

```bash
cast code $USDC_BASE_SEPOLIA --rpc-url $BASE_SEPOLIA_RPC_URL
cast code $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL
```

如果输出是 `0x`，通常说明这个地址没有合约代码；如果不是 `0x`，说明有合约 bytecode。

反汇编 bytecode：

```bash
RUNTIME_CODE=$(cast code $USDC_BASE_SEPOLIA --rpc-url $BASE_SEPOLIA_RPC_URL)
cast disassemble "$(echo "$RUNTIME_CODE" | cut -c1-82)"
```

看一次只读调用的 trace：

```bash
cast call "$USDC_BASE_SEPOLIA" \
  "balanceOf(address)(uint256)" \
  "$TEST_ADDRESS" \
  --rpc-url $BASE_SEPOLIA_RPC_URL \
  --trace
```

看原始 storage slot：

```bash
cast storage "$USDC_BASE_SEPOLIA" 0 --rpc-url $BASE_SEPOLIA_RPC_URL
```

模拟一次可能失败的只读调用：

```bash
cast call "$USDC_BASE_SEPOLIA" \
  "transfer(address,uint256)(bool)" \
  0x1111111111111111111111111111111111111111 \
  999999999999999999 \
  --from "$TEST_ADDRESS" \
  --rpc-url $BASE_SEPOLIA_RPC_URL
```

看一笔已经上链的交易 trace：

```bash
cast run "$TX_HASH" --rpc-url $BASE_SEPOLIA_RPC_URL
```

## 5. trace 和 receipt 的区别

| 你想知道什么 | 先看什么 |
| --- | --- |
| 成功还是失败 | receipt |
| 花了多少 gas | receipt |
| 发出了哪些 logs | receipt / 区块浏览器 |
| 具体调用路径 | trace |
| 为什么复杂调用失败 | trace / Tenderly |
| 某个 opcode 是什么意思 | evm.codes |

一句话：

```text
receipt 看结果，trace 看过程。
```

## 6. 最容易混淆的地方

- `storage`、`memory`、`calldata` 不是一回事。
- `revert` 代表回滚这次执行，不留下状态变更。
- `delegatecall` 通常说明你在看代理合约模式。
- `cast call --trace` 是看执行路径，不是真正发交易。
- `cast run` 是看一笔已经上链交易的执行轨迹。
- 不是每笔交易都要看 trace；能用 receipt 解决就先看 receipt。

## 7. 安全底线

- 不要把 storage slot 乱解释成业务字段，尤其是 proxy 合约。
- 不要因为看到 trace 有 warning 就以为执行失败；先看实际返回和 gas。
- 不要在主网随便拿真实交易做实验。

## 模块 7：事件 logs 与最小索引意识

模块 7 讲的是：合约执行时发出的 event，最终会变成 receipt 里的 logs。前端、后端、agent、indexer 都可以用 logs 追踪链上发生过什么。

## 1. 这一模块要记住什么

核心流程：

```text
交易执行
  -> 合约 emit event
  -> 事件进入 receipt.logs
  -> cast logs / viem getLogs / indexer 查询这些 logs
```

一句话：

```text
logs 是链上应用对外发出的可索引记录。
```

它适合做：

- 前端历史记录。
- 任务状态追踪。
- 支付记录。
- 审计和回放。
- 后端 indexer 数据源。

但它不是：

```text
合约内部状态 storage
```

## 2. event / topics / data

以 ERC-20 `Transfer` 为例：

```solidity
event Transfer(address indexed from, address indexed to, uint256 value);
```

它的 log 通常是：

```text
topic0 = keccak256("Transfer(address,address,uint256)")
topic1 = from
topic2 = to
data   = value
```

| 字段 | 放什么 | 为什么 |
| --- | --- | --- |
| `topic0` | 事件签名哈希 | 用来识别事件类型 |
| `topic1` | indexed `from` | 方便按转出地址过滤 |
| `topic2` | indexed `to` | 方便按转入地址过滤 |
| `data` | 非 indexed `value` | 存放不能直接作为 topic 过滤的参数 |

一句话：

```text
topics 放事件类型和 indexed 参数，data 放非 indexed 参数。
```

## 3. topic0 是怎么知道事件类型的

链上 log 里只存 hash，不直接存 `Transfer` 这个名字。

识别过程是：

```text
链上 topic0
  -> 用 ABI / 标准 / 签名数据库找候选事件
  -> 对候选事件签名做 keccak256
  -> 哪个结果等于 topic0，就识别成哪个事件
```

ERC-20 `Transfer` 是标准事件，所以大家都知道：

```text
keccak256("Transfer(address,address,uint256)")
= 0xddf252ad1be2c89b69c2b068fc378daa952ba7f163c4a11628f55a4df523b3ef
```

## 4. 模块 7 必会命令

计算 `Transfer` 的 topic0：

```bash
cast sig-event "Transfer(address,address,uint256)"
```

演示 indexed 参数如何进入 topics：

```bash
cast abi-encode-event \
  "Transfer(address indexed,address indexed,uint256)" \
  0x0000000000000000000000000000000000000000 \
  0x1111111111111111111111111111111111111111 \
  1000000
```

查询最近 10 个区块里的 USDC Transfer logs：

```bash
LATEST_BLOCK=$(cast block-number --rpc-url $BASE_SEPOLIA_RPC_URL)
FROM_BLOCK=$((LATEST_BLOCK - 9))

cast logs \
  --address "$USDC_BASE_SEPOLIA" \
  --from-block "$FROM_BLOCK" \
  --to-block latest \
  "Transfer(address,address,uint256)" \
  --rpc-url $BASE_SEPOLIA_RPC_URL
```

为什么只查 10 个区块：

```text
有些免费 RPC 的 eth_getLogs 一次最多只允许 10 个区块范围。
```

用 receipt 查看某笔交易的 logs：

```bash
TRANSFER_TX=0x...
cast receipt "$TRANSFER_TX" --rpc-url $BASE_SEPOLIA_RPC_URL
```

## 5. 怎么读一条 Transfer log

假设 log 里有：

```text
address: 0x036C...
topics[0]: 0xddf252ad...
topics[1]: 0x000000000000000000000000dd17bdc3650de5b3b0e28b2a5fd380936518e65c
topics[2]: 0x00000000000000000000000046fef4fa806340a4e5969586be0892a61f4655e6
data:      0x00000000000000000000000000000000000000000000000000000000006acfc0
```

解读：

```text
address  = 发出事件的 USDC 合约
topic0   = Transfer 事件
topic1   = from 地址
topic2   = to 地址
data     = value
```

其中：

```text
0x6acfc0 = 7000000
```

USDC 是 6 位小数，所以：

```text
7000000 / 1_000_000 = 7 USDC
```

## 6. `cast logs` 和 `cast receipt` 的关系

`cast receipt` 看的是：

```text
某一笔交易产生了哪些 logs
```

`cast logs` 看的是：

```text
一段区块范围里，哪些 logs 符合筛选条件
```

所以：

```text
cast logs 查到的 log
  ==
某笔交易 receipt.logs 里的其中一条
```

## 7. 什么时候直接查 logs，什么时候用 indexer

| 场景 | 做法 |
| --- | --- |
| 临时调试、查最近几个区块 | `cast logs` |
| 前端展示少量事件 | `viem getLogs` / 后端缓存 |
| 查询大量历史任务 / 支付记录 | The Graph / indexer 服务 |
| 生产级实时同步 | 现成 indexer / 队列 / 数据库 |

一句话：

```text
logs 是链上真相，indexer 是把真相整理成好查的数据库。
```

## 8. 和 Web3 agent 项目的关系

后面的项目会用 logs 追踪：

- 任务创建。
- 任务完成。
- 支付完成。
- session key 授权 / 撤销。
- agent 注册 / 信誉更新。

agent 或前端不应该只相信本地状态，而应该能通过链上 logs 回放关键动作。

## 9. 最容易混淆的地方

- topic0 不是事件名字，而是事件签名的哈希。
- 链上不保存 `0xddf... = Transfer` 这张表；工具靠 ABI / 标准 / 签名库匹配。
- indexed 参数进 topics，非 indexed 参数进 data。
- logs 不是 storage，合约内部不能像读 storage 那样读历史 logs。
- `cast logs` 是跨区块筛选，`cast receipt` 是看单笔交易结果。
- 大范围扫 logs 容易被 RPC 限流，所以生产要用 indexer。

## 模块 8：L1、L2、Rollup 与 finality

模块 8 讲的是：为什么你的 Web3 agent 项目更适合先放在 L2，尤其是 Base 这类网络。

## 1. 这一模块要记住什么

最核心的关系：

```text
L1 负责更强的安全锚定和最终结算
L2 负责更便宜、更快的执行
```

你可以先记成一句话：

```text
L2 负责便宜快速执行，L1 负责更强的最终结算和安全锚定。
```

## 2. 几个核心概念

| 概念 | 简单理解 | 你要记住什么 |
| --- | --- | --- |
| L1 | 底层主链，例如 Ethereum | 安全性强、成本高、确认更慢 |
| L2 | 建在 L1 上的扩容网络，例如 Base / Arbitrum / Optimism | 成本低、确认快，最终依赖 L1 结算 |
| Rollup | 把很多 L2 交易打包后提交到 L1 | 让 L2 继承一部分 L1 安全性 |
| Sequencer | L2 负责排序和出块的组件 | 用户感知上交易很快，但要记得它不是 L1 |
| Finality | 最终性 / 不可逆程度 | “很快看到成功”和“最终结算到 L1”不是一回事 |
| Bridge | 跨链桥 | 不是普通转账，风险更高 |

## 3. Optimistic Rollup 和 ZK Rollup

先记高层区别：

| 类型 | 简单理解 | 常见特点 |
| --- | --- | --- |
| Optimistic Rollup | 先默认交易是对的，之后给挑战期 | 生态成熟，提现到 L1 可能要等 |
| ZK Rollup | 先给数学证明，再接受交易正确性 | 证明系统复杂，最终性体验可能更强 |

最短记法：

```text
Optimistic = 先信，后查
ZK = 先证，后信
```

Base 先按 Optimistic / OP Stack 方向理解即可，具体机制以后要看官方最新文档。

## 5. L2 confirmation 和 L1 settlement 的区别

这两个不是一回事：

```text
L2 confirmed：你在 L2 上很快看到交易成功
L1 settlement / finality：这批 L2 数据最终提交到 L1 并获得更强确认
```

对于你的项目：

- 小额、高频、测试网服务调用，优先接受 L2 confirmation。
- 大额、跨链退出、高风险动作，更要关注 finality 和桥接流程。

## 6. 为什么 agent 项目更适合先跑在 L2

因为：

- gas 更低。
- 确认更快。
- 小额高频交互更现实。
- Base 这类网络对 USDC、x402、agent 场景更友好。

所以 MVP 阶段通常优先：

```text
Base Sepolia -> 测试 -> 再考虑主网或跨链
```

## 7. 跨链桥是什么

桥接不是普通转账。

更接近：

```text
在 A 链锁定 / 销毁资产
  -> 桥或协议确认消息
  -> 在 B 链释放 / 铸造对应资产
```

风险包括：

- 桥合约漏洞。
- 消息验证问题。
- 错链 / 错 token。
- 提现等待期。
- 第三方桥的信任假设。

所以模块 8 的底线是：

```text
不手写桥，不碰主网大额桥接，优先官方桥和成熟 SDK。
```

## 8. 和 Web3 agent 项目的关系

你后面要做的 x402、A2A 付费服务、agent-to-agent 小额交易，优先放在 L2，是因为：

- 成本更低。
- 响应更快。
- 更适合测试网 MVP。
- 更适合频繁调用和小额支付。

但仍然要记住：

- 快确认不等于最终结算。
- 桥接不是普通转账。
- 安全假设不能全压在 L2“看起来很快”上。

## 9. 最容易混淆的地方

- L1 和 L2 不是同一层。
- Optimistic Rollup 不是“更差”，只是安全假设和流程不同。
- ZK Rollup 不是“自动更好”，而是证明系统更复杂。
- finality 不是你在前端看到成功就算最终完成。
- bridge 不是普通 token transfer。
