# Layer 1 总结与复习

## 模块 1：Layer 1 心智模型与工具准备

这一模块不是让你立刻写合约，而是先搞清楚：你用什么工具连接链、查链、验证交易结果。后面所有读链、发交易、查 receipt、看 logs，都会建立在这里。

## 1. 这一模块要记住什么

最核心的关系：

```text
人 / 脚本
  -> 钱包 / cast / viem
  -> RPC 节点
  -> 区块链网络
```

区块浏览器是旁边的“链上数据查看器”：你用钱包、`cast` 或 `viem` 发起操作后，可以去区块浏览器核对链上到底发生了什么。

## 2. 五个角色分别是什么

| 角色 | 负责什么 | 你现在怎么用 |
| --- | --- | --- |
| 钱包 | 管理私钥、签名、发交易 | 用测试钱包领取测试币、手动签名、手动发交易 |
| RPC | 让工具连接到某条链 | `https://sepolia.base.org` 连接 Base Sepolia |
| 区块浏览器 | 用网页查看链上数据 | 查地址余额、交易状态、区块、receipt、logs |
| `cast` | Foundry 的命令行链工具 | 查 chainId、余额、区块、交易，也能发简单交易 |
| `viem` | TypeScript 链交互库 | 后面用代码读链、写链、解析 ABI / logs |

一句话记法：

```text
钱包管钥匙，RPC 是通道，区块浏览器负责看结果，cast 是命令行工具，viem 是代码里的链工具。
```

## 3. Base Sepolia 基本信息

```text
Network Name: Base Sepolia
RPC URL: https://sepolia.base.org
Chain ID: 84532
Currency Symbol: ETH
Block Explorer: https://sepolia.basescan.org
```

注意：Base Sepolia 的 gas 代币是测试网 ETH，不是 `BASE` 代币。

## 4. `.env.example` 和 `.env` 的区别

`.env.example` 是模板，告诉你项目需要哪些配置。它通常可以提交到 Git。

`.env` 是真实配置，里面会放 RPC、测试私钥、测试地址。它不能提交到 Git。

模块 1 需要这几个配置：

```env
BASE_SEPOLIA_RPC_URL=https://sepolia.base.org
SEPOLIA_RPC_URL=https://ethereum-sepolia.publicnode.com
TEST_PRIVATE_KEY=你的测试钱包私钥
TEST_ADDRESS=你的测试钱包地址
```

如果你只练 Base Sepolia，`SEPOLIA_RPC_URL` 可以暂时不填。

## 5. 加载 `.env`

如果你把配置写在 `.env`，终端不会自动读取。需要加载：

```bash
set -a
source .env
set +a
```

意思是：

```text
set -a      开启自动 export
source .env 读取 .env 里的变量
set +a      关闭自动 export
```

这样后面的 `cast` 命令才能读到 `$BASE_SEPOLIA_RPC_URL`、`$TEST_ADDRESS` 等变量。

## 6. 模块 1 必会命令

确认 Foundry 工具是否可用：

```bash
cast --version
anvil --version
```

确认连到的是 Base Sepolia：

```bash
cast chain-id --rpc-url $BASE_SEPOLIA_RPC_URL
```

正确输出应该是：

```text
84532
```

查看最新区块高度：

```bash
cast block-number --rpc-url $BASE_SEPOLIA_RPC_URL
```

这个数字会一直增长，说明链在继续出块。

从私钥推导地址：

```bash
cast wallet address --private-key $TEST_PRIVATE_KEY
```

查询地址余额：

```bash
cast balance $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL --ether
```

`--ether` 的作用是把 wei 换算成 ETH 显示，比较容易读。

用区块浏览器核对地址：

```text
https://sepolia.basescan.org/address/<你的地址>
```

## 7. 常见报错

报错：

```text
error: a value is required for '--rpc-url <URL>' but none was supplied
```

意思是：你传给 `--rpc-url` 的变量是空的。

常见原因：

```bash
cast chain-id --rpc-url $BASE_SEPOLIA_RPC_URL
```

但 `$BASE_SEPOLIA_RPC_URL` 没有值。

排查：

```bash
echo "$BASE_SEPOLIA_RPC_URL"
```

如果没有输出，说明你还没加载 `.env`，或者 `.env` 里没填。

临时测试可以直接写：

```bash
cast chain-id --rpc-url https://sepolia.base.org
```

## 8. 模块 1 的安全底线

- 不要用主钱包练习。
- 不要把主网私钥放进 `.env`。
- 不要把测试私钥提交到 Git。
- `.env.example` 里不要写真实私钥。
- 先确认 `chainId = 84532`，再做 Base Sepolia 练习。
- 余额为 `0` 时，先去 faucet 领取 Base Sepolia ETH 测试币。

## 模块 2：账户、地址、私钥与签名

这一模块是给后面的 EIP-712、EIP-1271、ERC-4337 打底。你要先分清楚：谁在控制账户、什么是地址、签名到底证明了什么。

## 1. 这一模块要记住什么

最短关系链：

```text
private key
  -> public key
  -> address
```

这里最容易记混的点是：

- `public key` 和 `address` 不是一个东西。
- `address` 是从公钥加工出来的短标识。
- `signature` 是私钥对一段内容的授权证明。

一句话总结：

```text
私钥负责签名，公钥负责验证，地址负责表示账户。
```

## 2. EOA 和合约账户

| 类型 | 简单理解 | 你要记住什么 |
| --- | --- | --- |
| EOA | 外部账户，普通钱包 | 有私钥，能主动发交易 |
| Contract Account | 合约账户 | 由合约代码控制，通常没有普通私钥 |

怎么看一个地址大概率是不是合约账户：

```bash
cast code $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL
```

- 如果输出是 `0x`，通常说明这是 EOA。
- 如果输出是一长串 bytecode，说明这是合约账户。

## 3. 三种常见签名

| 签名类型 | 用来做什么 | 会不会花 gas |
| --- | --- | --- |
| 交易签名 | 转账、调用合约、改变链上状态 | 会 |
| `personal_sign` | 普通消息签名，例如登录证明 | 不会 |
| EIP-712 | 结构化数据签名，例如任务意图、授权单 | 不会，签完之后如果拿去发交易才会进入链上流程 |

你现在先记住：

```text
签消息 != 发交易
```

签消息只是证明“这个地址同意了这段内容”；发交易才会真的改链上状态。

## 4. 模块 2 必会命令

生成一个新测试账户：

```bash
cast wallet new
```

从私钥推导地址：

```bash
cast wallet address --private-key $TEST_PRIVATE_KEY
```

从私钥推导公钥：

```bash
cast wallet public-key --raw-private-key $TEST_PRIVATE_KEY
```

给一段消息签名：

```bash
MESSAGE="hello layer1"
SIGNATURE=$(cast wallet sign "$MESSAGE" --private-key $TEST_PRIVATE_KEY)
```

验证签名：

```bash
cast wallet verify --address $TEST_ADDRESS "$MESSAGE" "$SIGNATURE"
```

查地址余额，确认这个账户能连到 Base Sepolia：

```bash
cast balance $TEST_ADDRESS --rpc-url $BASE_SEPOLIA_RPC_URL
```

## 5. 最容易混淆的地方

- 公钥不是地址，地址只是从公钥派生出来的短标识。
- 签名不是消息本身，签名是对消息的数学证明。
- `cast wallet sign` 不会发交易，也不会花 gas。
- `cast wallet verify` 只是在检查签名是否匹配地址。
- 能验证签名，不代表签名人真的看懂了内容。

最后一点很重要：签名验证通过，只能说明“是这个地址背后的私钥签的”，不能说明“签的人理解了后果”。

## 6. 模块 2 的安全底线

- 不要把测试私钥提交到 Git。
- 不要在不理解内容时乱签名。
- 不要把“签消息”和“发交易”混为一谈。
- 不要把地址和公钥当成一个东西。
