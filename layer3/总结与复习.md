# Layer 3 总结与复习

## 模块 1：前端项目脚手架与链配置

这一模块的目标不是做复杂页面，而是先把 Web3 前端项目的地基搭稳：

```text
Next.js 工程
  -> 依赖版本
  -> 环境变量边界
  -> Base Sepolia 链配置
  -> 合约地址和部署块
  -> ABI 来源
  -> 最小质量命令
```

模块 1 做完后，你应该能回答三个问题：

1. 这个前端只支持哪条链？
2. 合约地址、部署块、ABI 分别从哪里来？
3. 哪些变量会进入浏览器，哪些绝对不能进前端？

### 一句话总结

模块 1 做的是“项目地基”：

```text
创建 task-console
安装 Web3 依赖
固定 webpack dev/build
准备 .env.example / .env.local
配置 Base Sepolia
同步 Layer 2 ABI
确认 lint/build/dev 能跑
```

它不急着做交易，也不急着做事件流。先保证配置清楚、依赖不冲突、不会误连主网。

### 1. 项目结构在干什么

前端项目放在：

```text
layer3/task-console/
```

核心目录是：

```text
src/app/          Next.js App Router 页面入口
src/abi/          合约 ABI 常量
src/config/       链、合约、wagmi 配置
src/lib/          格式化、错误处理、通用工具
```

不要把 RPC、合约地址、ABI、业务状态全塞进页面组件里。模块 1 的重点就是把职责拆开。

### 2. 依赖版本要注意什么

Web3 依赖安装命令是：

```bash
pnpm add viem@2 wagmi@2 @tanstack/react-query @rainbow-me/rainbowkit
```

这里 `wagmi@2` 不要省。RainbowKit 2.x 仍要求 `wagmi ^2.9.0`，如果裸装 `wagmi`，可能会装到 wagmi 3.x，然后出现 peer dependency 冲突。

`pnpm` 可能会提示一些 deprecated subdependencies。那通常是 RainbowKit / wagmi / WalletConnect / MetaMask SDK 的间接依赖警告，不是安装失败。真正要紧张的是 build 失败、peer dependency 大版本错配、或安全漏洞。

### 3. 为什么先不用 Turbopack

当前课程固定用 webpack：

```json
{
  "scripts": {
    "dev": "next dev --webpack -H 0.0.0.0",
    "build": "next build --webpack",
    "start": "next start",
    "lint": "eslint"
  }
}
```

原因有两个：

- Turbopack 在当前 Next.js + Tailwind/PostCSS 组合下可能会在 `globals.css` 阶段 panic。
- WSL + Windows 浏览器访问时，`-H 0.0.0.0` 能让 dev server 监听 IPv4，`http://127.0.0.1:3000/` 更稳定。

所以平时还是正常跑：

```bash
pnpm dev
```

只是脚本内部已经固定成 webpack + IPv4 监听。

### 4. 环境变量边界

前端环境变量分两类：

| 文件 / 前缀 | 用途 | 能不能提交 | 会不会进浏览器 |
| --- | --- | --- | --- |
| `.env.example` | 模板，只放变量名和公开默认值 | 可以 | 否 |
| `.env.local` | 本机真实值 | 不提交 | 取决于变量名前缀 |
| `NEXT_PUBLIC_*` | 浏览器可读取的公开变量 | 变量名可以提交 | 会 |
| 非 `NEXT_PUBLIC_*` | 只给 Next 服务端读取 | 不提交真实值 | 不会直接进浏览器 |

本模块已经要求：

```gitignore
.env
.env.local
.env.*.local
!.env.example
```

也就是说：

```text
.env.local    放真实配置，不提交
.env.example  放安全模板，可以提交
```

### 5. `.env.example` 应该长什么样

模板里放这些：

```env
NEXT_PUBLIC_BASE_SEPOLIA_RPC_URL=https://sepolia.base.org
NEXT_PUBLIC_CHAIN_ID=84532
NEXT_PUBLIC_BASE_SEPOLIA_EXPLORER_URL=https://sepolia.basescan.org
NEXT_PUBLIC_WC_PROJECT_ID=
NEXT_PUBLIC_AGENT_TASK_ESCROW_ADDRESS=
NEXT_PUBLIC_AGENT_TASK_ESCROW_DEPLOY_BLOCK=
NEXT_PUBLIC_USDC_ADDRESS=0x036CbD53842c5426634e7929541eC2318f3dCF7e
NEXT_PUBLIC_PERMIT_TOKEN_ADDRESS=
NEXT_PUBLIC_PERMIT_TOKEN_DEPLOY_BLOCK=
```

现在必须有的是：

```text
RPC URL
chain id
explorer URL
```

合约地址和部署块要等 Layer 2 部署完成后再填。不要乱填假地址，也不要把部署块写成 `0`。

如果使用 Alchemy / Infura / QuickNode 这类带 key 的 RPC URL，只要它写成 `NEXT_PUBLIC_*`，就会暴露在浏览器里。课程练习可以用，但最好做域名限制和额度限制。

### 6. 链配置在做什么

`src/config/chains.ts` 里会集中放 Base Sepolia：

```ts
export const BASE_SEPOLIA_CHAIN_ID = 84532;

export const BASE_SEPOLIA = {
  id: BASE_SEPOLIA_CHAIN_ID,
  name: "Base Sepolia",
  rpcUrl: process.env.NEXT_PUBLIC_BASE_SEPOLIA_RPC_URL,
  explorerUrl: process.env.NEXT_PUBLIC_BASE_SEPOLIA_EXPLORER_URL,
} as const;
```

这段的意义是：项目只支持 Base Sepolia，后面网络检查、BaseScan 链接、viem client 都从这里取配置。

正确心智是：

```text
应用只支持 Base Sepolia
  -> 钱包不在 84532 就提示切换
  -> chainId 不对就禁用签名和写链
```

不要写成“默认跟随钱包当前网络”。

### 7. 合约配置要放什么

合约配置建议放在：

```text
src/config/contracts.ts
```

至少包含：

```text
contract name
address
deployBlock
explorer URL
```

`address` 是前端调用合约的目标地址。

`deployBlock` 是部署交易所在区块，后续事件流或 indexer 从这里开始，不用从创世块扫。

但是模块 1 不做全历史扫描。事件流后面要按小窗口、分批、断点续扫来做。

### 8. ABI 是什么

ABI 是前端调用合约的接口说明书。

ABI 错了，前端可能会出现：

- 函数名不存在。
- 参数顺序错。
- 类型错。
- 事件解析失败。

本项目 ABI 只从 Layer 2 源码导出，不从区块浏览器随便复制。

### 9. 怎么从 Layer 2 同步 ABI

Foundry 当前版本要加 `--json`，否则 `forge inspect` 会输出 selector 表，不是 JSON 数组。

正确命令是：

```bash
cd /home/lenovo/solidity-course/ata/layer2

forge inspect src/AgentTaskEscrowWithPermit.sol:AgentTaskEscrowWithPermit --json abi
forge inspect src/PermitToken.sol:PermitToken --json abi
```

ABI 太长时，不要在终端里复制，直接生成前端 TS 文件：

```bash
{
  echo "export const agentTaskEscrowAbi = ("
  forge inspect src/AgentTaskEscrowWithPermit.sol:AgentTaskEscrowWithPermit --json abi
  echo ") as const;"
} > ../layer3/task-console/src/abi/AgentTaskEscrowWithPermit.ts

{
  echo "export const permitTokenAbi = ("
  forge inspect src/PermitToken.sol:PermitToken --json abi
  echo ") as const;"
} > ../layer3/task-console/src/abi/PermitToken.ts
```

这里用括号包起来是为了避免 TypeScript 自动分号插入导致：

```text
Unexpected keyword or identifier
Cannot find name 'as'
```

正确形状是：

```ts
export const agentTaskEscrowAbi = (
[
  // ABI
]
) as const;
```

### 10. 最小 ERC-20 ABI

普通 ERC-20 读余额和授权不需要完整 ABI。可以维护一个小的：

```ts
import { parseAbi } from "viem";

export const erc20Abi = parseAbi([
  "function symbol() view returns (string)",
  "function decimals() view returns (uint8)",
  "function balanceOf(address account) view returns (uint256)",
  "function allowance(address owner, address spender) view returns (uint256)",
  "function approve(address spender, uint256 amount) returns (bool)",
]);
```

它适合 USDC、PermitToken 这类 ERC-20 基础交互。

## 模块 2：viem 基础读链

这一模块的目标是：不连接钱包、不发交易，只用 `viem` 的 `publicClient` 读取 Base Sepolia 上的公开链上状态。

模块 2 的主线不是“做复杂页面”，而是把这条链路跑通：

```text
RPC URL
  -> publicClient
  -> getBlockNumber
  -> readContract
  -> bigint / address / timestamp 格式化
  -> 页面展示
```

完成模块 2 后，你应该能看懂：

- 为什么读链不需要钱包。
- 为什么写链才需要钱包签名。
- ERC-20 余额为什么是 `bigint`。
- `decimals` 和 `formatUnits` 如何配合。
- 合约地址、token 地址、owner 地址不能混填。
- `readContract` 失败时应该怎么定位原因。

### 一句话总结

模块 2 做的是“只读链上事实”：

```text
创建 publicClient
读取当前区块
读取 ERC-20 信息和余额
读取 escrow 基本状态
读取单个 task
格式化金额、地址、状态和时间
把错误展示成人能理解的提示
```

它暂时不做：

- 钱包连接。
- `approve`。
- `createTask` / `fundTask`。
- EIP-712 签名。
- 交易发送。
- 事件流扫描。
- 数据库存储。

这些内容会放到后面的模块。

### 1. publicClient 是什么

`publicClient` 是 viem 里用来读取公开链上数据的客户端。

它可以做：

```text
getBlockNumber
getBalance
readContract
getCode
```

它不能做：

```text
签名
发送需要用户授权的交易
代表钱包花钱
```

当前项目里的文件是：

```text
src/lib/clients.ts
```

核心代码是：

```ts
export const publicClient = createPublicClient({
  chain: baseSepolia,
  transport: http(process.env.NEXT_PUBLIC_BASE_SEPOLIA_RPC_URL),
});
```

只要 `publicClient.getBlockNumber()` 能返回区块高度，就说明：

```text
RPC URL 可用
链配置能用
前端可以访问 Base Sepolia
```

### 2. 当前读链控制台做了什么

模块 2 已经做了一个最小读链控制台。

主要文件是：

```text
src/app/page.tsx
src/lib/read-console.ts
src/lib/readers.ts
src/lib/format.ts
src/components/task-reader/
```

职责拆分是：

| 文件 | 负责什么 |
| --- | --- |
| `page.tsx` | 组装页面，读取 URL 参数，触发三类读取 |
| `read-console.ts` | 整理 Token / Escrow / Task 的读取状态 |
| `readers.ts` | 直接调用 viem `readContract` |
| `format.ts` | 格式化地址、金额、状态、时间 |
| `components/task-reader/*` | 表单和结果展示 |

现在页面包含四块：

```text
RPC 状态
  当前链
  当前区块
  RPC connected

Token Reader
  token 地址
  owner 地址
  symbol / decimals / balance / ETH balance

Escrow Reader
  escrow 地址
  taskCount / paused / escrow token balance
  role account 是否 PAUSER_ROLE

Task Reader
  taskId
  client / operator / token / amount / service / resultURI / refundAfter / status
```

### 3. Token Reader 需要什么参数

Token Reader 读取的是 ERC-20。

需要：

```text
Token 地址：必填
Owner 地址：可选
```

Token 地址可以是：

```text
Base Sepolia USDC
自己部署的 PermitToken
```

Owner 地址用于读取：

```text
balanceOf(owner)
getBalance(owner)
```

也就是：

```text
这个 owner 持有多少 token
这个 owner 持有多少 ETH
```

如果不填 owner，也可以读：

```text
symbol
decimals
```

### 4. Escrow Reader 需要什么参数

Escrow Reader 读取的是 `AgentTaskEscrowWithPermit`。

需要：

```text
Escrow 地址：必填，必须是 AgentTaskEscrowWithPermit 的部署地址
Token 地址：必填，用来读取 escrow 持有的 token balance
Role Account：可选，用来检查是否有 PAUSER_ROLE
```

它会读取：

```text
taskCount()
paused()
PAUSER_ROLE()
hasRole(PAUSER_ROLE, roleAccount)
token.balanceOf(escrow)
```

注意：`paused()` 不需要额外参数。它只要求你填对 escrow 合约地址。

如果把 `PermitToken` 或 USDC 地址填到 Escrow 地址栏，就会失败。比如之前这个地址：

```text
0x1C9720D6d2F9792edC05177D1f96D53d53bAbBfA
```

它是：

```text
name: Permit Token
symbol: PRM
decimals: 18
```

所以它应该填到 Token 地址，不应该填到 Escrow 地址。

正确心智是：

```text
Token 地址  -> ERC-20 / PermitToken / USDC
Escrow 地址 -> AgentTaskEscrowWithPermit
Owner 地址  -> 用户或测试账户
Role Account -> 要检查权限的地址
```

### 5. Task Reader 需要什么参数

Task Reader 读取的是：

```solidity
tasks(taskId)
```

需要：

```text
Escrow 地址：AgentTaskEscrowWithPermit 的部署地址
Task ID：大于 0，且不能大于 taskCount
```

前端会先读：

```text
taskCount()
```

再判断：

```text
taskId <= taskCount
```

这样可以避免把 public mapping 的空默认值误当成真实任务。

任务详情包括：

```text
client
operator
token
amount
service
resultURI
refundAfter
status
```

其中：

- `amount` 要按任务 token 的 `decimals` 格式化。
- `refundAfter` 是 Unix timestamp，要转成本地时间。
- `status` 是 enum 数字，要映射成文字。

状态映射是：

```text
0 -> Created
1 -> Funded
2 -> Completed
3 -> Refunded
4 -> Cancelled
```

### 6. bigint、decimals、formatUnits 的关系

链上 ERC-20 金额不是小数，而是整数。

比如 USDC 是 6 位小数：

```ts
formatUnits(1500000n, 6) // "1.5"
```

三者关系是：

```text
bigint       链上原始整数
decimals    小数点应该往左移动几位
formatUnits 把 bigint 按 decimals 转成人类字符串
```

读链时：

```text
balanceOf() -> bigint -> formatUnits -> 页面展示
```

写链时反过来：

```text
用户输入 -> parseUnits -> bigint -> 合约参数
```

模块 2 只做读链，所以重点是 `formatUnits`。

### 7. Unix timestamp 怎么展示

链上时间一般是秒级 Unix timestamp。

前端 `Date` 需要毫秒，所以要乘以 `1000`：

```ts
new Date(Number(timestamp) * 1000)
```

当前工具函数是：

```ts
export function formatUnixTimestamp(timestamp: bigint) {
  if (timestamp === BigInt(0)) {
    return "-";
  }

  return new Date(Number(timestamp) * 1000).toLocaleString("zh-CN", {
    hour12: false,
  });
}
```

意思是：

```text
链上秒级 bigint 时间戳
  -> 转成 JS 毫秒
  -> 转成本地中文时间字符串
```

### 8. readContract 失败怎么判断原因

排查顺序：

```text
RPC 是否通
  -> 地址格式是否正确
  -> 地址在当前链上是否有合约代码
  -> ABI 是否包含这个函数
  -> 地址是不是对应这个 ABI 的合约
```

最小判断：

```ts
await publicClient.getBlockNumber();
```

如果这一步失败，通常是：

```text
RPC URL 错
RPC 服务不可用
网络不通
.env.local 改完没有重启 dev server
```

再检查地址：

```ts
isAddress(address)
```

如果地址格式不对，就是输入问题。

再检查合约代码：

```ts
await publicClient.getCode({ address })
```

如果没有 code，通常是：

```text
地址填错
合约没部署
链错了
```

如果有 code，但 `readContract` 失败，重点看：

```text
这个地址是不是正确合约
ABI 是否来自当前源码
functionName 是否存在
参数类型是否正确
```

典型错误：

```text
把 ERC-20 地址填到 Escrow 地址
拿 AgentTaskEscrow ABI 去读 USDC
拿旧 ABI 去读新合约
```

### 9. 为什么模块 2 暂时不用 useReadContract

`useReadContract` 是 wagmi 的 React hook。

它很好，但它会提前引入：

```text
WagmiProvider
QueryClientProvider
React hook 生命周期
缓存
enabled 条件
refetch
账户切换
链切换
```

这些是模块 3 和模块 4 的重点。

模块 2 先用：

```ts
publicClient.readContract(...)
```

是为了先看清楚底层：

```text
到底读哪个合约
传了什么参数
返回了什么 bigint
失败是 RPC、地址还是 ABI 问题
```

学习顺序是：

```text
模块 2：先学会怎么读链
模块 4：再学会怎么在 React 里优雅读链
```

### 10. 常见问题

#### `paused()` reverted

通常不是少参数。

`paused()` 不需要参数，它只要求目标地址是支持 `paused()` 的合约。

如果报：

```text
The contract function "paused" reverted
```

优先检查：


### 2. hooks 是什么

在这个项目里，可以这样分：

```text

```

比如：


所以模块 4 的 hooks 主要负责：


组件负责把这些状态展示出来。

### 3. `publicClient.readContract` 和 `useReadContract` 的区别

`publicClient.readContract` 是 viem 的普通函数。

它适合：

```text
脚本
调试
服务端工具
事件扫描
一次性读取
不在 React 组件里的读取
```

它管的是：

```text
发起一次请求
返回一次结果
```

`useReadContract` 是 wagmi 的 React hook。

它适合：

```text
React 页面展示
依赖当前账户
依赖当前 chain
需要 loading / error
需要缓存
需要手动刷新
需要随参数变化自动重新读
```

一句话：

```text
publicClient.readContract 管一次请求
useReadContract 管 React 页面里的读链状态生命周期
```

### 4. `useBalance` 做什么

当前封装：

```ts
export function useNativeBalance(
  address: `0x${string}` | undefined,
  { enabled = true }: HookOptions = {},
) {
  return useBalance({
    address,
    query: {
      enabled: enabled && Boolean(address),
      staleTime: 10_000,
    },
  });
}
```

它用来读取原生 ETH 余额。

在 Base Sepolia 上，原生币是测试网 ETH。

关键点：

```text
address 不存在 -> 不请求
enabled=false -> 不请求
10 秒内缓存算新鲜
```

返回值里有：

```ts
nativeBalance.data
nativeBalance.isLoading
nativeBalance.error
nativeBalance.refetch()
```

### 5. `useErc20Summary` 做什么

它一次性读取 ERC-20 的：

```text
symbol()
decimals()
balanceOf(owner)
```

参数：

```text
token  ERC-20 合约地址
owner  要查询余额的账户地址，可选
```

如果没有 `owner`，就只读：

```text
symbol
decimals
```

如果有 `owner`，再读：

```text
balanceOf(owner)
```

`useReadContracts` 的返回顺序和传入数组顺序一致：

```text
data[0] -> symbol
data[1] -> decimals
data[2] -> balanceOf(owner)
```

### 6. 为什么 `useReadContracts` 适合批量读

比如 escrow 面板要一起展示：

```text
taskCount()
paused()
PAUSER_ROLE()
```

这三个都是同一个合约上的只读函数，而且页面通常一起展示。

用 `useReadContracts` 可以：

```text
一次描述多个读取
共享 loading / error / refetch 状态
减少组件里重复 hook
适合同一块 UI 面板
```

返回顺序：

```text
data[0] -> taskCount
data[1] -> paused
data[2] -> PAUSER_ROLE
```

### 7. `enabled` 解决什么问题

`enabled` 是请求开关。

比如：

```ts
enabled: enabled && Boolean(address)
```

意思是：

```text
外部允许
并且 address 存在
才真的发请求
```

它解决的是“条件不满足时不要乱发 RPC”。

常见场景：

```text
钱包未连接，不读余额
合约地址未配置，不读合约
当前链不是 Base Sepolia，不读 Base Sepolia 合约
taskId 为空，不读 tasks(taskId)
role/account 不存在，不读 hasRole
```

没有 `enabled`，页面一打开就可能出现一堆无意义错误。

### 8. `staleTime` 和 `refetchInterval`

```ts
staleTime: 10_000
```

意思是：

```text
10 秒内，这份数据算新鲜
```

组件重新渲染或重新挂载时，wagmi / react-query 可以优先用缓存，不急着重新请求。

```ts
refetchInterval: 15_000
```

意思是：

```text
每 15 秒自动重新请求一次
```

关系：

```text
staleTime        多久内别嫌数据旧
refetchInterval  多久主动刷新一次
```

`staleTime` 不等于自动刷新。自动刷新靠 `refetchInterval`。

### 9. 缓存是怎么来的

wagmi 背后用的是 TanStack React Query。

每个读链 hook 都会生成一个 query。

query key 大概由这些信息组成：

```text
chainId
合约地址
functionName
args
读的是 readContract 还是 readContracts
```

如果两个组件调用同样的 hook，参数也一样，wagmi / react-query 会认为：

```text
这是同一个 query
```

于是可以复用缓存，不需要重复打 RPC。

核心关系：

```text
queryKey 相同       -> 复用缓存
staleTime           -> 多久内缓存算新鲜
refetch             -> 手动刷新
refetchInterval     -> 定时刷新
invalidateQueries   -> 主动标记缓存过期
```

### 10. `refetch` 是什么

每个读链 hook 都有：

```ts
refetch()
```

它表示：

```text
现在立刻重新读一次
```

一个按钮刷新多组数据时，可以：

```ts
await Promise.all([
  escrowStatus.refetch(),
  tokenSummary.refetch(),
  taskDetail.refetch(),
]);
```

意思是：

```text
并发刷新 escrow、token、task 三组数据
等它们都结束后继续
```

比一个一个 `await` 更快。

### 11. 为什么写链成功后要 `invalidateQueries`

写链成功后，链上状态已经变了，但前端缓存可能还是旧的。

比如：

```text
fundTask(taskId)
```

可能让任务状态：

```text
Created -> Funded
```

但是页面上的 `useReadContract` 可能还缓存着旧的 `Created`。

所以交易确认后，要告诉 react-query：

```text
这份缓存过期了，请重新读链
```

这就是：

```ts
queryClient.invalidateQueries();
```

模块 4 只是先建立这个心智。模块 5 / 7 写链成功后再接具体的 invalidate。

### 2. 模块 5 的完整流程

用户点击“创建并注资”后，前端实际做的是：

```text
检查钱包和网络
  -> 校验表单
  -> parseUnits 转金额
  -> simulate createTask
  -> 钱包确认 createTask
  -> 等 createTask receipt
  -> 从 TaskCreated event 拿 taskId
  -> 读取 allowance(account, escrow)
  -> 如果 allowance < amount，先 approve
  -> 等 approve receipt
  -> simulate fundTask
  -> 钱包确认 fundTask
  -> 等 fundTask receipt
  -> 刷新读链缓存
```

最多三笔交易：

```text
createTask
approve
fundTask
```

如果授权额度本来就够，只有两笔：

```text
createTask
fundTask
```

### 3. 为什么 ERC-20 注资前要 approve

ERC-20 里，合约不能随便从用户钱包里转 token。

用户必须先授权：

```text
approve(spender, amount)
```

在本项目里：

```text
owner   = 当前钱包 account
spender = escrow 合约地址
amount  = 本次任务金额
```

也就是：

```text
approve(escrow, amount)
```

这句话的意思是：

```text
我允许 escrow 合约最多转走我 amount 数量的 token
```

注意：

```text
approve 不会扣款
approve 只是授权
fundTask 才是真正扣 token
```

### 4. `allowance(owner, spender)` 在检查什么

`allowance` 读取的是：

```text
owner 授权给 spender 的剩余额度
```

模块 5 里用的是：

```ts
allowance(account, escrow)
```

含义是：

```text
当前钱包授权给 escrow 的 token 额度还剩多少？
```

判断规则：

```text
allowance >= amount
  -> 可以直接 fundTask

allowance < amount
  -> 必须先 approve
```

### 5. `parseUnits`、`decimals` 和 `bigint`

用户在页面输入的是小数字符串：

```text
1
1.5
0.01
```

但链上 ERC-20 金额必须是整数 `uint256`。

如果 token 的 `decimals = 18`：

```text
1 token   -> 1000000000000000000
1.5 token -> 1500000000000000000
```

如果 token 的 `decimals = 6`：

```text
1 token   -> 1000000
1.5 token -> 1500000
```

所以前端要用：

```ts
parseUnits(amountInput, decimals)
```

关系是：

```text
用户输入字符串
  + token decimals
  -> parseUnits
  -> bigint
  -> 合约 uint256 amount
```

不要用 JS `number` 存 token 金额，因为大整数和小数精度容易出问题。

### 6. `simulateContract` 是什么

`simulateContract` 是发交易前的预演。

它不会真的写链，不会花 gas，也不会弹出钱包确认。

它的作用是先问节点：

```text
如果现在用这个 account 调这个函数，会不会成功？
```

模块 5 里会先模拟：

```text
createTask
approve
fundTask
```

它能提前发现：

```text
operator 地址不合法
token 地址不合法
amount 为 0
service 为空
refundAfter 已过期
合约 paused
allowance 不足
余额不足
任务状态不对
```

但模拟成功不等于真实交易一定成功。因为真实交易 pending 的过程中，链上状态可能变化。

### 7. `writeContract` 和 receipt

`writeContract` 是真实发交易：

```ts
const hash = await walletClient.writeContract(request);
```

它会：

```text
弹钱包
用户确认
广播交易
返回交易 hash
```

然后要等交易确认：

```ts
const receipt = await publicClient.waitForTransactionReceipt({ hash });
```

只有 receipt 成功，才能认为链上状态真的变了。

所以模块 5 的心智是：

```text
simulateContract 先检查
writeContract 发交易
waitForTransactionReceipt 等确认
receipt.status 判断结果
```

### 8. 为什么 `createTask` 拿不到 return value

Solidity 里：

```solidity
function createTask(...) external returns (uint256 taskId)
```

看起来会返回 `taskId`。

但前端发真实交易时：

```ts
walletClient.writeContract(...)
```

拿到的是：

```text
tx hash
```

不是 Solidity 的 return value。

所以真实 `taskId` 要从事件里拿：

```ts
const logs = parseEventLogs({
  abi: agentTaskEscrowAbi,
  logs: receipt.logs,
  eventName: "TaskCreated",
});
```

含义是：

```text
从交易 receipt.logs 里
用 ABI 解码
找 TaskCreated 事件
拿里面的 taskId
```

### 9. 事件和 logs

合约里 `emit TaskCreated(...)` 后，事件会变成 log。

流程是：

```text
合约 emit Event
  -> EVM 生成 log
  -> log 进入交易 receipt
  -> 节点通过 RPC 返回 receipt.logs
  -> 前端用 ABI 解析 logs
```

事件不是合约 storage。

也就是说：

```text
合约自己不能读取过去 emit 过的事件
前端 / 区块浏览器 / indexer 可以读取事件 logs
```

模块 5 只解析当前交易 receipt 里的 `TaskCreated`。完整事件流留到模块 8。

### 10. `useTaskFunding` 在做什么

`useTaskFunding` 是模块 5 的写链流程 hook。

### 12. 写链成功后为什么要刷新缓存

模块 4 的读链结果有缓存。

模块 5 写链后，链上状态变了：

```text
createTask 成功
  -> taskCount 变了
  -> 新 task 出现

approve 成功
  -> allowance 变了

fundTask 成功
  -> task.status 从 Created 变成 Funded
  -> 用户 token balance 变了
  -> escrow token balance 变了
```

所以要：

```ts
queryClient.invalidateQueries();
```

或者手动：

```ts
await Promise.all([
  allowance.refetch(),
  tokenSummary.refetch(),
  nativeBalance.refetch(),
]);
```

核心心智：

```text
链上状态变了
  -> 前端缓存旧了
  -> invalidate / refetch
  -> 重新读链
```

### 13. 常见错误

#### 用户拒签

用户在钱包里点取消。

UI 应该显示：

```text
你取消了这次操作，链上没有变化。
```

这不是系统错误，也不是合约错误。

#### allowance 不足

说明还没有授权，或者授权额度小于任务金额。

解决：

```text
先 approve(escrow, amount)
```

#### approve 成功但 fundTask 失败

这时链上状态是：

```text
任务已经 Created
token 授权已经成功
任务还没有 Funded
```

可以修正问题后重新调用 `fundTask`。

#### token 余额不足

即使 allowance 足够，如果 token balance 不够，`fundTask` 也会失败。

#### 合约 paused

如果 escrow 被暂停，`createTask` 和 `fundTask` 都不能继续。

#### 地址填错

常见错误：

```text
把 PermitToken 地址填到 Escrow 地址
把 Escrow 地址填到 Token 地址
```

判断方式：

```text
Escrow 地址应该能调用 taskCount / paused / tasks
Token 地址应该能调用 symbol / decimals / balanceOf / allowance
```

### 19. 模块 5 补充复习：关键问答

#### `parseUnits("1.5", 6)` 会得到什么？

会得到：

```ts
1500000n
```

因为 `decimals = 6`，表示：

```text
1 token = 1 * 10^6 = 1000000
```

所以：

```text
1.5 token = 1.5 * 10^6 = 1500000
```

`parseUnits` 返回的是 `bigint`，所以在 TypeScript 里显示成：

```ts
1500000n
```

核心心智：

```text
用户输入的小数字符串
  -> parseUnits
  -> 链上能接收的整数 bigint
```

#### `writeContract` 返回的是什么？

`writeContract` 返回的是：

```text
交易 hash
```

例如：

```ts
const hash = await walletClient.writeContract(request);
```

这里的 `hash` 类似：

```text
0xabc123...
```

它只说明：

```text
交易已经提交到网络
```

不代表交易已经成功。

所以后面必须等 receipt：

```ts
const receipt = await publicClient.waitForTransactionReceipt({ hash });
```

真正判断成功要看：

```ts
receipt.status === "success"
```

#### 为什么 `createTask returns (uint256 taskId)`，前端写交易却不能直接拿 return value？

因为真实写链交易不是普通函数调用。

Solidity 里虽然写了：

```solidity
function createTask(...) external returns (uint256 taskId)
```

但是前端通过钱包发交易时：

```ts
walletClient.writeContract(...)
```

拿到的是：

```text
交易 hash
```

不是 Solidity 的返回值。

真实交易会经历：

```text
签名
广播
打包
执行
生成 receipt
```

执行过程中产生的 return value 不会像 JS 函数返回值一样直接给前端。

所以前端要拿真实 `taskId`，通常用：

```text
解析 TaskCreated 事件
```

例如：

```ts
const logs = parseEventLogs({
  abi: agentTaskEscrowAbi,
  logs: receipt.logs,
  eventName: "TaskCreated",
});
```

然后从事件参数里拿：

```ts
logs[0].args.taskId
```

#### `approve` 成功但 `fundTask` 失败时，链上处于什么状态？

链上状态是：

```text
任务已经 Created
授权已经成功
注资没有成功
任务还不是 Funded
token 还没有转进 escrow
```

更具体：

```text
createTask 成功
  -> task.status = Created

approve 成功
  -> allowance(client, escrow) >= amount

fundTask 失败
  -> client token balance 没减少
  -> escrow token balance 没增加
  -> task.status 仍然是 Created
```

也就是说，`approve` 只是授权：

```text
我允许 escrow 转 token
```

但它不会真正扣款。

真正扣款发生在：

```text
fundTask(taskId)
```

如果 `fundTask` 失败，可以修正原因后重新调用：

```text
fundTask(taskId)
```

常见失败原因：

```text
token 余额不足
任务不是当前钱包创建的
任务状态已经不是 Created
escrow 被暂停
taskId 填错
RPC 或钱包异常
```

一句话总结：

```text
approve 成功只代表授权好了；
fundTask 成功才代表任务真的完成注资。
```

## 模块 6：EIP-712 任务意图签名

模块 6 学的是：

```text
client 用钱包签一份 EIP-712 TaskIntent
```

这一步只生成签名，不发交易。

也就是说：

```text
不花 gas
没有 tx hash
没有 receipt
链上状态不变
任务还是 Funded
```

模块 6 的产物是：

```text
intent + signature
```

模块 7 才会把它提交给：

```solidity
completeTask(intent, signature, resultURI)
```

### 1. 模块 6 和模块 7 的区别

一句话：

```text
模块 6：client 链下签“我同意”
模块 7：operator 上链执行“我来完成”
```

对照表：

| 对比 | 模块 6 | 模块 7 |
| --- | --- | --- |
| 核心动作 | 签名 | 提交签名 |
| 函数 | `signTypedData` | `completeTask` |
| 执行位置 | 钱包本地 / 链下 | 链上合约 |
| 是否花 gas | 不花 | 花 |
| 是否有 tx hash | 没有 | 有 |
| 是否有 receipt | 没有 | 有 |
| 链上状态 | 不变 | 改变 |
| 任务状态 | 仍然 `Funded` | 变成 `Completed` |
| 操作者 | client | operator |
| 产物 | `intent + signature` | tx hash、receipt、events |
| nonce | 读取 nonce | 消耗 nonce |

模块 6 不调用 `completeTask`，是为了把两个心智拆开：

```text
模块 6：我签了什么？
模块 7：合约怎么验证并执行？
```

### 3. TaskIntent 是什么

合约里有：

```solidity
struct TaskIntent {
    address client;
    uint256 taskId;
    address operator;
    string action;
    uint256 nonce;
    uint256 deadline;
}
```

模块 6 前端生成的是：

```text
client: 当前钱包地址
taskId: 要完成的任务 ID
operator: 任务里的 operator
action: "complete"
nonce: nonces(client)
deadline: 签名过期时间
```

它的意思是：

```text
我这个 client 同意 operator 在 deadline 前完成 taskId 这个任务。
```

### 4. client 是谁

`client` 是任务的：

```text
创建者
付款方
委托方
```

在 `createTask` 里：

```solidity
client: msg.sender
```

也就是说：

```text
谁调用 createTask，谁就是 client
```

模块 6 要求：

```text
当前钱包地址 === task.client
```

否则不能签。

因为这份签名代表：

```text
client 授权 operator 完成任务
```

不是任何人都能替 client 签。

### 5. useClientNonce 在做什么

代码：

```ts
export function useClientNonce(client: `0x${string}` | undefined) {
  const escrow = CONTRACTS.escrow.address;

  return useReadContract({
    address: escrow,
    abi: agentTaskEscrowAbi,
    functionName: "nonces",
    args: client ? [client] : undefined,
    query: {
      enabled: Boolean(escrow && client),
      staleTime: 5_000,
    },
  });
}
```

它是在读：

```solidity
nonces(client)
```

作用是拿到 client 当前的链上 nonce。

nonce 用来防重放：

```text
nonce = 0 的签名被成功使用后
下一次必须用 nonce = 1
旧签名不能重复提交
```

所以 nonce 不能前端自己加一，必须读链上。

### 6. EIP-712 的三块：domain、types、message

EIP-712 typed data 可以理解成三块：

```text
domain   这份签名在哪条链、哪个合约、哪个协议版本有效
types    要签的数据结构长什么样
message  这一次具体签的内容
```

#### domain

模块 6 里：

```ts
const domain = {
  name: "AgentTaskEscrow",
  version: "1",
  chainId: 84532,
  verifyingContract: escrow,
} as const;
```

含义：

```text
name               协议/合约域名
version            协议版本
chainId            只能在 Base Sepolia 用
verifyingContract  只能给这个 escrow 合约验证
```

`verifyingContract` 是链下构造签名数据时填进去的，但它指向链上的合约地址。

它的意义是：

```text
这份签名只给当前 escrow 合约使用
```

如果填成 token 地址、旧 escrow 地址、别的合约地址，后面模块 7 验签会失败。

#### types

模块 6 里：

```ts
export const taskIntentTypes = {
  TaskIntent: [
    { name: "client", type: "address" },
    { name: "taskId", type: "uint256" },
    { name: "operator", type: "address" },
    { name: "action", type: "string" },
    { name: "nonce", type: "uint256" },
    { name: "deadline", type: "uint256" },
  ],
} as const;
```

`types` 是模板：

```text
TaskIntent 这个结构有哪些字段
每个字段是什么类型
字段顺序是什么
```

它必须和 Solidity struct 对齐。

#### primaryType

签名时：

```ts
primaryType: "TaskIntent"
```

意思是：

```text
这次主要签的是 TaskIntent 这个结构
```

类比：

```text
types       表格模板
primaryType 这次用哪个模板
message     填进模板里的具体内容
```

#### message

模块 6 的 message 类似：

```ts
{
  client: "0xClient...",
  taskId: 1n,
  operator: "0xOperator...",
  action: "complete",
  nonce: 0n,
  deadline: 178...
}
```

这是用户这一次真正签的内容。

### 7. 为什么 action 固定为 "complete"

不是所有字段都固定。

只有 `action` 在当前模块固定。

| 字段 | 是否固定 | 原因 |
| --- | --- | --- |
| `client` | 不固定 | 每个用户钱包不同 |
| `taskId` | 不固定 | 每次签的任务不同 |
| `operator` | 不固定 | 每个任务执行方不同 |
| `action` | 固定为 `"complete"` | 当前模块只授权完成任务 |
| `nonce` | 不固定 | 每次签名/使用后会变化 |
| `deadline` | 不固定 | 每次签名可以设置不同过期时间 |

`action` 固定为 `"complete"`，是因为模块 6 当前只签：

```text
允许 operator 完成任务
```

不是签：

```text
退款
取消
改金额
改 operator
```

合约里会检查：

```solidity
keccak256(bytes(intent.action)) == COMPLETE_ACTION_HASH
```

而：

```solidity
COMPLETE_ACTION_HASH = keccak256("complete")
```

所以：

```text
"complete"   可以
"Complete"   不可以
"complete "  不可以
"refund"     不可以
```

这能防止签名被拿去做别的动作。

### 8. signTypedData 在做什么

模块 6 调用：

```ts
const signature = await signTypedDataAsync({
  domain,
  types: taskIntentTypes,
  primaryType: "TaskIntent",
  message,
});
```

它会让钱包弹出 EIP-712 结构化签名。

成功后返回：

```text
signature = 0x...
```

注意：

```text
signature 不是 tx hash
不能去 BaseScan 查交易
签名本身不花 gas
签名本身不改变链上状态
```

这份 signature 后面会给模块 7 的：

```solidity
completeTask(intent, signature, resultURI)
```

### 9. verifyTypedData 在做什么

模块 6 签完后会本地验证：

```ts
const locallyVerified = await verifyTypedData({
  address: input.client,
  domain,
  types: taskIntentTypes,
  primaryType: "TaskIntent",
  message,
  signature,
});
```

它验证的是：

```text
这个 signature 是否由 input.client 对这份 typed data 签出来
```

返回：

```text
true / false
```

但这只是本地验签，不等于 `completeTask` 一定成功。

它不能保证：

```text
task 现在还是 Funded
nonce 还没被用掉
deadline 还没过期
operator 仍然匹配链上 task
```

这些要模块 7 交给合约验证。

### 12. EIP-712 和 personal_sign 的区别

`personal_sign` 像是：

```text
签一段普通文本
```

EIP-712 像是：

```text
签一份结构化表单
```

区别：

| 对比 | personal_sign | EIP-712 |
| --- | --- | --- |
| 数据形态 | 普通字符串 / bytes | 结构化 typed data |
| 用户可读性 | 可能很差 | 更清楚 |
| 字段类型 | 不明确 | address / uint256 / string 等强类型 |
| 合约验证 | 需要自己约定文本格式 | 标准化 |
| chainId 保护 | 通常没有 | 有 |
| verifyingContract 保护 | 通常没有 | 有 |
| 适合业务授权 | 较弱 | 更适合 |

模块 6 用 EIP-712，是因为它能清楚表达：

```text
谁
哪个任务
授权谁
做什么
哪一次 nonce
什么时候过期
在哪条链
给哪个合约
```

### 13. event.preventDefault 和 setCopyStatus

在表单提交里：

```ts
event.preventDefault();
```

意思是：

```text
阻止浏览器默认提交表单导致页面刷新
让 React 自己处理签名流程
```

否则点击签名按钮可能刷新页面，签名状态会丢失。

`setCopyStatus("")` 是清空复制提示。

比如复制签名后显示：

```text

```

如果重新签了一份新 signature，就要清空旧提示，避免用户误以为新签名已经复制。

### 14. 常见问题

#### 签名成功后为什么任务还是 Funded？

因为签名不是交易。

只有模块 7 调用 `completeTask` 成功后，任务才会：

```text
Funded -> Completed
```

#### signature 能不能去 BaseScan 查？

不能。

signature 不是交易 hash。

#### 为什么当前钱包不是 client 就不能签？

因为这份签名代表 client 的授权。

operator 或其他人不能替 client 签。

#### 为什么 task 必须是 Funded？

因为 `completeTask` 完成的是已经注资的任务。

如果任务还没注资，operator 完成后也没有可支付的 token。

合约也会检查：

```text
task.status == Funded
```

#### deadline 和 refundAfter 是一回事吗？

不是。

```text
deadline    签名过期时间
refundAfter 任务退款时间
```

## 模块 7：提交签名并完成任务

这一模块的目标是：把模块 6 生成的 EIP-712 签名真正提交到链上，调用 `completeTask`，让任务从 `Funded` 变成 `Completed`。

模块 7 的主线是：

```text
模块 6 生成 intent + signature
  -> 模块 7 读取 task / nonce / verifyCompleteIntent
  -> operator 模拟 completeTask
  -> operator 发送 completeTask 交易
  -> 等待 transaction receipt
  -> 解析 TaskIntentUsed / TaskCompleted
  -> 刷新 task、nonce 和缓存
```

完成模块 7 后，你应该能看懂：

- 为什么 `client` 负责签名，`operator` 负责发交易。
- 为什么签名成功不等于任务完成。
- 为什么 `completeTask` 需要 `intent`、`signature`、`resultURI` 三个参数。
- 为什么提交前要先 `verifyCompleteIntent` 和 `simulateContract`。
- 为什么同一份签名不能重复使用。
- receipt logs 里的事件怎么变成前端能展示的数据。

### 一句话总结

模块 7 做的是“用授权签名完成任务”：

```text
client 离线授权
operator 上链执行
合约验证签名、nonce、任务状态和 operator 身份
成功后消耗 nonce
任务 Completed
资金转给 operator
前端展示 tx、receipt 和事件
```

模块 6 只是“拿到一张授权书”，模块 7 是“拿着授权书去链上办事”。

### 1. 模块 6 和模块 7 的关系

模块 6 的产物是：

```text
intent/message
signature
```

模块 7 要把它们提交给：

```solidity
completeTask(intent, signature, resultURI)
```

注意：传给合约的是 `message` 里的 `TaskIntent`，不是整个 EIP-712 对象。

也就是说，不是传：

```text
domain + types + primaryType + message + signature
```

而是传：

```text
message + signature + resultURI
```

因为合约自己知道：

```text
domain.name = AgentTaskEscrow
domain.version = 1
chainId = 当前链
verifyingContract = 当前合约
TaskIntent typehash
```

### 2. 角色一定要分清

模块 7 最容易混的是 `client` 和 `operator`。

```text
client
  -> 任务发起人
  -> 资金提供者
  -> 模块 6 里签 EIP-712 授权

operator
  -> 任务执行人
  -> 模块 7 里提交 completeTask 交易
  -> 完成后收到 token
```

合约里会要求：

```text
签名者 == intent.client
msg.sender == intent.operator
intent.client == task.client
intent.operator == task.operator
```

所以：

- 当前钱包是 `client` 但不是 `operator`，不能提交 `completeTask`。
- 当前钱包是 `operator`，但签名不是 `client` 签的，也不能成功。
- 任务里的 operator 和 intent 里的 operator 不一致，也不能成功。

### 3. 当前代码改了哪些地方

模块 7 主要看这几个文件：

| 文件 | 重点 |
| --- | --- |
| `src/components/task-console.tsx` | 把模块 6 的签名状态传给模块 7 |
| `src/components/intent-panel.tsx` | 签名成功后通过 `onSigned` 通知父组件 |
| `src/components/completion-panel.tsx` | 展示签名包、校验表单、提交 `completeTask` |
| `src/hooks/useTaskCompletion.ts` | `verifyCompleteIntent`、模拟交易、发交易、等 receipt |
| `src/lib/events.ts` | 从 receipt logs 解析 `TaskIntentUsed` 和 `TaskCompleted` |

其中最重要的连接点是：

```tsx
const [signedIntent, setSignedIntent] = useState<SignedTaskIntent | undefined>();

<IntentPanel onSigned={setSignedIntent} />
<CompletionPanel signedIntent={signedIntent} />
```

意思是：

```text
IntentPanel 负责生成签名
TaskConsole 保存签名
CompletionPanel 使用签名提交交易
```

### 4. IntentPanel 的 onSigned 是什么

`IntentPanel` 的参数：

```ts
export function IntentPanel({ onSigned }: { onSigned?: (signed: SignedTaskIntent) => void })
```

意思是：

```text
这个组件可以接收一个可选回调函数 onSigned
签名成功后，把 signed 传给父组件
```

签名成功后调用：

```ts
onSigned?.(signed);
```

这里的 `?.` 是可选调用：

```text
如果 onSigned 存在，就调用
如果 onSigned 不存在，就什么都不做
```

这样模块 6 不需要知道模块 7 的细节，只需要在签名成功时把结果“交出去”。

### 5. CompletionPanel 主要做什么

`CompletionPanel` 是模块 7 的页面核心。

它做几件事：

```text
接收 signedIntent
读取当前钱包和 chainId
读取 task 详情
读取 client 当前 nonce
调用 verifyCompleteIntent
校验 resultURI
判断按钮能不能提交
提交 completeTask
展示交易状态、receipt 和事件
```

它不是直接一上来就发交易，而是先做很多预检查。

这样用户能提前看到：

```text
钱包没连
链不对
还没有签名
resultURI 为空
nonce 已变化
签名已过期
当前钱包不是 operator
任务不是 Funded
verifyCompleteIntent 返回 false
```

### 6. validationMessages 在做什么

模块 7 里有：

```ts
const validationMessages = useMemo(
  () =>
    validateCompletionForm({
      isConnected,
      isBaseSepolia,
      account: address,
      escrow,
      signedIntent,
      task,

      resultURI,

  [
    isConnected,

  ],

```

这段是在组件渲染时实时算出“现在有哪些问题”。

依赖变化时会重新计算，比如：

```text
钱包连接状态变化
切换网络
账户变化
模块 6 重新签名
task 读链结果变化
nonce 读链结果变化
resultURI 输入变化
verifyCompleteIntent 结果变化
```

如果 `validationMessages.length === 0`，说明前端预检查通过，可以允许提交按钮。

### 7. verifyCompleteIntent 是预检查

模块 7 提交交易前先调用：

```ts
useVerifyCompleteIntent(intent, signature)
```

底层读的是合约：

```solidity
verifyCompleteIntent(intent, signature)
```

它是只读调用：

```text
不花 gas
不改链上状态
不会消耗 nonce
不会完成任务
```

它主要检查：

```text
deadline 没过期
nonce 是当前 nonces(client)
action 是 complete
taskId 存在
task.status 是 Funded
intent.client / operator 和 task 一致
签名恢复出的地址是 client
```

但它返回 `true` 也不代表交易一定成功。

因为真实交易发出前，链上状态可能又变了：

```text
nonce 被别的交易消耗
task 被别人完成
task 被退款
合约被 pause
deadline 在等待期间过期
```

所以后面还要 `simulateContract`。

### 8. simulateContract 是什么

模块 7 里有：

```ts
const simulation = await publicClient.simulateContract({
  account: input.account,
  address: input.escrow,
  abi: agentTaskEscrowAbi,
  functionName: "completeTask",
  args: [input.intent, input.signature, input.resultURI],
});
```

它的作用是：

```text
先让 RPC 节点模拟跑一遍 completeTask
看看如果当前 account 去发这笔交易会不会 revert
```

它不会真的发交易，不花 gas，不改链上状态。

这里虽然用的是 `publicClient`，但要传：

```ts
account: input.account
```

因为合约会检查：

```solidity
msg.sender == intent.operator
```

模拟时的 `account` 就是模拟出来的 `msg.sender`。

### 9. writeContract 才是真正发交易

模拟通过后，才会执行：

```ts
const hash = await walletClient.writeContract(simulation.request);
```

这一步会弹钱包确认。

它和模块 6 的签名不同：

| 对比 | 模块 6 签名 | 模块 7 写链 |
| --- | --- | --- |
| 方法 | `signTypedData` | `writeContract` |
| 是否花 gas | 不花 | 花 |
| 是否产生 tx hash | 不产生 | 产生 |
| 是否改变链上状态 | 不改变 | 改变 |
| 谁操作 | client | operator |

`writeContract` 返回的是交易哈希：

```text
0x...
```

这个 hash 可以拿去 BaseScan 查交易。

### 10. waitForTransactionReceipt 在等什么

发出交易后：

```ts
const receipt = await publicClient.waitForTransactionReceipt({ hash });
```

意思是：

```text
等这笔交易被链打包
拿到交易回执 receipt
```

receipt 里有：

```text
status
blockNumber
gasUsed
logs
```

模块 7 关心两件事：

```text
receipt.status 是否 success
receipt.logs 里有没有目标事件
```

如果 `receipt.status !== "success"`，说明交易上链了，但执行失败。

### 11. completeTask 成功后链上发生什么

合约里 `completeTask` 成功后会做：

```solidity
task.status = TaskStatus.Completed;
task.resultURI = resultURI;
IERC20(task.token).safeTransfer(task.operator, task.amount);

emit TaskIntentUsed(...);
emit TaskCompleted(...);
```

所以链上结果是：

```text
任务状态：Funded -> Completed
任务 resultURI 被写入
escrow 合约里的 token 减少
operator 的 token 增加
client 的 nonce 被消耗
发出 TaskIntentUsed 事件
发出 TaskCompleted 事件
```

注意：token 是从 escrow 合约转给 operator，不是从 client 钱包直接转。

因为模块 5 已经把资金注入 escrow 了。

### 12. nonce 是怎么防重复提交的

模块 7 的防重放有两层。

第一层是前端提示。

前端读取当前 nonce：

```ts
const currentNonce = useClientNonce(intent?.client);
```

然后比较：

```ts
signedIntent.message.nonce !== currentNonce
```

如果不一致，提示：

```text
nonce 已变化，这份签名可能已经使用过，请重新签名。
```

第二层是合约真正拒绝。

合约成功执行 `completeTask` 时会调用：

```solidity
_useCheckedNonce(intent.client, intent.nonce);
```

它会：

```text
检查 intent.nonce 是否等于当前 nonces(client)
如果相等，就把 nonce +1
如果不相等，就 revert
```

所以同一份签名成功使用一次后，第二次再提交一定会失败。

### 13. TaskIntentUsed 和 TaskCompleted 的区别

模块 7 会解析两个事件。

```text
TaskIntentUsed
  -> 说明 client 的这份授权签名已经被使用
  -> nonce 已经消耗
  -> 重点是“授权被用掉”

TaskCompleted
  -> 说明任务已经完成
  -> resultURI 已经写入
  -> token 已经支付给 operator
  -> 重点是“任务完成”
```

两者不是重复信息。

一个看授权，一个看业务结果。

### 14. parseTaskCompletionEvents 在做什么

模块 7 里：

```ts
parseTaskCompletionEvents(receipt)
```

做的是：

```text
读取 receipt.logs
用 AgentTaskEscrowWithPermit ABI 解析日志
找到 TaskIntentUsed
找到 TaskCompleted
整理成前端容易展示的对象
```

核心是：

```ts
parseEventLogs({
  abi: agentTaskEscrowAbi,
  logs: receipt.logs,
});
```

如果 ABI 里没有对应事件，或者 receipt 不是这个合约发出的事件，就解析不出目标事件。

### 15. 成功后为什么要刷新缓存

`completeTask` 成功后，很多链上数据都变了：

```text
tasks(taskId)
nonces(client)
escrow token balance
operator token balance
verifyCompleteIntent 结果
```

所以模块 7 成功后会做：

```ts
await Promise.all([taskDetail.refetch(), currentNonce.refetch(), verify.refetch()]);
await queryClient.invalidateQueries();
```

目的不是“为了形式上刷新”，而是避免页面还显示旧状态。

成功后你应该看到：

```text
task.status 变成 Completed
current nonce 增加
verifyCompleteIntent 变成 false
旧签名不能再次提交
```

### 16. 常见错误怎么判断

#### 当前钱包不是 operator

前端会提示：

```text
当前钱包不是 task.operator，不能提交 completeTask。
```

合约里也会因为：

```text
msg.sender != intent.operator
```

revert。

#### nonce 已变化

说明：

```text
这份签名可能已经用过
或者 client 用同一个 nonce 签的另一份授权已经先上链
```

处理方式：

```text
回模块 6 重新读取 nonce 并重新签名
```

#### verifyCompleteIntent 返回 false

常见原因：

```text
签名过期
nonce 不匹配
任务不是 Funded
taskId 错了
escrow 地址错了
chainId / verifyingContract 和签名时不一致
签名者不是 client
operator 不一致
action 不是 complete
```

#### simulateContract 失败

说明交易按当前状态提交大概率会 revert。

它可能比 `verifyCompleteIntent` 检查得更接近真实交易，因为它会带上：

```text
msg.sender
resultURI
当前最新链上状态
```

#### 用户拒绝钱包确认

这是用户取消，不是合约错误。

UI 应该显示可读错误，不要误判成链上执行失败。

### 17. 模块 7 不做什么

这一模块暂时不做：

- 任务退款。
- 任务取消。
- 批量任务管理。
- 完整历史事件流。
- 后端 API。
- 数据库存储。
- The Graph / indexer。
- 自动把签名上传给 operator 服务。

完整事件历史和任务时间线会放到模块 8。

模块 7 只关心一件事：

```text
拿到一份有效 TaskIntent 签名
由 operator 提交 completeTask
确认任务完成
```
