# Layer 4 总结与复习 — 账户抽象 & Agent 钱包

这一层解决了整条 Web3-Agent 路线里最关键的一块：**怎么让 agent 自主上链花钱，又不至于一被诱导就资金清零。** 答案是把“谁能做什么、能做多久、最多花多少”变成可编程、可撤销、可审计的一层——账户抽象。

一句话贯穿全层：

```text
不给 agent 主私钥，给它一把受限 session key；
边界写在链上强制，不只靠前端和 agent 代码；
底层用成熟 SDK，自己只写业务账户 / 策略，人类始终握着刹车。
```

## 分模块回顾

### 模块 1：账户抽象导论

- EOA = 一把私钥 = 全部权限，不可限额、不可撤销。把它给 agent 等于交出主卡和密码。
- prompt injection 用“代码里加 if”挡不住，边界必须落在链上。
- 智能账户把账户变成合约，从而能写“白名单 / 额度 / 过期 / 撤销 / 代付 / EIP-1271 验签”。
- 主线 ERC-4337，进阶 ERC-7702；智能账户实现 EIP-1271，接上 Layer 2 的尾巴。

### 模块 2：ERC-4337 架构

- `UserOperation` 是“操作意图”，`sender` 是智能账户，签名在 `signature` 字段。
- 五角色：Bundler（打包）、EntryPoint（唯一入口、验证 + 执行两阶段）、Account（可编程验证）、Paymaster（代付）、Factory（反事实部署）。
- `validateUserOp` 返回 `validationData`：0 成功、最低位 1 失败、高位打包时间窗。
- `callData` 是 `execute(target,value,data)`，内层 `data` 才是真正的业务调用。

### 模块 3：用 SDK 创建智能账户

- permissionless.js + Pimlico：`toSimpleSmartAccount` 推导地址、`createSmartAccountClient` 接上 bundler + paymaster。
- 账户地址在部署前就能算出（反事实地址），首笔交易触发部署。
- 智能账户零 ETH 也能发交易，gas 由 paymaster 赞助。全程不手写 UserOperation。

### 模块 4：会话密钥与权限策略（心脏）

- session key 是发给 agent 的“限额子卡”，另一把私钥，受 owner 登记的策略约束。
- 7 维度：目标白名单、函数白名单、生效 / 过期、单笔 / 每日额度、ERC-20 授权上限、撤销。
- 策略在链上 `validateUserOp` 强制，检查的是**内层调用**的目标 / 函数 / 金额。
- 生命周期：创建 / 激活 / 轮换 / 暂停 / 撤销；撤销后旧 key 立刻失能。

### 模块 5：Paymaster 与 gas 代付

- Verifying paymaster（别人替你付）vs ERC-20 paymaster（用代币付 gas）。
- 安全红线：**不能为任意 calldata / 未知合约付 gas**，否则变免费 gas 水龙头。
- Pimlico sponsorship policy 收窄赞助范围；deposit / stake 要监控。
- session key policy 保护资产，paymaster policy 保护 gas 预算，两道正交的闸。

### 模块 6（选修）：最小智能账户合约

- 账户对 EntryPoint 只暴露 `validateUserOp`，它是策略强制现场。
- owner 是 root；session key 只能走 `execute`，策略检查内层调用。
- EIP-1271 `isValidSignature` 让合约账户能“签” TaskIntent。
- 用“测试合约当 EntryPoint”把账户完整测出来（22 个测试）。

### 模块 7：ERC-7702

- 让现有 EOA 通过签名授权，把代码委托到智能账户实现，获得 AA 能力，地址 / 余额不变、可开可关。
- 可与 4337 组合：用 7702 把 EOA 升级成 4337 账户再挂 session key。
- 坑：委托目标必须可信、私钥仍有效（不解决泄露）、storage 冲突、历史 approve、每链授权。
- agent 受限钱包主线仍用 4337。

### 模块 8：把 Agent 接到智能账户

- 三角色：owner（root）、agent（持 session key 的手）、client（智能账户，付款方）、operator（执行方）。agent ≠ client。
- 两道闸门：本地（快、省 gas）+ 链上（不可绕过）。
- 闭环那一刀：escrow 把 `ECDSA.recover` 换成 `SignatureChecker`，才能接受智能账户当 client。
- 分权：花小钱用 session key，授权关键动作让 owner 亲签；revoke 是紧急刹车。

### 模块 9：控制台与安全验收

- 账户可视化：session key 作用域 / 额度 / 过期 / 状态、paymaster 状态、agent 最近动作。
- 执行链路状态全程可见，失败能定位到验证 / 执行 / paymaster / bundler。
- 故障恢复原则：高风险或越权动作能降级到人工确认；疑似被注入先 revoke。
- 安全验收十条硬指标，前四条由 Foundry 测试逐条强制。

## 关键概念对照表

| 概念 | 一句话 | 在本层哪里用 |
| --- | --- | --- |
| EOA | 一把私钥 = 全部权限，不可限额 | 反面教材（模块 1） |
| 智能账户 | 账户是合约，验证逻辑可编程 | 全层 |
| UserOperation | 操作意图，不是交易 | 模块 2 |
| EntryPoint | 全链唯一入口，两阶段跑 op | 模块 2 |
| Bundler | 把 op 打包成交易上链 | 模块 2/3（Pimlico） |
| Paymaster | 替别人付 gas | 模块 3/5 |
| Factory / 反事实地址 | 部署前就能算出账户地址 | 模块 3 |
| session key | 受限子卡，agent 用 | 模块 4/8（心脏） |
| policy | 作用域 / 额度 / 过期 / 撤销 | 模块 4/6 |
| validateUserOp | 账户验签 + 策略强制现场 | 模块 6 |
| validationData | 打包签名结果 + 时间窗 | 模块 2/6 |
| EIP-1271 | 合约账户怎么“签”消息 | 模块 6/8 |
| ERC-7702 | 让 EOA 临时获得 AA 能力 | 模块 7 |
| SignatureChecker | 同时验 EOA 和合约签名 | 模块 8（闭环那一刀） |

## 综合复习自测

1. 用银行卡类比，说清 EOA、智能账户、session key 各对应什么。
2. 画出一笔赞助 UserOperation 的完整一生，标出五个角色各在哪一步。

4. session key 的 7 个策略维度，哪些在验证的“范围检查”里判、哪些交给 EntryPoint 的时间窗、哪个在“记账”里判？
5. 为什么“权限边界必须落在链上”？只在前端 / agent 代码里判有什么漏洞？

7. session key policy 和 paymaster policy 分别保护什么？为什么正交？
8. 合约账户为什么不能用 `ecrecover` 验签？EIP-1271 怎么解决？
9. 要让 `AgentTaskEscrowWithPermit` 接受智能账户当 client，改哪一行？改完 EOA client 还能用吗？

12. 两道策略闸门分别在哪？本地闸门挡不住时，链上为什么还能挡？

## 下一步

Layer 4 让 agent 有了“受限、可撤销、可审计的手”。再往后：

```text
Layer 4：受限 session key + policy + paymaster（能自主花，也花得住手）
后续  ：x402 / AP2 支付授权、A2A 协作、可观测性与安全加固
```

高金额、高风险动作再引入 AP2 mandate 之类的人工审批；普通小额调用继续走 session key 的自动路径。整条链路的黄金法则始终没变：**最小权限 + 链上强制 + 人类兜底。**
