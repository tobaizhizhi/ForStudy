# 模块 4：会话密钥（Session Key）与权限策略 —— 本层的心脏

这是 Layer 4 最重要的一模块。前面我们建好了智能账户，现在要解决那个核心问题：**怎么给 agent 一把“只能做特定事、花特定钱、到期作废、随时可收回”的钥匙。**

这把钥匙就是 **session key（会话密钥）**，约束它的规则就是 **policy（权限策略）**。

> 目标：能给 agent 发一把 session key，明确它的作用域（能调哪些合约、哪些函数）、额度（单笔 / 每日）、过期时间，并能撤销、轮换。而且这些约束是**链上强制**的，不是前端摆设。

## 4.1 先讲清楚 session key 是什么

回到银行卡类比：

> owner（人类）是主卡持有人。session key 是你发给助理（agent）的一张**限额子卡**：只能在指定商户刷、单笔封顶、每天封顶、下周自动失效、你随时能剪卡。子卡再怎么被盗刷，损失也被框在这些限制里。

技术上，session key 是**另一把私钥**（和 owner 私钥不同），agent 持有它。owner 在智能账户里登记：“这把 key 被允许做 X、Y、Z，额度多少，何时过期。” 之后 agent 用这把 key 签 `UserOperation`，账户在 `validateUserOp` 里逐条检查策略，不满足就判签名无效。

**为什么这比“给 agent 主私钥”安全一个量级？**

```text
主私钥泄露 = 全部资产、任意合约、永久有效、不可撤销   -> 资金清零
session key 泄露 = 白名单合约、白名单函数、限额内、会过期、可撤销  -> 损失有限、可控、可收回
```

## 4.2 一把 session key 的权限策略有哪些维度

本课程把策略做小、做清楚，覆盖最关键的 7 个维度（对应选修模块 `SessionKeyPolicy.sol` 里的 `SessionKey` 结构）：

| 维度 | 字段 | 作用 | 类比 |
| --- | --- | --- | --- |
| 目标合约白名单 | `allowedTarget` | 只能调指定合约（如 escrow、token） | 只能在指定商户刷 |
| 函数选择器白名单 | `allowedSelector` | 只能调指定函数（如 createTask / approve） | 只能买指定品类 |
| 生效时间 | `validAfter` | N 之前不能用 | 卡的启用日 |
| 过期时间 | `validUntil` | N 之后自动失效 | 卡的到期日 |
| 单笔额度 | `perCallCap` | 单笔 native value 上限 | 单笔封顶 |
| 每日额度 | `dailyCap` | 每天累计上限（按自然日滚动） | 日累计封顶 |
| ERC-20 授权上限 | `erc20CallCap` | 单笔 `approve` 金额上限 | 单笔授权额度封顶 |
| 撤销 | `revoked` | 人类一键作废 | 剪卡 |

【学习提示】“花费上限”是策略里最难做全的部分。真实场景要覆盖 approve / transfer / transferFrom / permit / 各种 DeFi 调用，一般用 Rhinestone、ZeroDev 等成熟 policy 模块。本课程只做到“看懂策略是怎么在链上被强制的”这一层，其余作为练习和进阶。

## 4.3 策略在链上是怎么被强制的

看选修模块 `SessionKeyPolicy.sol` 里的核心检查函数（这段代码已通过 22 个测试）：

```solidity
/// 硬性范围检查：目标 / 选择器 / 撤销 / 单笔 native 上限 / ERC-20 授权上限。
function _checkSessionScope(address key, address target, uint256 value, bytes memory innerCall)
    internal view returns (bool)
{
    SessionKey storage sk = _sessionKeys[key];
    if (!sk.registered || sk.revoked) return false;          // 未登记 / 已撤销
    if (!_allowedTarget[key][target]) return false;          // 目标不在白名单
    if (value > sk.perCallCap) return false;                 // 超单笔 native 额度

    bytes4 selector = _selectorOf(innerCall);
    if (!_allowedSelector[key][selector]) return false;      // 函数不在白名单

    // approve(address,uint256)：解出金额，和该 token 的上限比
    if (selector == 0x095ea7b3 && innerCall.length >= 68) {
        uint256 amount = _wordAt(innerCall, 36);
        if (amount > _erc20CallCap[key][target]) return false;
    }
    return true;
}
```

关键在这里理解 `innerCall`：模块 2 讲过，`UserOperation.callData` 是 `execute(target, value, data)`，**内层的 `data` 才是真正要调的业务函数**。策略检查的正是这个内层调用：`target` 是不是白名单合约、`data` 的前 4 字节（selector）是不是白名单函数、`approve` 的金额有没有超上限。

每日额度是状态写入，放在验证阶段（ERC-4337 允许账户在验证时读写自己的存储）：

```solidity
function _consumeDailyBudget(address key, uint256 value) internal returns (bool) {
    SessionKey storage sk = _sessionKeys[key];
    uint48 slot = uint48(block.timestamp / 1 days);
    if (sk.daySlot != slot) { sk.daySlot = slot; sk.spentToday = 0; }  // 跨天自动重置
    uint256 next = sk.spentToday + value;
    if (next > sk.dailyCap) return false;                              // 超日额度
    sk.spentToday = next;
    return true;
}
```

过期 / 未生效则交给 ERC-4337 的时间窗（`validationData` 的高位），由 EntryPoint 比对当前时间强制——这更符合标准。模块 6 会把这三块（范围检查、日额度、时间窗）怎么组合进 `validateUserOp` 讲透。

**核心心智：** agent 拿着 session key 能签出一个合法签名，但只要这笔操作的目标 / 函数 / 金额 / 时间不在策略允许范围内，账户就判它签名无效，op 根本不会执行。**边界在链上，不在 agent 的代码里。**

## 4.4 权限生命周期：创建、激活、轮换、暂停、撤销

session key 不是发一次就完事，它有完整生命周期：

```text
创建（register）── 设作用域 / 额度 / 有效期
     │
激活 ── validAfter 到了、还没到 validUntil、未撤销 => active
     │
使用 ── agent 用它签 UserOperation，在策略内干活
     │
轮换（rotate）── 发一把新 key，撤销旧 key（比如怀疑泄露、或定期轮换）
     │
撤销（revoke）── 人类一键作废，旧 key 立刻失能
```

## 4.5 动手：owner 给 agent 发一把受限 session key

配合选修模块部署好的 `AgentSmartAccount`，`src/sessionKey.ts` 演示人类 owner 如何登记一把受限 key（节选，脚手架里是完整版）：

```ts
import { createWalletClient, http, encodeFunctionData, parseUnits, toFunctionSelector } from "viem";
import { privateKeyToAccount, generatePrivateKey } from "viem/accounts";
import { agentSmartAccountAbi } from "./abi/agentSmartAccount";
// ... 省略 config 导入

const ownerWallet = createWalletClient({ account: privateKeyToAccount(OWNER_PRIVATE_KEY), chain: CHAIN, transport: http(RPC_URL) });

// agent 的钥匙：真实场景 agent 侧生成并持有私钥，只把地址给 owner
const agent = privateKeyToAccount(AGENT_PRIVATE_KEY || generatePrivateKey());

const validUntil = Math.floor(Date.now() / 1000) + 7 * 24 * 3600; // 7 天后过期

// 1) 登记基础策略：7 天过期
await ownerWallet.sendTransaction({
  to: AGENT_SMART_ACCOUNT_ADDRESS,
  data: encodeFunctionData({
    abi: agentSmartAccountAbi,
    functionName: "registerSessionKey",
    args: [agent.address, 0, validUntil, 0n, 0n], // native 上限设 0（这条链路不发 ETH）
  }),
});

// 2) 作用域：只允许调 escrow 和 token
await ownerWallet.sendTransaction({
  to: AGENT_SMART_ACCOUNT_ADDRESS,
  data: encodeFunctionData({ abi: agentSmartAccountAbi, functionName: "setSessionKeyTarget", args: [agent.address, ESCROW_ADDRESS, true] }),
});

// 3) 只允许调 createTask / fundTask / approve
const createTaskSelector = toFunctionSelector("createTask(address,address,uint256,string,uint256)");
await ownerWallet.sendTransaction({
  to: AGENT_SMART_ACCOUNT_ADDRESS,
  data: encodeFunctionData({ abi: agentSmartAccountAbi, functionName: "setSessionKeySelector", args: [agent.address, createTaskSelector, true] }),
});

// 4) ERC-20 单笔授权上限 = 10 token
await ownerWallet.sendTransaction({
  to: AGENT_SMART_ACCOUNT_ADDRESS,
  data: encodeFunctionData({ abi: agentSmartAccountAbi, functionName: "setSessionKeyErc20Cap", args: [agent.address, TOKEN_ADDRESS, parseUnits("10", 18)] }),
});
```

运行：

```bash
pnpm session-key
```

它会依次登记策略，最后读一次 `isSessionKeyActive(agent)` 确认这把钥匙已经激活。撤销只需要一句：

```ts
await ownerWallet.sendTransaction({
  to: AGENT_SMART_ACCOUNT_ADDRESS,
  data: encodeFunctionData({ abi: agentSmartAccountAbi, functionName: "revokeSessionKey", args: [agent.address] }),
});
```

撤销后，agent 那把 key 再签什么，账户都判签名无效——**旧 key 立刻失能，这是安全验收的硬指标。**

## 4.6 生产里怎么做 session key（对照）

我们的 `AgentSmartAccount` 是“看得懂内部”的教学账户。生产环境有两条更成熟的路：

- **ZeroDev / Kernel**：session key / policy 体验最好，内置多种 policy（合约调用范围、ERC-20 花费上限、gas 策略、时间窗等），签名和权限用模块化方式挂载。最契合“受限自主花钱”。
- **Rhinestone modules + ERC-7579**：把 policy 做成可复用的账户模块，标准化程度高。

它们强制的策略维度和我们讲的一模一样，只是实现更完备、经过审计。**理解了本模块的原理，再去用这些 SDK，你会清楚每个开关背后链上在拦什么。**

## 4.7 一个容易踩的坑：策略只在验证阶段拦一次

我们的教学账户把额度记账放在验证阶段。如果一笔 op 验证通过、扣了日额度，但执行阶段业务合约 revert 了，日额度已经被消耗——这是教学实现的简化。生产模块通常用更精细的方式（如执行后回补、或把记账放到执行阶段并用 transient storage 关联签名者）。知道这个边界即可，模块 6 会点明。

## 本模块小结

- session key 是发给 agent 的“限额子卡”，用另一把私钥，受 owner 登记的策略约束。
- 7 个策略维度：目标白名单、函数白名单、生效 / 过期、单笔 / 每日额度、ERC-20 授权上限、撤销。
- 策略在**链上 `validateUserOp` 里强制**：agent 签得出签名，但越界的 op 会被判无效、不执行。
- 内层 `execute` 的 `target` + `data` 才是策略检查的对象，别只看外层。
- 生命周期完整：创建、激活、轮换、暂停、撤销；撤销后旧 key 立刻失能。
- 生产用 ZeroDev / Rhinestone 等成熟模块，原理和本模块一致。
