# 模块 4：会话密钥（Session Key）与权限策略 —— 本层的心脏

这是 Layer 4 最重要的一模块。前面我们建好了智能账户，现在要解决那个核心问题：**怎么给 agent 一把“只能做特定事、花特定钱、到期作废、随时可收回”的钥匙。**

这把钥匙就是 **session key（会话密钥）**，约束它的规则就是 **policy（权限策略）**。

> 目标：能给 agent 发一把 session key，明确它的作用域（能调哪些合约、哪些函数）、额度（单笔 / 每日）、过期时间，并能撤销、轮换。而且这些约束是**链上强制**的，不是前端摆设。

## 4.1 先讲清楚 session key 是什么

回到银行卡类比：

> owner（人类）是主卡持有人。session key 是你发给助理（agent）的一张**限额子卡**：只能在指定商户刷、单笔封顶、每天封顶、下周自动失效、你随时能剪卡。子卡再怎么被盗刷，损失也被框在这些限制里。

技术上，session key 是**另一把私钥**（和 owner 私钥不同），agent 持有它。owner 在智能账户里登记：“这把 key 被允许做 X、Y、Z，额度多少，何时过期。” 之后 agent 用这把 key 签 `UserOperation`，账户在 `validateUserOp` 里逐条检查策略，不满足就判签名无效。

**为什么这比“给 agent 主私钥”安全一个量级？**

```text
主私钥泄露 = 全部资产、任意合约、永久有效、不可撤销   -> 资金清零
session key 泄露 = 白名单合约、白名单函数、限额内、会过期、可撤销  -> 损失有限、可控、可收回
```

## 4.2 一把 session key 的权限策略有哪些维度

本课程把策略做小、做清楚，覆盖最关键的 7 个维度（对应选修模块 `SessionKeyPolicy.sol` 里的 `SessionKey` 结构）：

| 维度 | 字段 | 作用 | 类比 |
| --- | --- | --- | --- |
| 目标合约白名单 | `allowedTarget` | 只能调指定合约（如 escrow、token） | 只能在指定商户刷 |
| 函数选择器白名单 | `allowedSelector` | 只能调指定函数（如 createTask / approve） | 只能买指定品类 |
| 生效时间 | `validAfter` | N 之前不能用 | 卡的启用日 |
| 过期时间 | `validUntil` | N 之后自动失效 | 卡的到期日 |
| 单笔额度 | `perCallCap` | 单笔 native value 上限 | 单笔封顶 |
| 每日额度 | `dailyCap` | 每天累计上限（按自然日滚动） | 日累计封顶 |
| ERC-20 授权上限 | `erc20CallCap` | 单笔 `approve` 金额上限 | 单笔授权额度封顶 |
| 撤销 | `revoked` | 人类一键作废 | 剪卡 |

【学习提示】“花费上限”是策略里最难做全的部分。真实场景要覆盖 approve / transfer / transferFrom / permit / 各种 DeFi 调用，一般用 Rhinestone、ZeroDev 等成熟 policy 模块。本课程只做到“看懂策略是怎么在链上被强制的”这一层，其余作为练习和进阶。

## 4.3 策略在链上是怎么被强制的

看选修模块 `SessionKeyPolicy.sol` 里的核心检查函数（这段代码已通过 23 个测试）：

```solidity
/// 硬性范围检查：目标 / 选择器 / 撤销 / 单笔 native 上限 / ERC-20 授权上限。
function _checkSessionScope(address key, address target, uint256 value, bytes memory innerCall)
    internal view returns (bool)
{
    SessionKey storage sk = _sessionKeys[key];
    if (!sk.registered || sk.revoked) return false;          // 未登记 / 已撤销
    if (!_allowedTarget[key][target]) return false;          // 目标不在白名单
    if (value > sk.perCallCap) return false;                 // 超单笔 native 额度

    bytes4 selector = _selectorOf(innerCall);
    if (!_allowedSelector[key][selector]) return false;      // 函数不在白名单

    // approve(address,uint256)：标准 ABI 长度必须刚好是 4 + 32 + 32 = 68 字节
    if (selector == 0x095ea7b3) {
        if (innerCall.length != 68) return false;
        uint256 amount = _wordAt(innerCall, 36);
        if (amount > _erc20CallCap[key][target]) return false;
    }
    return true;
}
```

关键在这里理解 `innerCall`：模块 2 讲过，`UserOperation.callData` 是 `execute(target, value, data)`，**内层的 `data` 才是真正要调的业务函数**。策略检查的正是这个内层调用：`target` 是不是白名单合约、`data` 的前 4 字节（selector）是不是白名单函数、`approve` 的金额有没有超上限。

每日额度是状态写入，放在验证阶段（ERC-4337 允许账户在验证时读写自己的存储）：

```solidity
function _consumeDailyBudget(address key, uint256 value) internal returns (bool) {
    SessionKey storage sk = _sessionKeys[key];
    uint48 slot = uint48(block.timestamp / 1 days);
    if (sk.daySlot != slot) { sk.daySlot = slot; sk.spentToday = 0; }  // 跨天自动重置
    uint256 next = sk.spentToday + value;
    if (next > sk.dailyCap) return false;                              // 超日额度
    sk.spentToday = next;
    return true;
}
```

过期 / 未生效则交给 ERC-4337 的时间窗（`validationData` 的高位），由 EntryPoint 比对当前时间强制——这更符合标准。模块 6 会把这三块（范围检查、日额度、时间窗）怎么组合进 `validateUserOp` 讲透。

**核心心智：** agent 拿着 session key 能签出一个合法签名，但只要这笔操作的目标 / 函数 / 金额 / 时间不在策略允许范围内，账户就判它签名无效，op 根本不会执行。**边界在链上，不在 agent 的代码里。**

## 4.4 权限生命周期：创建、激活、轮换、暂停、撤销

session key 不是发一次就完事，它有完整生命周期：

```text
创建（register）── 设作用域 / 额度 / 有效期
     │
激活 ── validAfter 到了、还没到 validUntil、未撤销 => active
     │
使用 ── agent 用它签 UserOperation，在策略内干活
     │
轮换（rotate）── 发一把新 key，撤销旧 key（比如怀疑泄露、或定期轮换）
     │
撤销（revoke）── 人类一键作废，旧 key 立刻失能
```

## 4.5 动手：owner 给 agent 发一把受限 session key

配合选修模块部署好的 `AgentSmartAccount`，`src/sessionKey.ts` 演示人类 owner 如何登记一把受限 key（节选，脚手架里是完整版）：

```ts
import { createWalletClient, http, encodeFunctionData, parseUnits, toFunctionSelector } from "viem";
import { privateKeyToAccount, generatePrivateKey } from "viem/accounts";
import { agentSmartAccountAbi } from "./abi/agentSmartAccount";
// ... 省略 config 导入

const ownerWallet = createWalletClient({ account: privateKeyToAccount(OWNER_PRIVATE_KEY), chain: CHAIN, transport: http(RPC_URL) });

// agent 的钥匙：真实场景 agent 侧生成并持有私钥，只把地址给 owner
const agent = privateKeyToAccount(AGENT_PRIVATE_KEY || generatePrivateKey());

const validUntil = Math.floor(Date.now() / 1000) + 7 * 24 * 3600; // 7 天后过期

// 1) 登记基础策略：7 天过期
await ownerWallet.sendTransaction({
  to: AGENT_SMART_ACCOUNT_ADDRESS,
  data: encodeFunctionData({
    abi: agentSmartAccountAbi,
    functionName: "registerSessionKey",
    args: [agent.address, 0, validUntil, 0n, 0n], // native 上限设 0（这条链路不发 ETH）
  }),
});

// 2) 作用域：只允许调 escrow 和 token
await ownerWallet.sendTransaction({
  to: AGENT_SMART_ACCOUNT_ADDRESS,
  data: encodeFunctionData({ abi: agentSmartAccountAbi, functionName: "setSessionKeyTarget", args: [agent.address, ESCROW_ADDRESS, true] }),
});

// 3) 只允许调 createTask / fundTask / approve
const createTaskSelector = toFunctionSelector("createTask(address,address,uint256,string,uint256)");
await ownerWallet.sendTransaction({
  to: AGENT_SMART_ACCOUNT_ADDRESS,
  data: encodeFunctionData({ abi: agentSmartAccountAbi, functionName: "setSessionKeySelector", args: [agent.address, createTaskSelector, true] }),
});

// 4) ERC-20 单笔授权上限 = 10 token
await ownerWallet.sendTransaction({
  to: AGENT_SMART_ACCOUNT_ADDRESS,
  data: encodeFunctionData({ abi: agentSmartAccountAbi, functionName: "setSessionKeyErc20Cap", args: [agent.address, TOKEN_ADDRESS, parseUnits("10", 18)] }),
});
```

运行：

```bash
pnpm session-key
```

它会依次登记策略，最后读一次 `isSessionKeyActive(agent)` 确认这把钥匙已经激活。撤销只需要一句：

```ts
await ownerWallet.sendTransaction({
  to: AGENT_SMART_ACCOUNT_ADDRESS,
  data: encodeFunctionData({ abi: agentSmartAccountAbi, functionName: "revokeSessionKey", args: [agent.address] }),
});
```

撤销后，agent 那把 key 再签什么，账户都判签名无效——**旧 key 立刻失能，这是安全验收的硬指标。**

## 4.6 用成熟 SDK 做 session key：ZeroDev Kernel + policy

到这里你可能会问一个很关键的问题：

> 我们模块 3 用 `permissionless.js + toSimpleSmartAccount` 建的“成熟 SDK 智能账户”，为什么没有 session key？这一模块的 session key 又全靠模块 07 那个自己写的 `AgentSmartAccount`？

这个疑问非常对，得把它讲透。答案是：**session key 是“账户合约实现层”的能力，不是 bundler / paymaster 那层给的**。分三层看：

### 4.6.1 三层心智：session key 到底由谁提供

```text
第 1 层  bundler + paymaster（permissionless.js + Pimlico）
        只管：把调用打包成 UserOp、找 paymaster 赞助、丢给 bundler 上链
        对“账户是什么”无所谓 —— 它不决定有没有 session key

第 2 层  账户实现（SimpleAccount / Kernel / Safe / 自定义 AgentSmartAccount）
        session key 在这里 —— 账户的 validateUserOp 里有没有“第二把 key + 策略”的逻辑

第 3 层  policy 模块（ZeroDev policies / Rhinestone modules / 我们的 SessionKeyPolicy）
        具体拦什么：调用白名单、花费上限、时间窗、限频……
```

`permissionless.js` 是**账户无关**的基础设施。同一个 `createSmartAccountClient` 可以驱动 `toSimpleSmartAccount`、`toSafeSmartAccount`、`toKernelSmartAccount`。有没有 session key，取决于你 `to...` 选了哪种账户：

| 账户实现 | 有没有原生 session key | 说明 |
| --- | --- | --- |
| **SimpleAccount**（模块 3 主线） | ❌ 没有 | eth-infinitism 参考实现，`validateUserOp` 就一句 `owner == recover(sig)`，天生单 owner |
| **自定义 `AgentSmartAccount`**（模块 07） | ✅ 自己写的 | 我们在 `validateUserOp` 里手写链上强制，为的是“看懂内部” |
| **ZeroDev Kernel / Safe+模块 / Biconomy Nexus** | ✅ SDK 一等公民 | 模块化账户（ERC-7579），session key 是可安装的验证器模块 |

所以模块 3/6 选 SimpleAccount，是为了先把“UserOp → paymaster → bundler”这条流水线讲干净——代价就是它没有 session key。**要“既用成熟 SDK、又要 session key”，正确答案不是 SimpleAccount，而是换成 Kernel 这类模块化账户。**

### 4.6.2 ZeroDev Kernel 的 session key 长什么样

ZeroDev 把 session key 拆成两半：**signer**（用哪把 key 签）+ **policies**（这把 key 被允许做什么）。owner 用自己的 sudo validator 把一组 policy “安装”给 agent 的 signer，序列化成一个字符串交给 agent；agent 反序列化后就只能在 policy 内发 UserOp。

先装包：

```bash
pnpm add @zerodev/sdk @zerodev/ecdsa-validator @zerodev/permissions
```

**owner 侧：拼装 policy、序列化一把受限 key 给 agent**

```ts
import { createKernelAccount, addressToEmptyAccount } from "@zerodev/sdk";
import { getEntryPoint, KERNEL_V3_1 } from "@zerodev/sdk/constants";
import { signerToEcdsaValidator } from "@zerodev/ecdsa-validator";
import { toPermissionValidator, serializePermissionAccount } from "@zerodev/permissions";
import { toECDSASigner } from "@zerodev/permissions/signers";
import { toCallPolicy, toTimestampPolicy, CallPolicyVersion, ParamCondition } from "@zerodev/permissions/policies";
import { createPublicClient, http, parseUnits } from "viem";
import { privateKeyToAccount } from "viem/accounts";
import { escrowAbi } from "./abi/escrow";
import { erc20Abi } from "./abi/erc20";

const entryPoint = getEntryPoint("0.7");
const kernelVersion = KERNEL_V3_1;

// 1) owner 的 sudo validator（全权，能安装/撤销 policy）
const ecdsaValidator = await signerToEcdsaValidator(publicClient, {
  signer: privateKeyToAccount(OWNER_PRIVATE_KEY),
  entryPoint,
  kernelVersion,
});

// 2) agent 的 session key signer —— 关键：owner 只拿到 agent 报上来的**地址**，
//    绝不接触 agent 的私钥。用 addressToEmptyAccount 把地址包成一个“空账户签名器”，
//    它只够用来登记 policy，签不了任何东西。
const sessionKeyAddress = AGENT_SESSION_KEY_ADDRESS; // agent 生成后只把地址发给 owner
const emptySessionKeySigner = await toECDSASigner({ signer: addressToEmptyAccount(sessionKeyAddress) });

// 3) 一组 policy —— 对应我们那 7 个维度
const callPolicy = toCallPolicy({
  policyVersion: CallPolicyVersion.V0_0_4,
  permissions: [
    // 只能调 escrow.createTask
    { target: ESCROW_ADDRESS, valueLimit: 0n, abi: escrowAbi, functionName: "createTask" },
    // 只能调 escrow.fundTask
    { target: ESCROW_ADDRESS, valueLimit: 0n, abi: escrowAbi, functionName: "fundTask" },
    // 只能调 token.approve，且金额上限 10 token（args 约束）
    {
      target: TOKEN_ADDRESS,
      valueLimit: 0n,
      abi: erc20Abi,
      functionName: "approve",
      args: [
        { condition: ParamCondition.EQUAL, value: ESCROW_ADDRESS },              // spender 必须是 escrow
        { condition: ParamCondition.LESS_THAN_OR_EQUAL, value: parseUnits("10", 18) }, // 金额 ≤ 10
      ],
    },
  ],
});

// 7 天过期
const timestampPolicy = toTimestampPolicy({
  validUntil: Math.floor(Date.now() / 1000) + 7 * 24 * 3600,
});

// 4) 把 signer + policies 组成一个 permission validator
const permissionPlugin = await toPermissionValidator(publicClient, {
  entryPoint,
  kernelVersion,
  signer: emptySessionKeySigner,
  policies: [callPolicy, timestampPolicy],
});

// 5) Kernel 账户 = sudo（owner 全权） + regular（受限 session key）
const sessionKeyAccount = await createKernelAccount(publicClient, {
  entryPoint,
  kernelVersion,
  plugins: { sudo: ecdsaValidator, regular: permissionPlugin },
});

// 6) 序列化这把受限 key，交给 agent（这串就是 agent 的“限额子卡”）
const approval = await serializePermissionAccount(sessionKeyAccount);
```

**agent 侧：反序列化，只能在 policy 内发 UserOp**

```ts
import { deserializePermissionAccount } from "@zerodev/permissions";
import { toECDSASigner } from "@zerodev/permissions/signers";
import { createKernelAccountClient } from "@zerodev/sdk";

const sessionKeySigner = await toECDSASigner({ signer: privateKeyToAccount(AGENT_PRIVATE_KEY) });

const account = await deserializePermissionAccount(
  publicClient,
  entryPoint,
  kernelVersion,
  approval,          // owner 给的那串
  sessionKeySigner,  // agent 自己的私钥
);

const kernelClient = createKernelAccountClient({
  account,
  chain: CHAIN,
  bundlerTransport: http(PIMLICO_URL), // 仍然走 bundler + paymaster
});

// 这笔会通过：createTask 在白名单里
await kernelClient.sendUserOperation({ callData: /* encode createTask(...) */ });

// 这笔会被账户判无效：转账目标不在 callPolicy 里 —— 边界在链上，不在 agent 代码里
```

**owner 侧：撤销 —— 卸载这把 key 的 permission 插件**

ZeroDev 的撤销不是我们自定义账户里的 `revokeSessionKey`，而是 owner 用 sudo 账户把这把 key 对应的 permission 插件**卸载**掉：

```ts
// owner 用 sudo validator 重建一个全权账户 + client
const sudoAccount = await createKernelAccount(publicClient, {
  entryPoint,
  kernelVersion,
  plugins: { sudo: ecdsaValidator },
});
const sudoKernelClient = createKernelAccountClient({ account: sudoAccount, chain: CHAIN, bundlerTransport: http(PIMLICO_URL) });

// 卸载 permission 插件 —— 卸载后这把 session key 立刻失能
const txHash = await sudoKernelClient.uninstallPlugin({ plugin: permissionPlugin });
```

### 4.6.3 ZeroDev policy ↔ 本课程 7 维度对照

我们自己写的 `SessionKeyPolicy` 和 ZeroDev 的 policy 强制的是同一批东西，只是它的更完备、经过审计：

| 本课程维度（`SessionKeyPolicy`） | ZeroDev 对应 policy |
| --- | --- |
| 目标合约白名单 `allowedTarget` | `toCallPolicy` 的 `permissions[].target` |
| 函数选择器白名单 `allowedSelector` | `toCallPolicy` 的 `functionName`（配 `abi`） |
| ERC-20 授权上限 `erc20CallCap` | `toCallPolicy` 的 `args` 条件（`LESS_THAN_OR_EQUAL` 等） |
| 单笔 native 上限 `perCallCap` | `toCallPolicy` 的 `valueLimit` |
| 生效 / 过期 `validAfter` / `validUntil` | `toTimestampPolicy` |
| 每日 / 限频额度 `dailyCap` | `toRateLimitPolicy`（按次/按周期） |
| 全权（owner） | `toSudoPolicy` |
| gas 上限 | `toGasPolicy` |

**理解了本模块自己写的那套原理，再来看这些 policy，你会清楚每个开关背后链上到底在拦什么**——这正是我们先写一遍 `SessionKeyPolicy` 的价值。

### 4.6.4 版本提示（重要）

- 上面这段**已对照官方文档和 npm 上的发布包核验过**（截至 2026-07）：`@zerodev/sdk@5.5.10`、`@zerodev/permissions@5.6.3`、`@zerodev/ecdsa-validator@5.4.9`。确认的关键点：
  - `getEntryPoint("0.7")` + `KERNEL_V3_1`（来自 `@zerodev/sdk/constants`）；SDK 会自动为 EntryPoint 0.7 选 Kernel v3。
  - `addressToEmptyAccount` 是 `@zerodev/sdk` 的顶层导出；owner 侧**只拿 session key 地址**，用它包一个空账户签名器来登记 policy——owner 全程不接触 agent 私钥。
  - `toCallPolicy` 用 `CallPolicyVersion.V0_0_4`（当前文档示例值，枚举还有 V0_0_1…V0_0_5）；`args` 里的 `condition` 是 **`ParamCondition` 枚举**（`EQUAL` / `LESS_THAN_OR_EQUAL` 等），不是字符串。
  - 撤销走 `sudoKernelClient.uninstallPlugin({ plugin })`，不是自定义的 `revokeSessionKey`。
- ZeroDev 迭代较快，`CallPolicyVersion`、`KERNEL_V3_x`、包路径仍可能随版本变化。**动手前以 [ZeroDev Transaction Automation 教程](https://docs.zerodev.app/smart-accounts/permissions/transaction-automation) 和 [Call Policy 页](https://docs.zerodev.app/smart-accounts/permissions/policies/call) 的当前版本为准。** 本节是对照讲解 + 已核验骨架，不进脚手架跑（脚手架主线仍是 SimpleAccount + 自定义账户两条已验证路线；要把这条 ZeroDev 路线做成可跑脚本，见下方 4.6.5）。
- 另一条同类路线：**Rhinestone modules + ERC-7579**，把 policy 做成可复用、可跨账户安装的标准模块；Safe / Biconomy Nexus 也走模块化 session key。选型时确认它对 EntryPoint v0.7/v0.8、Kernel 版本的当前支持度。

## 4.7 一个容易踩的坑：策略只在验证阶段拦一次

我们的教学账户把额度记账放在验证阶段。如果一笔 op 验证通过、扣了日额度，但执行阶段业务合约 revert 了，日额度已经被消耗——这是教学实现的简化。生产模块通常用更精细的方式（如执行后回补、或把记账放到执行阶段并用 transient storage 关联签名者）。知道这个边界即可，模块 6 会点明。

## 本模块小结

- session key 是发给 agent 的“限额子卡”，用另一把私钥，受 owner 登记的策略约束。
- 7 个策略维度：目标白名单、函数白名单、生效 / 过期、单笔 / 每日额度、ERC-20 授权上限、撤销。
- 策略在**链上 `validateUserOp` 里强制**：agent 签得出签名，但越界的 op 会被判无效、不执行。
- 内层 `execute` 的 `target` + `data` 才是策略检查的对象，别只看外层。
- 生命周期完整：创建、激活、轮换、暂停、撤销；撤销后旧 key 立刻失能。
- 生产用 ZeroDev / Rhinestone 等成熟模块，原理和本模块一致。
