# 模块 2：ERC-4337 架构详解 —— 一笔 UserOperation 的一生

上一模块我们知道了“要用智能账户”。这一模块把 ERC-4337 这套系统彻底拆开，让你能自己画出整条链路。**这一模块几乎不写代码，但它是后面所有 SDK 调用的地图**——没有这张地图，SDK 报错时你会完全不知道是哪一层出了问题。

> 目标：模块 2 结束后，你要能回答“我发一笔赞助交易，从签名到上链，中间经过了哪些角色，各做了什么”。

## 2.1 先建立一个总览图

先记住这张图，后面每一节都是在给它填细节：

```text
        (1) 组装 + 签名                 (2) 进 UserOp 内存池
你/agent ───────────────▶ UserOperation ───────────────▶  Bundler
                                                             │
                                                    (3) 打包成一笔普通交易
                                                             ▼
                                                        EntryPoint 合约
                                                     ┌───────┴────────┐
                                              (4) 验证阶段         (5) 执行阶段
                                        account.validateUserOp   account.execute
                                        paymaster.validate...    paymaster.postOp
```

一句话串起来：

> **你签一个“想干什么”的意图（UserOperation），Bundler 帮你打包，EntryPoint 先让你的账户和 paymaster 各自“验一遍”，验过了再真正执行。**

对比一下普通 EOA 交易：EOA 是“协议直接认你的签名并执行”；ERC-4337 是“把验证这件事，从协议手里交还给账户合约自己决定”。这就是“账户**抽象**”里“抽象”二字的由来——验证逻辑不再写死，而是可编程。

## 2.2 UserOperation：不是交易，是“操作意图”

`UserOperation` 是这套系统的核心数据结构。它长得像交易，但不是交易——它是“我想让我的智能账户做某件事”的意图，要经过 Bundler 和 EntryPoint 才会变成真正的链上交易。

EntryPoint v0.7 / v0.8 用的是“打包后”的结构 `PackedUserOperation`（几个字段被压进 `bytes32` 省 calldata）。这就是我们选修模块里内联的那个 struct：

```solidity
struct PackedUserOperation {
    address sender;             // 哪个智能账户发起
    uint256 nonce;              // 防重放（4337 是 2D nonce：key + seq）
    bytes initCode;             // 账户还没部署时，放“工厂地址 + 部署 calldata”
    bytes callData;             // 账户被 EntryPoint 回调时要执行什么（通常是 execute(...)）
    bytes32 accountGasLimits;   // 打包了 verificationGasLimit 和 callGasLimit
    uint256 preVerificationGas; // 给 bundler 的额外 gas 补偿
    bytes32 gasFees;            // 打包了 maxFeePerGas 和 maxPriorityFeePerGas
    bytes paymasterAndData;     // 谁来代付 gas + 代付所需数据（空 = 账户自己付）
    bytes signature;            // 账户验证时要检查的签名（EOA 签、session key 签、多签…）
}
```

【学习提示】几个最容易混的字段：

- `sender` 是**智能账户地址**，不是你的 EOA。你的 EOA（owner）只在 `signature` 里体现。
- `callData` 通常是 `execute(target, value, data)` 的编码——即“让我的账户去调 target”。注意它有两层：外层是账户的 `execute`，内层 `data` 才是真正要调的业务函数（比如 `escrow.fundTask`）。这一点在 session key 策略里极其关键（模块 4 会用到）。
- `signature` 里放什么，完全由账户的 `validateUserOp` 决定。这就是可编程验证：owner 签、session key 签、passkey 签都行。
- `initCode` 非空表示“这个账户还没上链，请先用工厂部署它”。这带来一个魔法：**账户地址可以在部署前就算出来**（反事实地址 / counterfactual address）。

## 2.3 五个角色，各自干什么

### Bundler（打包者）

Bundler 是一个链下服务，干两件事：

1. 从 UserOperation 内存池里收集待处理的 op，做**链下模拟**（用 `eth_estimateUserOperationGas` / 模拟验证）确认它不会失败、能付得起 gas。
2. 把一批 op 打包成**一笔普通以太坊交易**，调用 `EntryPoint.handleOps(ops, beneficiary)`，自己先垫 gas，再从账户 / paymaster 那里收回来。

类比：Bundler 像**快递集散中心**——把很多人的包裹（op）攒成一车（一笔交易）发出去，运费先垫后收。本课程用 Pimlico 提供的 bundler，不自建。

### EntryPoint（唯一入口合约）

EntryPoint 是一个**全链唯一、各链地址一致**的合约（v0.7 是 `0x0000000071727De22E5E9d8BAf0edAc6f37da032`）。它是所有智能账户的总入口，负责按两个阶段跑每个 op：

```text
handleOps 里对每个 op：
  验证阶段（Verification loop）
    - account.validateUserOp(op, userOpHash, missingAccountFunds)
    - 如果有 paymaster：paymaster.validatePaymasterUserOp(...)
    - 账户 / paymaster 在这里把 gas 押金补足
  执行阶段（Execution loop）
    - 调 account，执行 op.callData（通常触发 account.execute(...)）
    - 如果有 paymaster：paymaster.postOp(...)（结算 gas）
```

**为什么要分两阶段？** 因为 Bundler 要先确信“这批 op 都验得过、付得起 gas”，才敢打包上链（否则它垫的 gas 收不回来）。验证阶段就是给这个确定性用的，所以对验证阶段有严格的存储访问规则（ERC-7562），防止账户在验证时读一些会让模拟结果和真实执行不一致的状态。

### 智能账户（Account）

账户只需要对 EntryPoint 暴露一个验证入口：

```solidity
interface IAccount {
    function validateUserOp(
        PackedUserOperation calldata userOp,
        bytes32 userOpHash,
        uint256 missingAccountFunds
    ) external returns (uint256 validationData);
}
```

- `userOpHash`：EntryPoint 算好的这笔 op 的哈希（含 chainId 和 EntryPoint 地址，防跨链 / 跨 EntryPoint 重放）。账户对它验签。
- 返回值 `validationData`：`0` 表示签名有效；最低位为 `1` 表示签名无效；高位可打包 `validAfter` / `validUntil` 时间窗（EntryPoint 会据此判断这把授权是否在有效期内）。**session key 的“过期”就是靠这个字段实现的。**
- `missingAccountFunds`：账户还差多少 gas 押金，要在验证阶段补给 EntryPoint。

模块 6（选修）我们会亲手写这个 `validateUserOp`，你会看到 session key 的所有策略就是在这里被强制的。

### Paymaster（代付者）

Paymaster 是一个**替别人付 gas** 的合约。有它，agent 的智能账户里可以一分 ETH 都没有，照样发交易——gas 由 paymaster 出。它也分两阶段：

```text
validatePaymasterUserOp：我愿不愿意为这笔 op 付 gas？（可以只赞助白名单合约 / 白名单用户 / 限额内）
postOp：交易执行后结算（比如从用户那扣等值 ERC-20 当作 gas 费）
```

**关键安全点（模块 5 展开）：** paymaster 绝不能“为任意 calldata、任意目标合约无脑付 gas”，否则会被人当免费 gas 水龙头刷爆。

### Factory（账户工厂）

第一次使用某个智能账户时，它还没上链。`initCode` 里放“工厂地址 + 工厂的部署 calldata”，EntryPoint 在验证阶段会先让工厂用 CREATE2 把账户部署出来。因为用 CREATE2，**账户地址在部署前就能确定地算出来**——这就是 SDK 里 `account.address` 在你还没发任何交易时就有值的原因。

## 2.4 一笔赞助 UserOperation 的完整一生

把上面的角色串成时间线（这就是模块 3 里 `smartAccountClient.sendTransaction` 背后发生的事）：

```text
1. 你（或 agent）说“让我的账户调 escrow.fundTask(1)”。
2. SDK 把它编码成 callData = execute(escrow, 0, fundTask(1))。
3. SDK 找 paymaster 要一段赞助数据（paymasterAndData），表示“gas 我出”。
4. SDK 估 gas、填好各字段，算出 userOpHash，用 owner / session key 私钥签名，得到完整 UserOperation。
5. SDK 把 op 发给 Bundler（Pimlico）。
6. Bundler 链下模拟：validateUserOp 过不过？paymaster 认不认？gas 够不够？
7. Bundler 把 op 打包成一笔普通交易，调 EntryPoint.handleOps。
8. EntryPoint 验证阶段：account.validateUserOp（验签 + 策略）、paymaster.validate（愿意付）。
9. EntryPoint 执行阶段：回调 account 执行 callData -> account.execute -> escrow.fundTask(1)。
10. paymaster.postOp 结算 gas。EntryPoint 事件里能查到这笔 op 的结果。
11. 你拿到一个交易 hash，去 BaseScan 能看到，也能查到 UserOperationEvent。
```

【学习提示】

- 你最终拿到的 `txHash` 是 Bundler 那笔 `handleOps` 交易的 hash；一笔交易里可能打包了多个人的 op。要精确定位“你这笔 op”，看 EntryPoint 发出的 `UserOperationEvent`（里面有 `userOpHash`）。
- 验证阶段和执行阶段的失败要分开看：验证失败（签名 / 策略 / 押金不足）op 根本不会被执行；执行失败（业务合约 revert）则是 `execute` 里的调用挂了。SDK / bundler 的报错里通常能看出是哪一层。

## 2.5 2D nonce：为什么 4337 的 nonce 不是简单 +1

EOA 的 nonce 是严格递增的单值。4337 的 nonce 是 **192 位 key + 64 位 seq** 组成的二维 nonce：同一个 key 下 seq 递增，不同 key 之间互不干扰。这让智能账户可以**并行发多个互不阻塞的操作序列**（比如一个序列跑支付、一个序列跑治理）。SDK 会帮你管，理解概念即可。

## 2.6 v0.6 / v0.7 / v0.8 版本差异（够用就行）

- v0.6：UserOperation 字段是散开的；paymaster 用 `paymasterAndData` 但布局不同。
- v0.7：引入 `PackedUserOperation`（本课程用），字段压缩，paymaster 数据结构更规整。
- v0.8：进一步优化，EIP-712 化 userOpHash 等。地址是 `0x4337084D9E255Ff0702461CF8895CE9E3b5Ff108`。

本课程用 **v0.7**（`entryPoint07Address`），生态支持最广。SDK 里换版本只是改一个参数（模块 3 会看到）。

## 本模块小结

- `UserOperation` 是“操作意图”，不是交易；`sender` 是智能账户，`signature` 里才是你的签名。
- 五个角色：Bundler（打包）、EntryPoint（唯一入口、两阶段跑）、Account（可编程验证）、Paymaster（代付 gas）、Factory（反事实部署账户）。
- 验证 / 执行两阶段的存在，是为了让 Bundler 能确信“打包上链不会亏 gas”。
- session key 的“过期”靠 `validationData` 的时间窗；“作用域 / 额度”靠账户 `validateUserOp` 里的策略。
- 本课程用 EntryPoint v0.7，用 SDK 屏蔽底层，但这张架构图你要能自己画出来。
