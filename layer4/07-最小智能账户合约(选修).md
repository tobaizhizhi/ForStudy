## 6.2 validateUserOp：策略强制的现场

这是整个账户的心脏。EntryPoint 调它，它决定“这笔 op 的签名有效吗、有没有时间窗、要不要补 gas 押金”：

```solidity
function validateUserOp(PackedUserOperation calldata userOp, bytes32 userOpHash, uint256 missingAccountFunds)
    external onlyEntryPoint returns (uint256 validationData)
{
    validationData = _validate(userOp, userOpHash);
    _payPrefund(missingAccountFunds);   // 给 EntryPoint 补它垫的 gas 押金
}
```

`_validate` 是核心逻辑，我们逐段看：

```solidity
function _validate(PackedUserOperation calldata userOp, bytes32 userOpHash) internal returns (uint256) {
    // 1) 恢复签名者
    bytes32 ethHash = userOpHash.toEthSignedMessageHash();
    (address signer, ECDSA.RecoverError err,) = ECDSA.tryRecover(ethHash, userOp.signature);
    if (err != ECDSA.RecoverError.NoError || signer == address(0)) return SIG_VALIDATION_FAILED;

    // 2) owner 是 root，无条件通过，不受 session 策略约束
    if (signer == owner) return SIG_VALIDATION_SUCCESS;

    // 3) session key 路径：这把钥匙只能通过 execute(target,value,data) 干活
    if (userOp.callData.length < 4 || bytes4(userOp.callData[0:4]) != EXECUTE_SELECTOR) {
        return SIG_VALIDATION_FAILED;
    }
    (address target, uint256 value, bytes memory innerCall) =
        abi.decode(userOp.callData[4:], (address, uint256, bytes));

    // 4) 目标 / 选择器 / 单笔额度 / ERC-20 授权额度：任一不过 => 签名无效
    if (!_checkSessionScope(signer, target, value, innerCall)) return SIG_VALIDATION_FAILED;

    // 5) 每日额度记账（验证阶段写自己的存储，ERC-4337 允许）
    if (!_consumeDailyBudget(signer, value)) return SIG_VALIDATION_FAILED;

    // 6) 时间窗交给 EntryPoint 强制（validAfter / validUntil）
    (uint48 validAfter, uint48 validUntil) = _sessionWindow(signer);
    return _packValidationData(validAfter, validUntil);
}
```

【学习提示】把它对照模块 4 的策略维度，就能看到“策略在链上被强制”这句话的具体含义：

- 第 2 步：owner 是 root。所以人类始终能做任何事，agent 才受限。
- 第 3 步：session key **只能走 `execute`**。这限制了攻击面——一把受限 key 不能直接调账户的管理函数（比如去给自己提额度）。
- 第 4 步：`_checkSessionScope` 检查的是**内层调用**（`target` + `innerCall` 的 selector + approve 金额），不是外层 `execute`。这就是模块 4 强调的“看内层”。
- 第 5 步：日额度累计，跨天重置。
- 第 6 步：过期不在这里判，而是打包进 `validationData` 的时间窗，交给 EntryPoint。

`validationData` 的打包遵循 ERC-4337 约定：

```solidity
function _packValidationData(uint48 validAfter, uint48 validUntil) internal pure returns (uint256) {
    // 低 160 位 authorizer = 0 表示成功；validUntil 在 [160,208)，validAfter 在 [208,256)
    return (uint256(validUntil) << 160) | (uint256(validAfter) << 208);
}
```

失败时直接返回 `1`（`SIG_VALIDATION_FAILED`，最低位为 1）。EntryPoint 看到最低位为 1 就判签名无效，整笔 op 不执行。

## 6.3 execute：验证过了才执行

```solidity
function execute(address target, uint256 value, bytes calldata data) external returns (bytes memory) {
    if (msg.sender != entryPoint && msg.sender != owner && msg.sender != address(this)) {
        revert NotAuthorizedCaller();
    }
    (bool ok, bytes memory ret) = target.call{value: value}(data);
    if (!ok) { assembly { revert(add(ret, 32), mload(ret)) } }  // 冒泡业务合约的 revert 原因
    emit Executed(target, value, data.length >= 4 ? bytes4(data[0:4]) : bytes4(0));
    return ret;
}
```

策略在验证阶段已经强制过，这里只保证调用者是 EntryPoint / owner，然后发起真正的调用，并把失败原因冒泡上来。

## 6.4 EIP-1271：让合约账户能“签” TaskIntent（接上 Layer 2 的尾巴）

合约账户没有私钥，`ecrecover` 对它没意义。外部合约（如 escrow）要验证“这份签名对这个合约账户是否有效”，必须调账户的 `isValidSignature`（EIP-1271）：

```solidity
function isValidSignature(bytes32 hash, bytes calldata signature) external view returns (bytes4) {
    (address signer, ECDSA.RecoverError err,) = ECDSA.tryRecover(hash, signature);
    if (err == ECDSA.RecoverError.NoError && signer == owner) return 0x1626ba7e; // magic value
    return 0xffffffff;
}
```

本最小实现只认 owner 的签名。这样：当智能账户作为 escrow 的 `client` 时，owner 用 `signTypedData` 对 `TaskIntent` 的 EIP-712 摘要签名，escrow 通过 `SignatureChecker.isValidSignatureNow(client, digest, sig)` 调到这里，验通过。

**这正是 Layer 3 留给 Layer 4 的尾巴。** 回看 Layer 2：`TaskIntentVerifier1271` 用了 `SignatureChecker`（能验合约账户），而 `AgentTaskEscrowWithPermit.completeTask` 用的是 `ECDSA.recover(...) == intent.client`（只认 EOA）。所以：

> 如果你的 escrow 要支持“智能账户当 client”，得把 `ECDSA.recover(...) == client` 换成 `SignatureChecker.isValidSignatureNow(client, digest, sig)`。这就是模块 8 集成时要做的“闭环那一刀”。

## 6.5 SessionKeyManager：策略模块本身

`SessionKeyManager` 是账户继承的策略层，管理 session key 的登记 / 撤销 / 白名单 / 额度。管理权限只给 owner：

```solidity
function _authorizeSessionAdmin() internal view override {
    if (msg.sender != owner && msg.sender != address(this)) revert NotSessionAdmin();
}
```

登记时强制“必须设过期时间”，这对 agent 钥匙是安全底线：

```solidity
function registerSessionKey(address key, uint48 validAfter, uint48 validUntil, uint256 perCallCap, uint256 dailyCap) external {
    _authorizeSessionAdmin();
    if (key == address(0)) revert InvalidSessionKey();
    if (validUntil == 0 || (validAfter != 0 && validUntil <= validAfter)) revert InvalidWindow();
    // ... 写入策略，重置当日计数
}
```

范围检查、日额度记账、时间窗读取（`_checkSessionScope` / `_consumeDailyBudget` / `_sessionWindow`）在模块 4 已经贴过，这里不重复。

## 6.6 怎么测：把账户当 EntryPoint

我们没有真实 EntryPoint，但测试很简单：**让测试合约自己当 EntryPoint**（部署时 `entryPoint = address(this)`），然后直接调 `validateUserOp` / `execute`。

```solidity
function setUp() public {
    owner = vm.addr(ownerPk);
    account = new AgentSmartAccount(owner, address(this)); // entryPoint = 本测试合约
}

function _sign(bytes32 userOpHash, uint256 pk) internal pure returns (bytes memory) {
    bytes32 ethHash = MessageHashUtils.toEthSignedMessageHash(userOpHash);
    (uint8 v, bytes32 r, bytes32 s) = vm.sign(pk, ethHash);
    return abi.encodePacked(r, s, v);
}
```

`SessionKeyPolicy.t.sol` 覆盖了每个策略维度，读它的测试名就是一份 checklist：

| 测试 | 验证什么 |
| --- | --- |
| `test_SessionKey_AllowedTargetAndSelector_Passes` | 白名单内的目标 + 函数：通过，且能真的 execute |
| `test_SessionKey_WrongTarget_Fails` | 调白名单外的合约：签名无效（返回 1） |
| `test_SessionKey_WrongSelector_Fails` | 调白名单外的函数：签名无效 |
| `test_SessionKey_PerCallCap_Fails` | 单笔超额：签名无效 |
| `test_SessionKey_DailyCap_Accumulates` | 日额度累计，第三笔超额被拦 |
| `test_SessionKey_DailyCap_ResetsNextDay` | 跨天后额度重置 |
| `test_SessionKey_Erc20Approve_UnderCap_Passes` / `OverCap_Fails` | approve 金额上限 |
| `test_SessionKey_Revoked_Fails` | 撤销后旧 key 立刻失能 |
| `test_SessionKey_Expiry_ReflectedInActiveAndWindow` | 过期反映在 active 状态和打包的时间窗 |
| `test_Owner_BypassesSessionScope` | owner 是 root，不受策略约束 |

`AgentSmartAccount.t.sol` 覆盖账户机制：只有 EntryPoint 能验证、只有 EntryPoint/owner 能 execute、execute 冒泡 revert、EIP-1271 认 owner 拒陌生人、session 管理只有 owner、垃圾签名判无效等。

## 本模块小结

- 账户对 EntryPoint 只暴露 `validateUserOp`；它是策略强制的现场。
- 策略检查的是**内层 `execute` 的目标 + 函数 + 金额**，owner 是 root 不受约束。
- `validationData` 打包签名结果和时间窗；过期交给 EntryPoint。
- EIP-1271 让合约账户能“签” `TaskIntent`；escrow 要支持它得用 `SignatureChecker` 而非 `ecrecover`。
- 用“测试合约当 EntryPoint”就能把账户完整测出来，22 个测试是一份策略 checklist。
